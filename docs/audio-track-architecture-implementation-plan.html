Audio Track — Implementation Plan

Turning audio-track-architecture.html into specific files, types, and functions: what gets added, what gets touched, and why — from the single global clip to a draggable audio block, scheduled by a dedicated audio engine.

This plan introduces one new directory — src/editor/core/audio/ (the audio engine: player pool, decode/peak cache, beat⟷second math) — plus edits to the transport engine (which sheds its player), the stores, the timeline, and the persistence layer (the document's first real version bump, v1 → v2). The visual engine (core/engine/) and the clock (TimeStore) are untouched. Decisions baked in from the architecture doc: separate audio engine · no time-stretch (width follows tempo) · silent scrub while paused · mute/solo at arm time · 50ms lookahead · adaptive waveform peaks.

A · Components that change — and one layer down

The highest level first, then a peek at the key function inside each. Files in monospace; status badges: new edit.

1 · The audio engine new core/audio/AudioEngine.ts

The new island — everything that makes sound, behind one door. A module singleton like the other engines. One layer down:

  • setBlocks(blocks, tracks) — hand it the current audio blocks + their tracks (for mute/solo); reconciles the Map<blockId, Tone.Player>, disposing orphans, creating newcomers routed through a shared master gain (headroom);
  • loadClips() — awaits decoded buffers via the waveform cache (one ToneAudioBuffer per ref, shared by players and peaks); pre-decode fires on block insert, so play() rarely waits;
  • armAll(atBeat, when, bpm, beatsPerBar) — loops the shared armBlock three-case helper (mid-clip / future / past) over every block, skipping muted/soloed-out tracks;
  • stopAll(when?) — pause / end-of-project;
  • It has no clock: when is always passed in from the transport. It never calls Tone.now() for scheduling.

2 · The waveform module new core/audio/waveform.ts

  • getBuffer(ref) — decode once per ref (bytes via getPlayableUrl), cache the AudioBuffer; shared by the player pool and the peaks;
  • getPeaks(ref, buckets) — min/max peak array; base ~1000 buckets, re-extracted at max(1000, widthPx/2) when a block is drawn wider (adaptive resolution, from E-DAW). Zoom = redraw; deep zoom = one cheap re-extraction; never a re-decode.

3 · The transport slims down edit core/playback.ts

  • Deletes its Tone.Player, loadAudio, and clipRef — the transport is now only the Tone transport + the RAF beat clock;
  • play(beat) / seek(beat) compute the shared when = Tone.now() + AUDIO_LOOKAHEAD (bumped 0.02 → 0.05) and call audioEngine.armAll(beat, when, …);
  • pause() / end-of-project call audioEngine.stopAll();
  • setBpm(bpm) — besides setting transport.bpm, re-arms all blocks when playing (every beat window just moved). Audio never time-stretches; it re-anchors.

4 · The placement model edit editor/types.ts · store/ProjectStore.ts · store/AudioStore.ts

  • types.ts — AudioBlock { id; clipRef; startBar; trimStart; trimEnd; gain?; fadeIn?; fadeOut? }, 'audio' added to TrackType, Track.audioBlocks?;
  • ProjectStore.ts — addAudioTrack, addAudioBlock, updateAudioBlock (drag → startBar; trim → trimStart/trimEnd), deleteAudioBlock. Document-only — no scheduling logic; edits are undoable automatically (generic snapshot boundary);
  • AudioStore.ts — clip: AudioClip | null → clips: Record<ref, AudioClip> + addClip/removeClip. Not undoable (loading a file isn't an edit), but persisted.

5 · Persistence goes to v2 edit persistence/serialize.ts · persistence/upgrade.ts · persistence/types.ts

  • serialize.ts — picks clips (the library) instead of audioClip; audioBlocks ride free inside tracks via the generic field pick;
  • upgrade.ts — CURRENT_VERSION = 2 and the chain's first real step: v1's audioClip → a clips library entry plus a synthesized audio track holding one block at bar 0 (trim = full clip) — old saves keep sounding identical;
  • persistence/types.ts — ProjectDocument gains clips, drops audioClip (v2 shape). Bytes need nothing: refs are already project-audio bucket paths.

6 · The audio track row new components/timeline/AudioBlockView.tsx · edit timeline/Track.tsx · TimelineArea.tsx

  • AudioBlockView.tsx — the audio analogue of Block.tsx: positioned at startBar * barWidthPx, width (trimEnd − trimStart) × bpm/60 ÷ beatsPerBar bars read reactively from bpm, a <canvas> waveform from the peak cache, left/right trim handles;
  • Track.tsx — render audioBlocks when type === 'audio'; TimelineArea.tsx — wire audio move/trim gestures (branch on track type; left-trim moves trimStart + startBar together).

7 · Load + play wiring edit hooks/usePlayback.ts · components/AudioBar.tsx

  • usePlayback.ts — play() gathers audio blocks + tracks, awaits audioEngine.loadClips(), then engine.play(start); a store subscription pushes block/mute edits to the audio engine while playing (dragging a block reschedules it live);
  • AudioBar.tsx — load path calls addClip (library) instead of setClip; becomes the entry point for dropping a clip onto an audio track.

B · Scope & the phase ladder

Incremental and always-shippable: each phase leaves a working app. Phase 1 is the data model + the document upgrade, phase 2 the audio engine (the keystone: a placed block plays at its position), 3–4 are additive UI.

In scope: the audio-block data model + document v2 upgrade, the audio engine (pool, armBlock, mute/solo, master gain, 50ms lookahead), transport wiring incl. bpm re-arm, waveform module + block view, drag + trim + multi-clip. Deferred (named, not detailed): grain-scrub while paused (silent-scrub is the decided behavior), block split, edge fades, loop-boundary retrigger (no loop regions yet), free-positioning off the snap grid, and audio-reactive visuals (a future bridge into the visual engine's ports — must respect the pause invariant; out of scope entirely).

C · Data model & the document v2 upgrade

Material (clips) vs placement (blocks) vs derived (beat windows). The window is never stored — recomputed from startBar + trim + tempo at schedule time, exactly like the visual engine's resolved graph.

C.2 — The v1 → v2 upgrade step

// persistence/upgrade.ts — the chain's first real entry. Pure, append-only, fixture-tested.
export const CURRENT_VERSION = 2;
UPGRADES[1] = (doc) => {
  const { audioClip, ...rest } = doc;
  const clips = audioClip ? { [audioClip.ref]: audioClip } : {};
  let tracks = rest.tracks, rootTrackIds = rest.rootTrackIds;
  if (audioClip) {
    // The old global clip was pinned to beat 0 — reproduce it exactly:
    // one audio track, one block at bar 0, trimmed to the full clip.
    const id = newId();
    tracks = { ...tracks, [id]: audioTrack(id, [blockAtBarZero(audioClip)]) };
    rootTrackIds = [...rootTrackIds, id];
  }
  return { ...rest, tracks, rootTrackIds, clips };
};
Why the synthesized block matters: it's the compatibility proof. A v1 project opens as a v2 project with the same audio at the same place — the keystone canary (§F phase 2) checks exactly this: a block at bar 0 must sound identical to yesterday's global clip.
Parallel audioBlocks vs the shared blocks array (open, revisit after trim): start with the parallel array — audio never enters MIDI-block code paths (piano roll, note gestures stay unaware). If trim/move gestures end up wanting the full block gesture system (marquee, copy/paste, alt-drag), migrate then. The abilities story suggests unified plumbing may win long-term; don't pre-pay for it.

D.1 — the shared armBlock helper (the one place beat⟷second math lives)

// core/audio/AudioEngine.ts
armBlock(b: AudioBlock, atBeat: number, when: number, bpm: number, bpb: number) {
  const startBeat = b.startBar * bpb;
  const endBeat   = startBeat + (b.trimEnd - b.trimStart) * bpm / 60;
  const player = this.players.get(b.id);
  if (!player) return;
  player.stop(when);
  if (atBeat >= endBeat) return;                    // past — idle
  if (atBeat <= startBeat)                           // future — schedule ahead
    player.start(when + (startBeat - atBeat) * 60 / bpm, b.trimStart);
  else                                               // mid-clip — join with an offset
    player.start(when, b.trimStart + (atBeat - startBeat) * 60 / bpm);
}

play(beat), seek(beat), and setBpm-while-playing all reduce to "transport computes when, audio engine loops armBlock". Today's single-player call is exactly the mid-clip branch for a block anchored at beat 0 — which is why the v1-compat canary works.

  • Mute/solo: armAll skips blocks whose track is muted or soloed-out (same semantics the visual engine uses to hide objects); a mute/solo toggle while playing triggers a re-arm pass via the store subscription in usePlayback.
  • Pool lifecycle: setBlocks reconciles Map<blockId, Tone.Player> — dispose orphans on block delete, create newcomers routed through the master gain. Buffers cached per ref (blocks sharing a clip share one decode), players keyed per block (overlaps need distinct players).
  • Master gain: one Tone.Gain → destination; players connect to it. Headroom against overlapping clips summing past 0dBFS; also the future seam for per-block gain/fades.
  • Lookahead: AUDIO_LOOKAHEAD = 0.05 (was 0.02) — arming N players wants the wider margin; E-DAW runs 50ms happily.
  • No clock of its own: the engine never calls Tone.now() for scheduling; when always arrives from the transport. One anchor = sample alignment.

D.2 — transport edits (playback.ts)

  • Remove player/loadAudio/clipRef; play/seek/pause/end-of-project delegate to the audio engine with (beat, when);
  • setBpm(bpm): set transport.bpm, and if playing, compute a fresh when and armAll(currentBeat, when, …) — every window moved. (Accepted consequence: a live bpm change mid-block audibly jumps the audio position; the beat clock wins. Paused bpm changes are artifact-free.)

Pause-invariant note: the waveform is static per block state — nothing animates. These are timeline components (not instruments), so the instrument lint bans don't apply; there's simply nothing time-driven to ban.

F · Phase-by-phase build (with files)

Phase 1 — Data model + document v2 stores · persistence

  • edit editor/types.ts (AudioBlock, 'audio', Track.audioBlocks?), store/AudioStore.ts (clips library), store/ProjectStore.ts (addAudioTrack/addAudioBlock/updateAudioBlock/deleteAudioBlock).
  • edit persistence/types.ts + serialize.ts (doc carries clips) + upgrade.ts (CURRENT_VERSION = 2, the v1→v2 step with the synthesized bar-0 block; fixture-test it against a captured v1 doc).
  • No playback change yet: you can create an audio track and place a block; it doesn't sound. Old projects open converted and still sound right via the (old) global-clip path — actually removed in phase 2, so phases 1+2 land together or back-to-back.

Phase 2 — Audio engine + transport wiring ★ keystone core/audio playback.ts

  • new core/audio/AudioEngine.ts (pool, armBlock, armAll with mute/solo, master gain, stopAll), minimal core/audio/waveform.ts (just getBuffer for now).
  • edit core/playback.ts (sheds player; delegates; 50ms lookahead; setBpm re-arm), hooks/usePlayback.ts (gather blocks, loadClips, live-edit subscription).
The keystone canary: open a v1 project (or drop a clip) — the upgraded bar-0 block must sound identical to yesterday's global clip: same start, same seek behavior, same end-of-project stop. Then drag the block to bar 8 and hit play before/inside/after its window — all three arm cases audible in the right places. If bar-0 differs, the beat⟷second math is wrong; fix it before any UI work.

Phase 3 — Waveform + block view + drag renderer

  • new timeline/AudioBlockView.tsx; finish core/audio/waveform.ts (getPeaks + adaptive resolution).
  • edit timeline/Track.tsx (render audio rows), TimelineArea.tsx (move gesture → updateAudioBlock(startBar)), AudioBar.tsx (→ addClip + drop-to-timeline).
  • Verify: bpm slider drag visibly resizes audio blocks while MIDI blocks hold width.

Phase 4 — Trim + multi-clip + library UI additive

  • edit AudioBlockView.tsx (edge handles), TimelineArea.tsx (right edge → trimEnd; left edge → trimStart + startBar together). Scheduling honors trim automatically.
  • Clip picker / library panel over AudioStore.clips; multiple blocks + overlapping blocks (already free in the engine — verify the master-gain headroom holds).

Phase 5 — Later deferred

  • Split at playhead (two blocks, complementary trim) · edge fades (per-player gain ramps) · grain-scrub while paused · free-positioning off the snap grid · loop-boundary retrigger when loop regions exist.

H · Risks & open questions

  • risk The beat⟷second math diverging between call sites. Mitigation: exactly one armBlock, called by play, seek, and the bpm re-arm — plus the phase-2 canary (bar-0 block ≡ yesterday's global clip).
  • risk Player pool leaks. Reconcile Map<blockId, Player> against the block set on every setBlocks; dispose orphans on delete; never key players by ref (overlaps).
  • risk Downbeat decode latency. Pre-decode on block insert via the shared cache; play() should almost never actually wait.
  • risk Upgrade-step correctness. The v1→v2 step is the first shipped upgrade — fixture-test it against a real captured v1 document (with and without a clip), and never edit it once shipped.
  • open Parallel audioBlocks vs shared block plumbing. Start parallel; revisit after trim if the gesture system wants unifying.
  • open Where the live-edit subscription lives. usePlayback subscribing to the store and nudging the audio engine (recommended, mirrors existing wiring) vs the engine subscribing itself (couples it to the store). Keep the engine store-free.
  • later Grain-scrub, split, fades, loop retrigger, audio-reactive ports — all named in the architecture doc §12; none designed here.
