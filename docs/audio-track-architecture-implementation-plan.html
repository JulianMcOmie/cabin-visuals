Audio Track — Implementation Plan

Turning audio-track-architecture.html into specific files, types, and functions: what gets added, what gets touched, and why — from the single global clip to a draggable audio block.

Contents

This plan touches three existing seams — the store/ (clip library + audio blocks), core/playback.ts (per-block scheduling), and components/timeline/ (an audio track row) — plus one small new module for waveforms. The visual engine (core/engine/) is not touched at all: audio scheduling lives entirely in the transport engine, and the beat clock is unchanged. Every section names the files it creates / edits / deletes.

A · Components that change — and one layer down

The highest level first, then a peek at the key function inside each. Files in monospace; status badges: new edit delete.

1 · The scheduling core edit core/playback.ts

Generalizes the single Tone.Player into a per-block pool, armed on beat windows. One layer down:

  • setAudioBlocks(blocks) — hand the engine the current audio blocks (called on play/seek/store-change); it keeps a Map<blockId, Tone.Player>, creating/disposing as the set changes;
  • armBlock(block, atBeat, when) — the shared three-case helper (mid-clip / future / past → start(when, offset)) that both play and seek call, so they can't diverge;
  • play / seek loop armBlock over every block instead of touching one global player; the AUDIO_LOOKAHEAD alignment and beatToPosition conversion carry over verbatim.

2 · The clip library edit store/AudioStore.ts

Grows the single clip slot into a keyed library so a project can hold many clips. Deeper:

  • clips: Record<ref, AudioClip> replacing clip: AudioClip | null; addClip(clip) / removeClip(ref);
  • AudioClip itself is unchanged (ref · fileName · duration) — already serializable and IndexedDB-ready.

3 · The placement model edit types.ts · store/ProjectStore.ts

Adds the audio track type and the AudioBlock that references a clip at a position. Deeper:

  • types.ts — AudioBlock { id; clipRef; startBar; trimStart; trimEnd; gain?; fades? }, 'audio' added to TrackType, Track.audioBlocks?;
  • ProjectStore.ts — addAudioBlock, updateAudioBlock (drag → startBar; trim → trimStart/trimEnd), deleteAudioBlock; an addAudioTrack helper. No scheduling logic here — the store is document-only.

4 · The waveform module new core/waveform.ts

Decode-once, size-independent peak extraction, cached per ref. Deeper:

  • getBuffer(ref) — decode the file (reusing AudioBar's existing decodeAudioData) and cache the AudioBuffer per ref, shared by the waveform and the Tone.Player;
  • getPeaks(ref, buckets) — a fixed-resolution min/max array computed once; the canvas draw samples from it at any pixel width (zoom = redraw, not re-analyze).

5 · The audio track row new components/timeline/AudioBlockView.tsx · edit timeline/Track.tsx · TimelineArea.tsx

  • AudioBlockView.tsx — draws a block at startBar * barWidthPx (like Block.tsx) with a <canvas> waveform and trim-edge handles;
  • Track.tsx — when track.type === 'audio', render audioBlocks instead of MIDI blocks; TimelineArea.tsx wires the audio move/trim gestures.

6 · Load + play wiring edit hooks/usePlayback.ts · components/AudioBar.tsx

  • usePlayback.ts — play() gathers the project's audio blocks, awaits engine.loadClips(refs) (was loadAudio(ref)), then engine.play(start); a store subscription pushes block edits to the engine while playing;
  • AudioBar.tsx — the file-load path calls addClip (library) instead of setClip (single slot); becomes the entry point for dropping a clip onto an audio track.

B · Scope & the phase ladder

This plan is incremental and always-shippable: each phase leaves a working app. The order mirrors the architecture doc — Phase 1 is the data model + a static block, Phase 2 makes it play at its position (the keystone), Phase 3 adds the waveform + drag, Phase 4 adds multiple clips + trim. Everything after Phase 2 is additive.

In scope here: the audio-block data model, the per-block scheduling in core/playback.ts, the waveform module + block renderer, drag + trim, and multiple clips — with file lists for all phases. Deferred (named, not detailed): audio scrubbing while paused (grain playback), block split, edge fades, and audio-reactive visuals (a future bridge into the visual engine's port matrix — out of scope entirely here).

C · Data model — the types and where they live

Two kinds of state: the material (clips — bytes + descriptor, in AudioStore/audioSource.ts) and the placement (audio blocks — position + trim, in the project document). The beat window each block occupies is derived, never stored — recomputed from startBar + trim + tempo, exactly like the visual engine's resolved graph.

  clips: Record<string, AudioClip>;   // was: clip | null

C.2 — Placement types edit types.ts

// types.ts
type TrackType = 'base' | 'add' | 'mute' | 'suppress' | 'override'
  | 'automation' | 'ability' | 'audio';   // + one member

interface AudioBlock {
  id: string;
  clipRef: string;        // → AudioClip.ref
  startBar: number;       // timeline position (mirrors Block.startBar)
  trimStart: number;      // seconds into the clip (default 0)
  trimEnd: number;        // seconds (default = clip.duration)
  gain?: number;          // per-block volume
  fadeIn?: number; fadeOut?: number;   // seconds (phase 5)
}

interface Track {          // existing shell, one optional field
  … type; name; color; muted; solo; parentId; childIds; blocks;
  audioBlocks?: AudioBlock[];   // present only when type === 'audio'
}
Mental model: AudioStore = "the sounds the project owns" (material), ProjectStore = "where they're placed" (edits). The placement is undoable (it's an edit, like moving a MIDI block); whether adding a clip to the library is undoable is an open call (§H). The engine reads both on play/seek; neither store knows about the engine.
Reuse blocks or add audioBlocks? Two options. (a) a parallel audioBlocks: AudioBlock[] (drawn above) — clean separation, audio never enters MIDI-block code paths. (b) put audio blocks in the existing blocks array with a discriminant — reuses the block gesture/selection plumbing wholesale (marquee, copy/paste, alt-drag) but forces every MIDI-block reader to guard against audio payloads. Recommendation: start with (a) for isolation; if the trim/move gestures end up wanting the full block gesture system, migrate to (b). This mirrors the visual engine's own "parallel structure vs. child tracks" decision (abilities), which ultimately unified on shared plumbing — so (b) may win long-term.

D · The scheduling core — core/playback.ts

core/playback.tshooks/usePlayback.tscore/audioSource.ts (as-is)

Today's engine owns one Tone.Player and pins it to beat 0. The change is mechanical: a player pool keyed by block id, and a single shared armBlock helper that both play and seek call.

D.1 — the shared armBlock helper

// core/playback.ts — the one place beat⟷second math lives
private armBlock(b: AudioBlock, atBeat: number, when: number) {
  const bpm = this.callbacks.getBpm();
  const bpb = this.callbacks.getBeatsPerBar();
  const startBeat = b.startBar * bpb;
  const endBeat   = startBeat + (b.trimEnd - b.trimStart) * bpm / 60;
  const player    = this.players.get(b.id);
  if (!player) return;
  player.stop(when);
  if (atBeat >= endBeat) return;                    // past — idle
  if (atBeat <= startBeat)                            // future — schedule ahead
    player.start(when + (startBeat - atBeat) * 60 / bpm, b.trimStart);
  else                                               // mid-clip — join with an offset
    player.start(when, b.trimStart + (atBeat - startBeat) * 60 / bpm);
}

play(beat) and seek(beat) both reduce to "set the transport position, pick a shared when, then armBlock(b, beat, when) for every block." Today's single-player call — start(when, startBeat*60/bpm) — is the mid-clip branch for a block anchored at beat 0; adding startBar shifts the anchor and introduces the future/past cases.

Player pool lifecycle. loadClips(refs) replaces loadAudio(ref): it disposes players whose block is gone, creates a Tone.Player per block (two blocks of the same clip may overlap → distinct players sharing one decoded ToneAudioBuffer per ref), and awaits Tone.loaded(). Buffers come from core/waveform.ts's per-ref cache, so audio and waveform share one decode. Never leak a player on block delete — key the map by block id and reconcile against the current block set.
Live bpm change. setBpm today just sets transport.bpm. With positioned audio, a live change moves every block's beat window (seconds fixed, beats not). Simplest correct behavior: on a live bpm change while playing, re-arm all blocks (a seek(currentBeat)-style pass). Audio never time-stretches — it re-anchors.

F · Phase-by-phase build (with files)

Phase 1 — Audio-block data model + type data

Stand up the types and store actions with no playback change: you can create an audio track and place a block; it just doesn't sound yet.

  • edit types.ts (AudioBlock, 'audio' in TrackType, Track.audioBlocks?), store/AudioStore.ts (clips library + addClip/removeClip).
  • edit store/ProjectStore.ts — addAudioTrack, addAudioBlock, updateAudioBlock, deleteAudioBlock. Follow the existing block-action shape; no scheduling logic here.

Phase 2 — Transport scheduling from block position ★ keystone the thesis

Make a placed block play at its position. This is the whole feature.

  • edit core/playback.ts — player pool (Map<blockId, Tone.Player>), loadClips(refs), the shared armBlock helper, and play/seek looping it. AUDIO_LOOKAHEAD + beatToPosition unchanged.
  • edit hooks/usePlayback.ts — play() gathers audio blocks + refs, awaits engine.loadClips, then engine.play. A store subscription pushes block edits to the engine while playing (so dragging a block reschedules it live).
Canary: a block placed at bar 0 must sound identical to today's global clip (the mid-clip branch at beat 0). If it doesn't, the beat⟷second math is wrong — the same "reproduce the old behavior exactly" check the visual-engine plan used for the Cube.

Phase 3 — Waveform block rendering + drag renderer

  • new core/waveform.ts (decode-cache + peaks, shared with the player pool), components/timeline/AudioBlockView.tsx (canvas waveform + position).
  • edit timeline/Track.tsx (render audio blocks), TimelineArea.tsx (move gesture → updateAudioBlock(startBar)), AudioBar.tsx (→ addClip).

Phase 4 — Multiple clips + trim additive

  • Multiple clips is free once the library + per-block pool exist — the §D loop already iterates all blocks. Add a clip picker / library panel drawing from AudioStore.clips.
  • Trim: edit AudioBlockView.tsx (edge handles) + TimelineArea.tsx (right edge → trimEnd; left edge → trimStart + startBar together). Scheduling honors trim automatically (armBlock already uses both).

Phase 5 — Polish: split, fades, scrub later

  • Split — cut at the playhead into two blocks sharing the ref (complementary trim/startBar). Free once trim exists.
  • Fades — fadeIn/fadeOut via a per-player gain node; drawn as a corner ramp on the waveform.
  • Audio scrubbing while paused — a throttled short-lived grain player at the scrub position. Ship silent-scrub (today's behavior) first; this is a real feature with real cost. deferred

H · Risks & open questions

  • semantics Seconds vs. beats under tempo change. The load-bearing decision. Recommended: no time-stretch; the block's width follows tempo and a live bpm change re-windows all blocks (§D.1). Nail this before building the renderer, or trim widths and drag math will fight the schedule.
  • risk Player pool leaks. One player per block; reconcile the Map<blockId, Player> against the block set on every loadClips, disposing orphans. The current single-player dispose() must generalize, not just repeat.
  • risk Load latency at the downbeat. play() awaits decode. With many clips, pre-decode on insert (via waveform.ts's cache) so the first beat isn't gated on decoding — otherwise a big project stutters at play.
  • open Parallel audioBlocks vs. shared blocks array. §C.2's (a) vs (b). Start isolated (a); the visual engine's abilities story suggests (b)'s unified gesture plumbing may win — revisit after trim lands.
  • open Is the clip library undoable? Placement (blocks) clearly belongs in the undoable ProjectStore. Adding clip bytes is more like AudioStore's current ephemeral role — decide whether "load a file" is an undo step.
  • open Scrub-while-paused. Silent first (matches today). Grain-scrub is a genuine feature — a throttled short-lived player — deferred to phase 5.
  • later Audio-reactive visuals. Positioned audio is the substrate: a clip at a known beat window could feed analyzed amplitude/FFT into a visual-engine modulator port. Explicitly out of scope here — keep the audio track independent of core/engine/ for now.
