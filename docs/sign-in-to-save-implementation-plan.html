Sign-in-to-Save — Implementation Plan

The phased build order for the architecture: anonymous sessions born on first edit, autosave on existing rails, in-place conversion at signup, carry-over at login. Each phase ships alone and degrades to today's behavior if stopped.

Sequencing principle: the flag-off state is exactly today's app. Phases 0–2 make anonymous work persist; the feature is already user-visible value at that point. Phases 3–4 make the exit doors (signup/login) honor it. Phase 5 is hygiene. Stop anywhere and nothing is broken.

Phase 0 — Supabase dashboard (manual, 5 minutes)

  • Enable Authentication → Sign In / Up → Allow anonymous sign-ins.
  • Leave CAPTCHA off; note the default anonymous rate limit (per-IP) in case testing trips it.
  • Add NEXT_PUBLIC_ANON_SESSIONS=1 to .env.local and Vercel (the code ships behind this flag; unset = feature off).

Phase 1 — Auth plumbing (the isAnonymous bit everywhere)

1a · src/persistence/hooks/useAuth.ts

Expose the flag: return { user, loading, isAnonymous: user?.is_anonymous ?? false }. Every "is this a real person" branch in the app switches to user && !isAnonymous.

1b · src/persistence/anonSession.ts (new)

ensureSession(): returns the current user if any; otherwise signInAnonymously() (behind the env flag) and returns the new user; returns null on failure so callers fall back to in-memory mode. Single-flight (one in-flight promise) so double-triggers can't race two sign-ins.

1c · Identity-vs-persistence branches

Editor header (src/editor/App.tsx): the amber chip and hidden-Upgrade logic key on !user || isAnonymous; the Projects/Home link keys on the same. Checkout route (app/api/stripe/checkout/route.ts): reject user.is_anonymous with 403 "Create an account to upgrade".

useAuth.ts · anonSession.ts (new) · App.tsx · checkout/route.ts

Phase 2 — Adoption: first edit becomes a saved project

2a · The trigger

A small hook (useAnonymousAdoption, mounted next to useProjectPersistence) that, when there's no ?project bound and no permanent user, subscribes to the project store and fires once on the first meaningful mutation (tracks/blocks/audio change — not bpm scrubs alone).

2b · The flow

const user = await ensureSession()            // anon sign-in if needed; null → stay in-memory
if (!user) return
const doc = serialize()                        // the edit that triggered us is already in here
const p = await projectStorage.create(nameFor(doc), doc)   // 'Untitled' or the template's name
router.replace(`/editor?project=${p.id}`)      // existing persistence flow arms autosave

The rebind hydrates the just-saved document, so the user sees no change — except the save chip going to work.

2c · Template path joins the same rails

app/projects/page.tsx / template modal: signed-out template click currently routes to ?template=<id> demo mode. With the flag on, it instead runs ensureSession() → create(template.name, clone(document)) → ?project=<id>. Demo mode stays as the fallback when ensureSession() returns null.

2d · Copy change

Chip for anonymous users: "Saved on this device — sign up to keep it forever" (it IS saved now; the risk is losing the browser session, not the tab).

hooks/useAnonymousAdoption.ts (new) · useProjectPersistence.ts · App.tsx (chip) · projects/page.tsx

Ship gate for Phase 2: fresh incognito → "Try it out" → add a track → refresh the tab → the work is still there (URL has ?project=, autosave chip cycles). That single loop is the feature.

Phase 3 — Conversion at signup (same uuid, no migration)

3a · Signup page detects anonymous

app/(auth)/signup: when the current session is anonymous, swap signUp() for the conversion flow — updateUser({ email }) → "check your inbox" state → on confirmed, updateUser({ password }) → create the profiles row (first/last name from the form). Google button: linkIdentity({ provider: 'google' }) instead of signInWithOAuth.

3b · The already-registered email

updateUser fails if the email has an account → show "That email already has an account — log in instead" and hand off to the Phase-4 login carry so the work still survives.

3c · Post-conversion

Nothing to do — the uuid kept every row. Land on /projects; their anonymous project is simply there, owned.

app/(auth)/signup/* · profiles insert (mirror existing signup)

Test with email confirmation ON (production setting): the limbo between updateUser({email}) and the click-through is where this flow breaks if the UI loses the session or the user closes the tab. Abandoning mid-conversion must leave them a working anonymous session.

Phase 4 — Carry-over at login (existing account)

app/(auth)/login: if the current session is anonymous and owns a project, serialize the open document before signInWithPassword(), then after login create it in the real account (projectStorage.create(name, carried)). If the account is at the free one-project limit, prompt: replace the carry with a discard, or send them to pricing. The stranded anonymous user is left for cleanup.

app/(auth)/login/* · a small carryOver helper beside anonSession.ts

Phase 5 — Hygiene: cleanup & guardrails

  • Stale-user sweep (Supabase SQL editor now; pg_cron weekly later): 
    delete from auth.users
where is_anonymous is true and created_at < now() - interval '30 days';
    Projects cascade; storage objects don't (known fossils gap — same bucket sweep persistence already owes).
  • Checkout rejection test — anonymous POST to /api/stripe/checkout returns 403.
  • Rate-limit sanity — hammer ensureSession() in a loop locally; confirm graceful null-fallback when Supabase says no.

Verification checklist (manual, ~10 minutes end to end)

  1. Incognito → landing → Try it out → add a track + block → URL gains ?project= → hard refresh → work persists.
  2. Same session → template modal → pick a template → it saves as a project (no demo chip).
  3. Same session → sign up with a new email (confirmation ON) → confirm → land on /projects → the project is there → export works, watermark still applies (free).
  4. New incognito → build something → log in to the account from step 3 → carried project appears alongside; free-limit prompt fires if at cap.
  5. Anonymous session → attempt checkout via devtools → 403.
  6. Kill switch: unset NEXT_PUBLIC_ANON_SESSIONS → behavior is exactly pre-feature (in-memory + honest chip).

Junk anonymous users pile upLazy creation (first edit, not pageview) + 30-day sweep.
