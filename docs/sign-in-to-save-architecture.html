Sign-in-to-Save Architecture

Letting a stranger start creating with zero friction and never lose the work: anonymous Supabase sessions make logged-out visitors real (temporary) users whose projects autosave normally — and signing up later converts the same user instead of migrating data.

★ The big picture

The entire feature rests on one property of Supabase anonymous sign-ins: signInAnonymously() creates a real auth user — a real auth.users row, a real JWT, a real auth.uid() — just flagged is_anonymous: true.

That means everything we already built keeps working unchanged: RLS policies key on auth.uid(), so an anonymous user can insert a projects row they own; projectStorage and autosave don't know the difference; the free-tier one-project limit applies because they simply have no subscriptions row.

And the payoff: when they sign up, we don't migrate anything. Supabase converts the anonymous user into a permanent one — updateUser({ email }) for email/password, linkIdentity() for Google — and the uuid never changes. Every project row they created while anonymous is already owned by their permanent account. Ownership never moves; no copy step, no merge step, no orphaned data.

The one path that does need a data carry is logging into an existing account (§5) — you can't link an anonymous user onto an account that already exists.

1 · Where we are today

A logged-out visitor can already do real work: the landing page's "Try it out" goes straight to /editor, and ?template=<id> hydrates a full template into the in-memory stores. But nothing persists — useProjectPersistence only arms autosave when a ?project row is bound, and creating a row requires a signed-in user (projectStorage.create throws "Not signed in"). The amber header chip is honest about it: "Your work isn't saved — sign up to keep it."

The failure this design removes: someone spends 40 minutes on a visual, hits signup, and lands in a fresh empty account. Today the chip is a warning; after this, it becomes a promise.

2 · The primitive: anonymous sign-ins

  • Dashboard switch. Anonymous sign-ins are off by default — enable under Authentication → Sign In / Up → Allow anonymous sign-ins. Do it in both environments (there's only one Supabase project, so once).
  • One call. supabase.auth.signInAnonymously() — the browser client persists the session in localStorage exactly like a normal login; onAuthStateChange fires; useAuth sees a user.
  • The flag. The user object and JWT carry is_anonymous: true. This is the single bit every UI and server decision keys on (§3, §6).
  • Abuse limits. Supabase rate-limits anonymous sign-ups per IP out of the box (~30/hour). CAPTCHA can be added later if bots become real; don't pay that UX cost speculatively.

The key UI rule: anonymous is "signed in" for persistence and "signed out" for identity. useAuth grows an isAnonymous flag, and every surface that currently branches on user picks which meaning it wants: autosave wants the session; the signup chip, Upgrade button, and checkout want a permanent user.

4 · When the session is born (lazily)

decided Not on page load. Creating an auth user for every drive-by pageview manufactures junk rows and eats the IP rate limit. The session is created at the first moment there is something worth saving:

  • First real edit in a blank editor (first project-store mutation — add track, draw block, drop audio), or
  • Opening a template while signed out (the click already expresses intent to make something).

At that moment, one flow runs: ensureSession() (sign in anonymously if no session) → projectStorage.create(name, currentDocument) seeded with whatever is already in the stores → router.replace('/editor?project=<id>') → the existing useProjectPersistence flow arms autosave. From that point the anonymous user is on the exact same rails as a logged-in one.

Seeding create() with the current document (not emptyDocument()) is what makes the adoption seamless — the edit that triggered the flow is already in the snapshot, and the rebind hydrates back the same state. The ?template demo mode stays in the codebase as the fallback when anonymous sign-in is disabled or fails (§8).

5 · Conversion — the two exits from anonymity

5a · Signup (new account) — the happy path, zero data movement

The signup page detects isAnonymous and, instead of signUp() (which would mint a different user), converts in place:

await supabase.auth.updateUser({ email })      // sends the confirmation email
// after the user confirms, set their password on the same session:
await supabase.auth.updateUser({ password })

Google OAuth converts with supabase.auth.linkIdentity({ provider: 'google' }). Either way the uuid is unchanged, so their projects, and any future audio uploads, are already theirs. The profiles row is created at this point (it never existed for the anonymous phase).

Email verification is a real interruption: with "confirm email" on, the email isn't attached until the user clicks the link, and the password can only be set after. The signup UI must handle the "check your inbox" limbo without losing the session — the session is the account. If the email already belongs to an existing account, updateUser fails — surface "that email has an account, log in instead" and route to 5b.

5b · Login (existing account) — the carry-over path

An anonymous user who already has a Cabin Visuals account can't be linked — signInWithPassword() simply replaces the session, and the anonymous user's rows would be stranded. So the login page, when the current session is anonymous and a project row exists, carries the work across client-side:

  1. Serialize the open project's document (it's sitting in the stores) before authenticating.
  2. Log in normally.
  3. projectStorage.create(name, carriedDocument) into the real account (respecting the free-tier limit — if they're at it, offer replace-or-discard).
  4. The abandoned anonymous user becomes cleanup fodder (§7).

Scope note: the carry covers the open project — the only one an anonymous user can have under the free limit, so "carry everything" and "carry the open one" are the same thing today.

6 · RLS, billing & free-tier guardrails

  • RLS: nothing to change. All policies check auth.uid() = user_id; anonymous users pass them for their own rows and nobody else's. (If a table ever needs to exclude anonymous users, the JWT exposes is_anonymous for a policy check — none needed now.)
  • Billing: hard-block anonymous checkout. The checkout route adds a user.is_anonymous rejection (an anonymous Pro subscription would be attached to a discardable identity — never allow it). The header hides Upgrade for anonymous users, same as it does for visitors today.
  • Free tier applies unchanged: no subscriptions row → free → watermark, 720p, one project. The one-project limit conveniently caps anonymous storage abuse too.
  • Audio uploads: the storage bucket policies also key on auth.uid(), so anonymous users can upload a backing track; it converts with them. Counted toward cleanup (§7).

7 · Cleaning up abandoned users

Every bounce-after-one-edit leaves an anonymous user + one project row forever. Supabase's own guidance is a periodic delete; ON DELETE CASCADE on projects.user_id means the rows go with the user:

-- Supabase SQL editor, or a pg_cron schedule (weekly):
delete from auth.users
where is_anonymous is true
  and created_at < now() - interval '30 days';
Storage doesn't cascade. Audio files uploaded by deleted anonymous users stay in the bucket (the same "fossils" gap persistence already has). Acceptable now; fold into a future orphan-sweep that solves both.

30 days is generous — an anonymous session lives in one browser's localStorage, so a user who hasn't returned in a month has almost certainly lost the session anyway.
