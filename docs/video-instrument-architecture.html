Video Instrument — Architecture

An object instrument that plays the user's own video clips, cut by MIDI: upload several clips onto one track, and each note selects — and restarts — a clip, like a VJ pad bank. The entire design hangs off one mapping: the frame on screen is a pure function of the beat and the notes, which is what makes scrubbing, pausing, and export land on exactly the same pixels.

1 · What it is

A Video object instrument. The user uploads an ordered list of clips (the track's pad bank). MIDI notes drive playback: a note-on picks the clip at index (pitch − baseNote) mod clipCount and restarts it from zero; the clip latches — it keeps playing after the note ends — until the next note-on replaces it. Outside the track's blocks the instrument renders nothing, per the block-gated visibility rule. The video is drawn on a positioned plane (params for position/size), so movers, parenting, and effects compose with it like any other object.

2 · The time model — the one invariant everything hangs off

Every question about this instrument reduces to: what frame is on screen at beat B? The answer never consults a wall clock:

  • Active note — the latest note-on at or before B, inside block coverage (beatInBlock).
  • Clip — (pitch − baseNote) mod clipCount of that note.
  • Clip time — (B − noteBeat) × secPerBeat, wrapped modulo the clip's duration when loop is on, clamped to the last frame when off.
The keystone: the mapping (beat, notes) → (clip, clipTime) is the pause invariant applied to video. Live playback merely approximates it smoothly (a playing <video> element, drift-corrected); pause, scrub, and export enforce it exactly (seek). Nothing anywhere stores "what's playing" as state — it is recomputed from the beat every frame.

3 · Storage — the audio pipeline, mirrored

Bytes never enter the document. Clips upload to a private project-videos Storage bucket at {userId}/{projectId}/{clipId} — the same path scheme and RLS posture as project-audio (first folder must equal auth.uid()). The document carries only descriptors:

  • videoClips catalog — ref → { fileName, duration, width, height }, persisted alongside audioClips in the serialized document (additive; no schema bump).
  • track.videoRefs — the ordered ref list on the Video track. Order IS the pad bank: index 0 answers baseNote.
  • Session fallback — with no project row yet (fresh in-memory editor), bytes live behind an object URL for the session, exactly like audio's pre-save behavior.
Dashboard step (one-time): the project-videos bucket and its RLS policies must be created in Supabase — same policies as project-audio with the bucket name swapped.

4 · Rendering — warm elements, one plane

The component keeps one hidden <video> element per clip (muted, playsInline, preload="auto"), created when the track's refs resolve and kept warm so switching clips never waits on a network fetch. One plane carries a THREE.VideoTexture pointed at the active clip's element; aspect comes from the video, scale/position from params.

Per frame (useInstrumentFrame), the callback computes (clip, clipTime) from the time model, then reconciles the element to it:

  • Transport playing — the active element .play()s natively (smooth, hardware-decoded); if |element.currentTime − clipTime| exceeds ~90 ms, snap it back. The beat clock is the boss; the element is a follower.
  • Transport paused / scrubbing — every element paused; the active one is seeked to clipTime. Seeks are async, so on seeked the component requests one frame (invalidate()) — this is the same contract the render governor's demand mode expects from every late-arriving input.
  • No active note / outside blocks — the plane is hidden and all elements pause. Block-gated visibility, enforced.

5 · Export — frame-exact by construction

Live playback tolerates ~90 ms of drift; an exported MP4 must not. The export engine's frame walk is already asynchronous per frame (its encoder awaits are the loop's backpressure), so it gains one seam: frame preparers.

Cost: one seek per exported frame while a video is on screen — typically 5–30 ms — so exports of video-heavy sections run slower. Correct and slower beats fast and smeared.

Pause invariantFrame = f(beat, notes, params). Live drift is bounded and corrected against the beat clock; pause/scrub/export seek exactly.

Frame sourceWarm <video> per clip + VideoTextureOne shared element with src swaps (seek/network storm on every cut); WebCodecs VideoDecoder everywhere (frame-exact but a decoder pipeline per clip — the right tool later if drift correction proves insufficient, overkill now).
Note semanticsLatch: note-on cuts + restarts; holds until next note-onGate (clip only while note held) — makes short notes useless for video; latch matches VJ pad banks. A gate param can arrive later without schema changes.
