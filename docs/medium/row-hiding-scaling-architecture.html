Row Hiding / Scaling - Architecture

The piano roll gets a ROW MODEL: one derived list of visible rows, filtered (used / instrument-mapped / in-scale) and scaled, that every part of the editor - grid, gutter, hit-testing, note drawing - reads instead of "all 128 pitches." It's the display answer to MIDI import, the fix for "adding midi at long fixed distances," and the foundation midi-vim binds hotkeys to.

1 · One derived list, everything reads it

2 · The filters

FilterVisible rowsThe use case it serves
All (default)The full range, exactly today's editor.Status quo; always one keypress away back.
UsedPitches present in the block (± an octave margin option).Editing an imported full-spectrum file without scrolling through silence - the MIDI-import companion.
MappedPitches the track's instrument actually responds to.The 7-lane instrument problem: the editor shows 7 fat rows, "add midi at fixed distances even when those distances are long" solved by omission.
ScalePitches in scale(root, kind) - major/minor/pentatonic to start.Julia's "only show notes in a certain scale"; wrong notes become impossible to draw rather than possible to spot.

3 · Decisions & non-goals

DecisionWhy
View state, not documentFilters/scale never change notes - they live beside midiRowScale in UIStore (persisted per user via localStorage later if wanted; never in the project document, no schema bump).
Data is never filteredHidden-row notes keep playing and keep exporting; the editor shows a count chip and "reveal" jumps the filter to Used.
Row scale already existsmidiRowScale (0.5–2×) shipped with the editor - this item only adds the filter half and widens the scale range if 7-row editing wants chunkier rows (cap ~3×).
Designed as midi-vim's substrateVim motions ("next visible row", "toggle filter") bind to the row model's API, not to pixels - the two docs cross-reference; build this first.
Non-goalsFold-with-ghost-rows UI (Ableton-style collapsed markers) · per-project persisted filters · custom scales beyond a starter set.
The one-sentence mental model: make "which rows exist" a single derived list with a filter knob (all/used/mapped/scale) and route every y⟷pitch conversion through it - the data never changes, the view finally can, and midi-vim gets the coordinate system it needs.
