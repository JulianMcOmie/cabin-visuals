Export Ranges — Architecture

Export bars N..M instead of the whole project. The engine was built range-shaped on purpose — the stepper takes a start beat, the audio placement helper takes a fromBeat — so this feature is a dialog control plus threading one offset through code that already expects it.

1 · One offset, three consumers

2 · Decisions & non-goals

DecisionWhy
Range in BARS, inclusive-exclusive [from, to)Bars are the ruler's language; [from,to) makes "bars 1–9" mean 8 bars with no fencepost debates in the math.
Range lives in the dialog, not the documentSame rule as every export setting: an action parameter, remembered in localStorage, never persisted or undoable.
"Loop region" is a PRESET of this, laterWhen a transport loop region exists (it doesn't yet), it becomes a third radio that fills from/to — this feature deliberately doesn't wait for it and doesn't design it.
Whole-project remains the defaultThe range UI is a radio: Whole project (default) / Bars … – …; invalid ranges (to ≤ from, past totalBars) clamp inline rather than erroring at export time.
Non-goalsMultiple ranges/stitching · per-range filenames · exporting block-loop repeats specially (they're just notes; ranges slice through them like anything else).
The one-sentence mental model: the walk starts at a different beat and the audio placement is told the same beat — both were parameterized for this on the day they were written, so the feature is a from/to control and the discipline to keep file timestamps zero-based.
