Drag Highlights — Implementation Plan

Two independent landings: restyle the library-drag indicator (pure CSS-and-props), then the canary-safe 3D highlight overlay for the modulator target picker.

A · Scope

In scopeOut of scope
(a) louder insert line · label-column wash during library drags · (b) hover channel · HighlightOverlay outlines · canaryIgnore flag · shared scope-expansion helperPost-processing glow outlines · timeline-hover highlighting · touching instrument materials · persisting anything

B · Phase-by-phase

Phase 1 — Library-drag presentation timeline

  • Insert line: 1px → 3px indigo bar with 30% glow halo, full row width including the label column (today it can read as a hairline between borders).
  • Label-column wash: while trackDrop != null AND the drag source is the library (add a source: 'library' | 'track' field to the bridge state), the label column gets a faint indigo tint — "this is where drops live."
  • Empty/short list: when dragging with few tracks, render a dashed "drop to add a track" placeholder row at the list's end (the drop-below-everything case is currently invisible).
  • Verify: drag each library section (instrument/modulator/modifier/effect — effects highlight the TrackEditor drop zone instead, unchanged); indicator legible at every row height; nest-into ring unchanged.

Phase 2 — Hover channel + shared scope helper UIStore · resolve · TrackEditor

  • edit UIStore.ts: hoveredTargetTrackIds: string[] + setter (ephemeral).
  • edit core/visual/resolve.ts: export the scope→objectIds expansion as expandScope(scope, tracks, rootTrackIds) (promote the internal objectsForScope; resolve consumes the same export — one brain).
  • edit TrackEditor.tsx target picker: option rows set the hover list on pointerenter (expandScope of that option's scope), clear on leave and on picker close/unmount (a useEffect cleanup so a closed panel can never leave a stale glow).

Phase 3 — The overlay HighlightOverlay pauseCanary

  • new components/visual/HighlightOverlay.tsx (inside Canvas): for each hovered trackId with a live object, draw a wireframe box (or the object's bounding sphere) at the object's world transform, pulsing subtly via opacity — driven by the hover list, positioned from the engine's object state each frame it's visible.
  • The overlay group sets userData.canaryIgnore = true; edit pauseCanary.ts: the hash walk skips subtrees flagged canaryIgnore. One explicit, documented escape hatch — grep-able, reviewed, never default.
  • Verify: pause the playhead, hover options — objects outline correctly per scope (single / #tag / branch), the CANARY STAYS SILENT (the acceptance test for the whole design); unhover → outlines gone; hover during playback follows moving objects.

D · Risks & notes

  • risk canaryIgnore becoming a loophole — the flag's doc comment states the rule: UI chrome only, never instrument content; anything else found using it is a review reject.
  • risk Stale hover if the picker unmounts mid-hover — the cleanup-on-unmount in phase 2 is the guard; test by closing the panel while hovering.
  • note Group-modulation targets (tag/branch) can light MANY objects — cap the overlay at ~32 outlines and dim beyond (a tag on everything shouldn't build 200 wireframes).
The one-sentence mental model: turn up the volume on the drop indicator that already exists, and answer "what will this modulator hit?" with outline ghosts in a group the canary is explicitly told is furniture, not scene.
