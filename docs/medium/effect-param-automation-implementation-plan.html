Effect-Param Automation - Implementation Plan

End-to-end through five thin layers: type, menu, resolve, overlay, wrappers. The keystone canary is a radius lane visibly breathing a radial-symmetry ring under play, scrub, AND export.

A · Scope

In scopeOut of scope
ParamAddress type · targetEffect lanes · add-automation menu groups per instance · resolve + computeAtBeat overlay · wrappers read merged · not-found state on effect deletionEnvelope modulator + tag automation (consumers, own pairs) · automating enabled · modulator→effect routing · per-keyframe curves

B · Phase-by-phase

Phase 1 - Types + store types.ts · ProjectStore

  • edit types.ts: ParamAddress union (exported - the cluster's shared type) and Track.targetEffect?: { instanceId: string; key: string }. Optional → no upgrade step.
  • edit ProjectStore.ts: addEffectAutomationTrack(parentId, instanceId, key, label) - mirror of addAutomationTrack with the one-lane-per-address duplicate check (compare instanceId+key). removeEffect does NOT delete its lanes (they enter not-found and stay re-aimable - same philosophy as recursive-delete keeping data decisions explicit).

Phase 2 - Menu + lane view TrackContextMenu · PianoRollPanel

  • Menu: after the params group, one group per effect instance (label = plugin name, numbered when duplicated: "Echo 2"), items from the plugin def's numeric settings, added ones checked/disabled. (On the shared NestedMenu if the picker pair landed; inline otherwise.)
  • PianoRollPanel: the automation-info lookup branches - targetEffect lanes pull label/min/max from the plugin def's setting schema; missing instance → the not-found fallback (interp-discoverability phase 0).

Phase 3 - Resolve + overlay ★ keystone resolve.ts · VisualEngine.ts

  • Resolve: effect-targeted lanes compile like param lanes but keyed by address: the object gains effectLanes: Map<instanceId, Map<key, Lane>>.
  • computeAtBeat: sample and merge → state.effectSettings: Map<instanceId, Record<key, number>> (base settings spread first, lane values over them). Same debounced-resolve / fast-param-sync split applies: editing a base setting slider stays 60fps via syncParams.
  • edit wrappers (TransformWrapper · CloneWrapper · ShaderWrapper via ObjectRenderer): read state.effectSettings.get(instance.id) ?? instance.settings. One helper, three call sites.
The keystone canary: radial-symmetry radius lane 0→5 over 4 bars - the ring expands under play; park at bar 2 and the frozen radius equals play's radius at bar 2 exactly (canary green); export 4 bars and the file matches the editor frame-for-frame. If scrub ≠ play, the overlay leaked out of the pure path; stop and fix before the UI polish.

Phase 4 - Edge cases + verify

  • Delete the effect → lane shows not-found, keyframes intact; undo restores the pairing.
  • Two echoes, lanes on each - independent motion (instanceId addressing proven).
  • Base-setting slider drag while its lane plays: base changes, lane still wins during coverage - document this in the automated-badge affordance if the automation-copy-bug work landed its live marker (share the treatment).

D · Risks & open questions

  • risk Setting schemas must declare min/max/step for lane bounds - audit the 12 plugin defs; any unbounded numeric setting gets explicit bounds in the same commit.
  • risk Wrapper reads are the only place merged-vs-base can be confused - the one helper rule (never touch instance.settings directly in wrappers again) plus a lint-grep in review.
  • open Shader uniforms update per frame already? Confirm the shader wrapper re-uploads changed uniforms cheaply (it should - FBO pipeline is per-frame by nature).
  • note Sequence the symmetric-effects pair immediately after - it's this feature's demo and stress test.
The one-sentence mental model: five thin layers, each the twin of something that exists - a second target field, a second menu group, a second lane compilation, a second overlay map, and wrappers that prefer the merged value; the ParamAddress you mint in layer one is the cluster's inheritance.
