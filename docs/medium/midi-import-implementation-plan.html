B · The parser module

// core/midiImport.ts — pure: bytes in, document-shaped data out. No store access.
import { Midi } from '@tonejs/midi'

export interface ImportedTrack { name: string; notes: Note[]; endBeat: number }

export function parseMidiFile(bytes: ArrayBuffer): ImportedTrack[] {
  const midi = new Midi(bytes)
  return midi.tracks
    .map((t) => ({
      name: t.name || t.instrument?.name || 'MIDI track',
      notes: t.notes.map((n) => ({
        id: crypto.randomUUID(),
        pitch: n.midi,
        startBeat: n.ticks / midi.header.ppq,          // ticks/PPQ = beats — no seconds
        durationBeats: Math.max(0.05, n.durationTicks / midi.header.ppq),
        velocity: Math.round(n.velocity * 127),
      })),
      endBeat: 0, // filled below from the last note
    }))
    .filter((t) => t.notes.length > 0)
}

Pure and testable with a fixture .mid checked into the repo (a 2-track, 8-bar file with known note positions).

C · Phase-by-phase

Phase 1 — Parser + store action core/midiImport.ts ProjectStore

  • new core/midiImport.ts as above.
  • edit store/ProjectStore.ts: importMidiTracks(imported: ImportedTrack[]) — per track: new track (default instrument cube, color from the palette rotation, name from the file), ONE block at bar 0 with durationBars = ceil(maxEndBeat / beatsPerBar), notes block-relative (they already are — block starts at 0). Bump totalBars if the content overruns. One set() → one undo step for the whole import.
  • Verify with the fixture: node-side unit check of parse output (beat positions exact), then in-app: import → tracks appear, undo removes them all at once.

Phase 2 — Entry points timeline UI

  • Button: an import icon next to the Tracks + button — <input type="file" accept=".mid,.midi">, read bytes, parse, dispatch.
  • Drop: the timeline area accepts a dragged .mid (dragover/drop with type sniff) — same dispatch. Audio-file drops are NOT claimed (the AudioBar owns those).
  • Summary toast per import: "3 tracks · 412 notes" plus, per track whose instrument has a narrow map, "N notes outside range" (computed against the instrument's declared pitch mapping where one exists; skip the count where the instrument maps all pitches).

Phase 3 — Verify end-to-end

  • A Logic-exported 2-track .mid: both tracks land, first note of each at its exact bar:beat under the ruler, playback triggers visuals on the grid.
  • A file longer than 32 bars grows totalBars; undo restores the old length with the tracks.
  • A 200k-note stress file parses off the main thread's critical path (parse is fast; if it ever isn't, the seam is "parse in a worker" — noted, not built).
  • Pitch fidelity: reopen the block in the piano roll — full-spectrum notes visible, block previews show the contour.

E · Risks & open questions

  • risk Type-0 files (everything on one MIDI track, channels instead of tracks): v1 imports it as one dense track — acceptable; channel-splitting is a refinement if it annoys.
  • risk Files with pickup/offset silence: notes land where the file says (bar 3 stays bar 3). If people want "snap content to bar 1," that's a checkbox later.
  • open Default instrument per imported track — Cube for all, or cycle the library? Cosmetic; decide at build time.
  • note CC lanes (mod wheel etc.) are parsed by @tonejs/midi and deliberately dropped in v1 — the natural target is the envelope/automation system once richer targets exist; leave a TODO naming it.
The one-sentence mental model: a pure ticks-over-PPQ walk mints ordinary tracks/blocks/notes in one undoable store write, reachable from a button and a drop — and every hard-sounding problem (tempo, range, display) was already dissolved by the beat domain and the import-faithfully decision.
