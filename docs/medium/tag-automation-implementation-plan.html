Tag Automation - Implementation Plan

A parentless automation track with a tag scope and a normalized lane, fanned out at resolve. Third consumer of the ParamAddress cluster - by now the plumbing has grooves.

B · Phase-by-phase

Phase 0 - Gate dependency

  • ParamAddress merged (effect-param-automation pair). Envelope modulator NOT required.

Phase 1 - Data + creation types · ProjectStore

  • edit types.ts: automation tracks gain optional targetTag?: string + targetAddress?: ParamAddress (parentless when targetTag is set). Optional fields → no upgrade.
  • edit ProjectStore.ts: addTagAutomationTrack(tag, address, label) - root track (below the pinned audio track, as ever), name #tag · label; duplicate rule per (tag, address).
  • Creation UI: the Tracks + gains a small menu (Track · Tag automation ▸) - NestedMenu again; tag list from the union of all track.tags, then param keys = union of tagged instruments' numeric params (plus universal Opacity when that ships), then effect settings by pluginId present on tagged tracks.

Phase 2 - Lane editing PianoRollPanel

  • Tag lanes are NORMALIZED: automation rows render 0–100% (generateAutomationRows with 0..1 bounds and a % label); the range gutter shows #drums · Base Color · drives 2/3.
  • Everything else (keyframes, interpolation, per-keyframe curves when they land) behaves identically - it's the same lane machinery with different bounds.

Phase 3 - Resolve fan-out ★ keystone resolve.ts · VisualEngine.ts

  • Resolve: for each tag lane, expandScope({kind:'tag'}) → object ids; for each target whose instrument/effect HAS the address key, attach the lane as a normalized source; record skipped targets for the UI count.
  • computeAtBeat: sample once per tag lane per frame (not per target), map t into each target's own min/max, write into the same overlay slot per-instrument lanes use - with the rule instrument-specific lane wins over tag lane when both cover the same address (resolve orders them; the overlay write for specific happens after general).
  • Verify: three #drums instruments, hue tag-lane 0→1 sweep → all three sweep their own hue ranges in lockstep; give ONE of them its own hue lane → that one follows its own, the others follow the tag; tag a fourth track mid-edit → it joins on the next resolve tick.

Phase 4 - Verify the tag lifecycle

  • Untag a track → it drops out (next resolve). Delete the tag from every track → the lane shows "drives 0/0" (not an error). Rename… tags aren't entities, they're strings on tracks - renaming means retagging; the 0/0 state covers the orphan case.
  • Scrub/park/export identical (the fan-out is resolve-time data; sampling is pure).
  • Persistence round-trip; undo of the creation removes the root track cleanly (recursive-delete rules apply - it has no children).

D · Risks & open questions

  • risk Key collisions with different meanings ("baseHue" on one instrument, semantically different elsewhere) - proportional mapping keeps it sane, the drives-N/M readout keeps it visible; per-target excludes are the future knob if it bites.
  • risk Ordering bugs between specific and tag lanes - one rule, one place, one test (phase 3's second verify case) that must be in the landing checklist.
  • open Where tag-automation rows SORT at root (top? grouped? below objects?) - cosmetic, decide at build with Julia.
  • note Modulator tag-scopes and this feature share expandScope - any tag-semantics change (hierarchies, renames-as-entities) lands in one helper for both.
The one-sentence mental model: a root lane that samples once and stretches its 0–1 into every tagged instrument's own bounds via the cluster's addressing - with exactly one precedence rule (specific beats general) standing between this feature and merge-order soup.
