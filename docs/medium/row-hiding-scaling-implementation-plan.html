Row Hiding / Scaling - Implementation Plan

Phase 1 is the whole risk: route every y⟷pitch conversion through one row model. After that, filters are list comprehensions and the toolbar is a dropdown.

B · The row model

// components/midi/rowModel.ts - pure
export type RowFilter =
  | { kind: 'all' }
  | { kind: 'used'; marginOctaves?: number }
  | { kind: 'mapped' }                        // instrument's declared pitch map
  | { kind: 'scale'; root: number; scale: 'major' | 'minor' | 'pentatonic' }

export interface RowModel {
  rows: Row[]                                  // visible, top-first (existing Row shape)
  indexOfPitch(pitch: number): number | -1     // -1 = hidden
  pitchAtIndex(i: number): number
  hiddenNoteCount(notes: Note[]): number
}
export function buildRowModel(filter, ctx: { notes; instrumentMap? }): RowModel

Automation lanes are exempt: their value-rows already come from generateAutomationRows and never filter.

C · Phase-by-phase

Phase 1 - Route the coordinate system ★ keystone midi editor

  • Introduce rowModel.ts with filter fixed to all - output identical to today's generateRows.
  • Replace every direct y⟷pitch computation with the model: yToRowIndex → model.pitchAtIndex(y / rowHeight | 0); pitchToRowIndex → model.indexOfPitch. Call sites: note draw (right-click), move (pitch delta must step through VISIBLE rows, not raw semitones), resize hit-tests, marquee bounds, scroll-to-first-note, the drawing preview.
  • Verify: with filter=all, the editor is behaviorally identical (this phase must be a no-op - that's what makes it safe to land alone).

Phase 2 - Filters rowModel · toolbar

  • Implement used/mapped/scale comprehensions. "Mapped" needs instruments to DECLARE their pitch map - add an optional pitchMap?: number[] | {min,max} to the instrument def type; instruments without one simply don't offer the filter.
  • Toolbar: a Rows dropdown (All · Used · Mapped · Scale ▸ root+kind) next to the Curve control; the hidden-note chip ("3 hidden") appears when >0, click = switch to Used.
  • Note-move semantics across hidden rows: dragging a note vertically moves it to the next VISIBLE pitch (that's the "fixed long distances" feature working); notes cannot be dropped onto hidden pitches.
  • Verify: mapped filter on a narrow instrument shows exactly its lanes; scale filter refuses off-scale drawing; switching filters never moves or deletes a note.

Phase 3 - Scale polish UIStore

  • Widen midiRowScale cap to 3× (7 fat rows should fill the panel); remember filter + scale in localStorage keyed globally (not per project).
  • Verify: 7-row mapped view at 3× fills the editor height; reload restores the view prefs.

E · Risks & open questions

  • risk A missed y⟷pitch call site corrupts edits only when rows are hidden - phase 1's "identical under all" gate plus a phase-2 sweep-test (draw/move/marquee under each filter) is the defense.
  • risk Vertical drag semantics: visible-step movement is right for hidden rows but changes muscle memory under Scale filter (a whole-tone drag ≠ semitone) - this is the feature, but confirm the feel with Julia early.
  • open Which instruments get pitchMap first - audit the drum-style ones when building phase 2.
  • note midi-vim consumes RowModel as its motion space; keep the API surface small and stable.
The one-sentence mental model: land a no-op row model first so every gesture speaks "visible row index," then filters are one comprehension each - and the scary part (hidden rows corrupting edits) was retired before any row ever hid.
