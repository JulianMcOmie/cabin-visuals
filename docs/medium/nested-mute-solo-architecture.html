2 · The three questions ★

QuestionOptionsRecommendation (pending Tyler)
Does parent MUTE cascade?(a) No — today's behavior. (b) Yes — muting a group silences the subtree, children remember their own state.(b). It's what every DAW does and what grouping is FOR. Render the inherited state differently (dim amber) so a child shows "muted because of my parent," like Logic's dark-vs-light mute.
Does parent SOLO include the subtree?(a) Solo = this object only. (b) Solo = this subtree — the group performs together.(b). Soloing a branch you built as a unit should show the unit. Implementation is "effective solo = own solo OR any ancestor's."
What does a CHILD's solo mean?(a) Global, same as any solo (isolates that child against the whole project). (b) Scoped: isolates only among its siblings, parent context keeps playing.(a) for now — one rule everywhere, no hidden scoping. (b) is the power-user feature to revisit if Tyler asks for in-group auditioning.

3 · The shape of the implementation (once decided)

All three answers become one pure pass at resolve time: walk each track's ancestry once and compute effectiveMuted / effectiveSolo, then feed those to the exact two places the flat flags feed today — resolve's objectOff for visuals, and the audio engine's audibility fold. No per-frame work, no schema change (the document keeps storing only the user's own clicks; effective state is derived).
