B · The evaluator — closed-form, scrub-exact

// instruments/modulators/Envelope.ts — value at `beat` from gate notes alone
export function evaluateEnvelope(gates: ResolvedNote[], beat: number, p: ADSR): number {
  let v = 0
  for (const g of gates) {                       // overlaps: max wins (documented)
    const dur = g.durationBeats
    const t = beat - g.beat
    let e = 0
    if (t < 0) { /* future gate */ }
    else if (t < p.attack)              e = t / p.attack
    else if (t < p.attack + p.decay)    e = 1 - (1 - p.sustain) * ((t - p.attack) / p.decay)
    else if (t < dur)                   e = p.sustain
    else if (t < dur + p.release)       e = p.sustain * (1 - (t - dur) / p.release)
    v = Math.max(v, e * velocityScale(g))
  }
  return v                                        // 0..1 — no state, pure f(beat)
}

Segments in beats; zero-length A/D/R guarded with epsilon. The same closed-form-over-notes discipline the pause-invariant push established.

C · Phase-by-phase

Phase 0 — Gate dependency

  • effect-param-automation merged (ParamAddress exists, wrappers read merged settings). Tag automation is NOT required — scope reuse is independent.

Phase 1 — Universal opacity visual engine · ObjectRenderer

  • Object state gains opacity (default 1, sourced like any param so automation lanes can also drive it). ObjectRenderer applies it at the placement group: cache the subtree's materials on mount, per frame set material.opacity = base·value and toggle transparent when < 1; skip work when unchanged.
  • Add "Opacity" to the add-automation menu as a universal entry (works before the envelope exists — an immediately useful side-product).
  • Verify: opacity automation lane fades a Cube in/out under play/scrub/export; canary green (it's beat-driven); materials restore at 1.

Phase 2 — The modulator + routing data Envelope.ts types · resolve · matrix path

  • new instruments/modulators/Envelope.ts: def (signal 'envelope', ADSR+depth params, library entry) + evaluator above.
  • edit types.ts: Routing gains optional target?: ParamAddress and blend?: 'multiply' | 'add' — port routings unchanged (additive optional fields, no upgrade).
  • edit resolve/engine: param-targeted routings compile into per-object modulator bindings; computeAtBeat merges in the documented order — base ← automation ← modulator (multiply/add per routing, clamped to the param's bounds).

Phase 3 — Routing UI TrackEditor

  • For envelope tracks, the target picker grows a second dimension: WHAT (Opacity · instrument params · effect settings — a ParamAddress picker grouped like the add-automation menu) × WHERE (the existing track/tag/branch scope) + blend toggle + depth slider (routing.amount).
  • Pulse's picker unchanged (its port is fixed by the def).

Phase 4 — Verify (the demo IS the test)

  • The headline: Envelope (A 0.1 · D 0.2 · S 0.7 · R 0.5) → Opacity on #drums; notes on the envelope lane make every tagged instrument breathe in sync; park mid-release shows the exact frozen level; export matches.
  • Blend check: multiply on scale vs add on Base Y — both behave, bounds clamp.
  • Overlap check: two overlapping gates → max envelope, releases don't pop.

E · Risks & open questions

  • risk Material traversal cost / shared materials: instruments sharing a material instance would fade together — the cache walk must clone materials on first opacity-touch (three.js material.clone per object subtree, once). This is the plan's fiddliest line; budget for it.
  • risk Merge-order confusion once three writers exist — the order lives in ONE function with the doc comment, and the automated-badge UI should eventually show "driven by: lane + envelope".
  • open Retrigger semantics beyond overlap-max (hard retrigger from zero?) — ship max, ask Julia after she plays with it.
  • note LFO is this file with a different evaluator — the def/plumbing built here is deliberately reusable for it.
