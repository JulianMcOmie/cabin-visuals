Drag Highlights - Architecture

Two affordances sharing one theme - show where an action will land: (a) louder drop-target feedback when dragging instruments from the library, (b) highlight objects in the 3D view while hovering modulator-target options. (b) crosses into the renderer and collides with the pause canary - that collision is the only interesting design here.

1 · (a) Library drag - amplify the channel that exists

2 · (b) Modulator-target hover - a highlight channel into the renderer

Hovering "#drums" or "Cube 2 (branch)" in the target picker should light the corresponding objects in the 3D view. The data is easy - the picker already computes each option's scope; expanding scope→trackIds is objectsForScope's exact logic. The design question is how a UI-hover may touch the scene without lying to the pause canary, which hashes the scene while paused and screams when anything changes without a beat or document edit.

OptionVerdict
1 · Overlay objects - render separate outline meshes in a group the canary skipsChosen. A HighlightOverlay component reads UIStore.hoveredTargetTrackIds, draws wireframe/outline boxes at the targets' world transforms, inside a group flagged userData.canaryIgnore. Object materials are never touched - no restore bugs, no canary lie.
2 · Emissive boost on the real materials + canary reset on hover changeRejected: mutating instrument materials from outside useInstrumentFrame breaks the "instruments own their appearance" rule, needs exact restore, and teaches the canary to accept UI-driven scene changes - a precedent that erodes the invariant.

3 · Decisions & non-goals

DecisionWhy
Overlay, never material mutationThe pause invariant's enforcement (canary) stays airtight; the overlay group is declared UI chrome via canaryIgnore - one explicit, greppable escape hatch instead of ad-hoc exceptions.
Hover state in UIStore, ephemeralhoveredTargetTrackIds: string[] + setter; set on option pointerenter, cleared on leave/close. Never persisted, never undoable.
Scope expansion reuses resolve's brainPromote objectsForScope's logic to a shared helper so the picker's hover and resolve's routing can't disagree about what "#drums" means.
Both features are view-onlyNo document change, no engine change, no schema - the canary-ignore flag in the renderer's hash walk is the single engine-adjacent line.
Non-goalsHover-highlight from the TIMELINE (selecting tracks already communicates) · glow/post-processing outlines (wireframe box first - cheap and legible) · highlighting modifiers/automation (they have no scene objects).
The one-sentence mental model: (a) is restyling a signal that already flows; (b) is a hover-driven overlay of outline boxes in a canary-ignored group - the scene's real objects are never touched, so the invariant machinery never has to learn about UI moods.
