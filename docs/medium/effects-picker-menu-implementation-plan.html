Effects Picker Menu — Implementation Plan

Extract the nested-menu shell into a shared component (it's about to have three consumers), generate its items from the effects registry, and mount it behind an "+ Add effect" slot. Small, but it pays down the one-off-menu debt on the way through.

A · Scope

In scopeOut of scope
Shared NestedMenu extraction · registry-driven items · "+ Add effect" slot + right-click on the effects listMenu search · effect reordering · removing drag-in · greying duplicates (allowed by design)

B · Phase-by-phase

Phase 0 — Extract the shared menu components/NestedMenu.tsx

  • Lift TrackContextMenu's shell into a generic NestedMenu({ x, y, groups, onPick, onClose }): fixed positioning with the flip-up/clamp measurement, hover-open submenus via the existing SubMenu (viewport-safe), outside-click/Escape close. Data shape: groups: { label, items: { id, label, icon?, disabled?, checked? }[] }[].
  • Re-implement TrackContextMenu ON the shared shell (its two groups + one-lane-per-param disabling map cleanly). Zero behavior change — verified against the flip-up tests that already exist as manual checks.
  • This is the debt payment: automation menu, effects picker, and the future interp Curve menu become three data sets over one component.

Phase 1 — The slot + the menu TrackEditor.tsx

  • Effects tab: after the instance list, an "+ Add effect" row (dashed border, plus icon — reads as an empty Logic slot). Click → NestedMenu at the slot's rect with groups built from PLUGIN_LIST grouped by category (Transform · Clone · Shader, fixed order).
  • Pick → addEffect(track.id, pluginId) — the same action drag-in calls; new instance appears expanded (its settings open) so the add flows straight into tweaking.
  • Right-click anywhere in the effects list opens the same menu at the cursor.

Phase 2 — Verify

  • Every category and plugin appears; picking each adds a working instance (spot-check one per category renders its wrapper: transform moves, clone multiplies, shader posts).
  • Menu opened near the panel bottom flips up; submenus clamp (the SubMenu behavior, inherited).
  • Registry regression: add a dummy plugin to the registry in a scratch branch → it appears in library AND menu with no other edits.
  • Automation menu unchanged after the extraction (right-click a track label: same items, same disabling, same flip behavior).

D · Risks & notes

  • risk The extraction regressing the automation menu — it's the only consumer with disabled+checked states; port its tests-by-hand (added items greyed with checkmarks) explicitly.
  • note Sequencing: land this AFTER the menu flip-up fix is committed (it builds on SubMenu) — already true on dev.
  • note The interp-discoverability phase 2 (Curve right-click) should be built on NestedMenu too — if that ships first, extract here anyway and migrate it.
