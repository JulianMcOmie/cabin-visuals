Per-Keyframe Curves - Implementation Plan

Engine first (two lines + tests-by-hand), UI second (segment hit-test + curve menu + override glyphs). The lane's Curve control survives as the default.

B · Phase-by-phase

Phase 1 - Data + engine types.ts · automation.ts · resolve

  • edit types.ts: Note.curve?: InterpolationMode. Optional → no upgrade; cloneNote spreads → copy/paste inherits.
  • edit core/visual/automation.ts: sampleLane resolves the mode per segment - left.curve ?? laneMode; step's special-casing unchanged.
  • Resolve passes keyframes through with curve intact (they already ride as notes - confirm the lane compilation copies the field).
  • Verify by hand-writing curves into two keyframes via a temp store call: one lane, three segments, three visibly different shapes on a position param; scrub ≡ play; canary green; save/reload round-trips the field.

Phase 2 - Segment UI midi editor

  • Hit-test: in automation lanes, a right-click that lands between two keyframes (x between their beats, anywhere in the lane's value area) selects that segment; on a keyframe = the segment it owns (to its right).
  • Menu: NestedMenu with the 7 modes + Use lane default (clears curve); the active mode checked. Multi-selected keyframes: apply to all their owned segments.
  • Glyph: overridden segments draw a small mode marker at the segment midpoint (a 10px curve icon or 3-letter tag); no glyph = following the lane default. The lane's toolbar Curve control relabels to "Default curve".
  • Deletion semantics for free: removing a keyframe removes its owned curve with it; the merged segment uses the surviving left keyframe's curve.

Phase 3 - Verify

  • Author the classic shape: slow-in rise (ease-in), hard cut (step), springy fall (exponential) on one lane - reads correctly in motion and parked.
  • Change the lane default → only non-overridden segments change.
  • Copy the lane to another instrument (copy-paste-automation pair) → curves travel.
  • Export: the eased motion in the file matches the editor (free, but look once).

D · Risks & notes

  • risk Right-click collision: right-click in automation lanes currently DRAWS a keyframe - the segment menu must only claim clicks that land clearly between keyframes vertically near the curve path, or move segment-menu to a modifier (Alt+right-click) if disambiguation feels bad. Prototype the feel first; this is the one UX risk.
  • risk Glyph clutter on dense lanes - cap glyph rendering below a keyframe-density threshold (they're overrides; dense lanes are usually uniform anyway).
  • note Tier 2 (drag-to-bend) would replace the mode enum with a curvature scalar - if Julia asks for it, that's the moment to design a real curve model rather than extending this one.
The one-sentence mental model: the engine change is "ask the left keyframe first"; everything else is teaching the lane to right-click a SEGMENT - and the draw-vs-menu click arbitration is the only part worth prototyping before committing.
