Export Ranges - Implementation Plan

Thread startBeat through the walk and the audio render, add a from/to control to the dialog, and verify with the onset-on-the-exact-frame canary shifted by the range start.

A · Scope

In scopeOut of scope
Range in ExportSettings (localStorage-remembered) · timebase from range · walk offset · audio fromBeat · dialog radio + bar inputs · clampingTransport loop region (the future preset) · multi-range · range selection by dragging on the ruler (nice later; inputs first)

B · Phase-by-phase

Phase 1 - Engine threading core/export

  • edit types.ts: range?: { fromBar: number; toBar: number } on ExportSettings; makeTimebase gains the range (computes startBeat + rangeBeats; whole-project when absent).
  • edit exportEngine.ts: walkFrames computes beat = timebase.startBeat + i·bpm/(60·fps); runExport passes timebase.startBeat into renderAudioTrack.
  • edit audioRender.ts: renderAudioTrack(…, fromBeat) → blockPlacement(block, fromBeat, …) (was hardcoded 0); duration = the range's seconds.
  • Verify headless (temp expose, the export test recipe): bars 2–4 of a 4-bar project → file duration exactly 2 bars' seconds; a clip at bar 0 straddling bar 2 enters mid-clip at the correct sample offset (compare against a full export's bar-2 audio).

Phase 2 - Dialog ExportDialog.tsx

  • Radio: Whole project (default) / Bars [from]–[to]. Number inputs 1-based for display, converted to 0-based [from,to) internally; clamp live to [1, totalBars] and to ≥ 1 bar length; show the resulting duration ("= 8 bars · 16.0s") so the math is visible.
  • Remembered with the other quality settings; "Bars" with a stale out-of-range memory falls back to Whole project silently.
  • Estimated size/time in the progress view already derives from frameCount - no change needed (it shrinks with the range automatically).

Phase 3 - Verify end-to-end

  • The shifted keystone canary: notes on every bar line, export bars 3–5 → the FIRST frame of the file shows bar 3's onset, and each subsequent bar-line onset lands at exactly (bar−3)·beatsPerBar·60/bpm·fps frames in.
  • Audio alignment: tone starting at bar 2, export bars 2–4 → file audio starts at 0:00 sample-exactly.
  • Cancel/error paths unchanged (they're range-agnostic); playhead untouched (beat override, not the store).

D · Risks & notes

  • risk Timestamp bleed: the one way to botch this is letting startBeat leak into VideoFrame/AudioData timestamps - they are file-local and stay index-based; the phase-3 canary catches it (video and audio would drift apart by exactly the range offset).
  • note When the transport loop region ships, add the third radio filling from/to from it - two lines, designed for here, built then.
  • note Ruler-drag range selection is the gestural upgrade; the inputs establish the semantics first.
The one-sentence mental model: shift the beat the walk asks about, hand the same shift to blockPlacement, keep every media timestamp starting at zero - the seams were pre-cut, this feature just uses them.
