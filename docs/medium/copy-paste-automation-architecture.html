2 · The re-target rules

Case at paste timeBehavior
Target instrument has the SAME param key (e.g. baseXPosition → baseXPosition)Paste silently, keyframes intact. Values are stored in normalized pitch-encoding, so differing min/max ranges map proportionally for free - a 0→max sweep stays a 0→max sweep.
Param key missing on the targetPaste and mark the lane "param not found" (the interp-discoverability work adds this state) with a param picker in the lane header - the user re-aims it in one click. No modal at paste time.
Target already automates that paramPaste is a no-op with a toast ("Base Size is already automated on Cube 2") - matches addAutomationTrack's existing one-lane-per-param rule.
No instrument selectedFall back to today's root paste? No - block with a toast ("select an instrument to paste automation onto"). An orphan lane is never the right outcome.

3 · Decisions & non-goals ★

DecisionWhy / status
Reuse the track clipboardCopy already captures everything (blocks→notes as keyframes, targetParam, interpolation). Only paste branches.
Proportional value mapping across rangesKeyframes are pitch-encoded against AUTOMATION_PITCH_MIN/MAX, decoded against the target param's min/max - cross-instrument paste inherits sensible scaling with zero math added.
★ Confirm the spec with TylerThe above assumes "duplicate a lane onto another instrument." If he meant copying keyframe selections between existing lanes, that's a piano-roll feature instead (notes copy already works inside a lane) - cheaper, different files. One question settles it.
Alt-drag laterDrag-a-lane-onto-another-track is the gestural version of the same operation; do keyboard first, reuse its re-target step for the drag.
The one-sentence mental model: the clipboard already carries the whole lane - teach paste that "automation + selected instrument" means re-parent under it and re-aim targetParam, with the three edge cases (missing param, duplicate lane, no selection) each getting a one-line answer.
