This plan introduces one new directory — src/editor/core/engine/ — plus a small renderer under components/visual/, and reshapes instruments/. The editor (timeline, piano roll, stores) is barely touched: the engine is a read-only observer of the project document. Every section names the files it creates / edits / deletes so the file-level mental model is always in view.

A · Architectural components — what gets added, and one layer down

The highest level first, then a peek at the key function / effect inside each. Files in monospace; status badges: new edit delete.

1 · The visual engine new core/engine/VisualEngine.ts

Adds a stateless engine that turns the project into per-frame object states. It owns the current resolved graph and a per-frame map of object → finalProps. One layer down:

  • a setProject(project) that kicks off a deferred, incremental re-resolve (it diffs which tracks changed and recomputes only those);
  • a computeAtBeat(beat) that binary-searches active notes, evaluates each modulator's output(beat), then runs the matrix to fill every object's portValues + finalProps;
  • a getObjectState(id) pull API the renderer calls each frame (returns the last computed finalProps).

2 · The resolver new core/engine/resolve.ts · core/engine/trackTypes.ts

Adds a pure function that flattens the project tree into a "resolved graph." Deeper:

  • resolveProject(project) — flattens the track tree, expands looping blocks, runs the event-modifier combine chain, splits instances into objects vs modulators, and builds the port → [routing] and tag → [objectId] indices;
  • resolveTrack(track) — memoized per track; returns its cached resolved output unless the track's inputs changed (detected by reference equality);
  • combine(parent, self) functions in trackTypes.ts — the event modifiers (suppress / mute / transpose / gate / rhythm / swing…), ported almost verbatim from Tyler.

3 · The per-frame sync new core/engine/VisualBeatSync.tsx

Adds an R3F component, mounted once inside the Canvas, that drives the engine. Deeper:

  • a useFrame that runs first each frame: reads currentBeat from TimeStore and calls engine.computeAtBeat(beat) so every object reads fresh state;
  • a module-level useProjectStore.subscribe(...) (registered on import) that debounces and calls engine.setProject — resolve runs off the edit's critical path, so editing never blocks.

4 · The modulation matrix new core/engine/matrix.ts

Adds the per-frame pass that turns modulator outputs into port values, then final props. Deeper:

  • runMatrix(graph, modulatorOutputs) — for each object, for each port: gather routings that hit it (direct + via its tags), combine per the port's rule (add/multiply/max/replace), clamp, then call the instrument's resolveProps(params, portValues) → finalProps.

5 · Built-in modulators & automation new core/engine/modulators.ts · core/engine/automation.ts · instruments/modulators/Pulse.ts

  • modulators.ts — the evaluate(ctx) → number | Color machinery; Pulse.ts is the first def (the Cube's inline decay math, generalized to drive any scalar port);
  • automation.ts — extractKeyframes(track) (note pitch → param value) and sampleLane(lane, beat) (binary-search + interpolate: step / linear / ease).

6 · The instrument model edit instruments/types.ts · new instruments/registry.ts · edit instruments/Cube.tsx

  • types.ts gains ObjectInstrumentDef (params · ports · abilities · render), ModulatorInstrumentDef, PortDef, AbilityLaneDef, RenderSpec;
  • registry.ts — getObject(id) / getModulator(id) over a map of defs (the library);
  • Cube.tsx becomes an ObjectInstrumentDef with ports (scale, energy, hue); its per-frame project-scan is deleted (the engine drives it now).

7 · The renderer new components/visual/VisualScene.tsx · ObjectRenderer.tsx

  • VisualScene — subscribes to the resolved graph's object list (re-renders only when objects appear/disappear, i.e. on resolve) and maps each to an <ObjectRenderer>;
  • ObjectRenderer — a useFrame that pulls engine.getObjectState(id) and applies finalProps to the mesh imperatively (no React re-render per frame — exactly today's Cube pattern, but fed by the engine).

8 · Data-model extensions & app wiring edit types.ts · store/ProjectStore.ts · App.tsx

  • types.ts — Track.tags: string[], a Routing type, an instrument kind discriminator;
  • ProjectStore.ts — actions setTrackTags, setRoutings (and later addModulatorTrack); no resolve logic lives here;
  • App.tsx — the Canvas swaps <Cube/> for <VisualScene/> + <VisualBeatSync/>.

B · Scope & the phase ladder

This plan is incremental and always-shippable: each phase leaves a working app. The order mirrors §6 of the architecture doc — but here every phase is pinned to files. Phases 1–3 are the foundation (and prove the whole thesis); 4–9 are additive. Status: phases 1–4 are built; phase 5 (nested hierarchy) is next.

In scope here: the data model, the engine module, the renderer, the instrument model, and a phased build down to the matrix (phase 3) in detail, with file lists for all phases. Deferred (named, not detailed): the Web-worker/OffscreenCanvas escalation (we settled on incremental + deferred resolve instead), and English→spec (phase 10) which only becomes viable once the RenderSpec is stable.

C · Data model — the types and where they live

Two kinds of types: the document (what the user edits, in the stores) and the resolved graph (what the engine derives, never persisted). Keeping them separate is what lets resolve be a pure, cacheable function.

C.1 — Document types edit types.ts edit instruments/types.ts

// src/editor/types.ts — extend the existing Track
interface Track {
  id; name; instrumentId; params?; color; muted; solo; blocks;   // (today)
  parentId?; childIds;            // nested hierarchy — becomes functional in phase 5
  tags?: string[];               // ✓ phase 4 — cross-cutting modulation targeting (shipped optional)
  targets?: Routing[];            // ✓ modulator-only: a LIST of Routings → one modulator drives MANY ports/groups
  lanes?: Record<string, Block[]>;  // ✓ phase 6 — abilities, keyed by AbilityLaneDef.key; a PARALLEL structure, NOT child tracks (shipped as Record, not the flat Block[] drawn here)
}
interface Routing {                // ✓ shipped (phase 4) exactly as below
  port: string;
  scope: { kind: 'track'; id } | { kind: 'tag'; tag } | { kind: 'subtree'; id };
  amount: number;
}

The instrument defs (ObjectInstrumentDef / ModulatorInstrumentDef / PortDef / AbilityLaneDef / RenderSpec) extend instruments/types.ts — they already host today's InstrumentDef + ParamDef, so this is the natural home and keeps the registry next door.

C.2 — Resolved-graph types new core/engine/types.ts

interface ResolvedGraph {
  objects: ObjectInstance[];        modulators: ModulatorInstance[];
  routingsByPort: Map<string, Routing[]>;   tagIndex: Map<string, string[]>;
}
interface ObjectInstance {
  id; def: ObjectInstrumentDef; params: Record<string,number>;
  abilityEvents: Map<string, Event[]>;  tags: string[];
  portValues: Record<string,number>;  finalProps: Record<string, unknown>;  // filled per frame
}
Mental model: types.ts = "what the user owns," core/engine/types.ts = "what the engine derives." If you ever see the engine importing the document types that's fine; the document never imports the graph types — the dependency points one way (engine → document), which is what keeps the editor independent of the engine.
Shipped shape (through phase 4): the block above is the north-star target (it includes phase-6/8 fields). What's built today: ResolvedObject = { trackId; instrumentId; muted; params; ports; notes; tags } — no abilityEvents (phase 6) or finalProps (phase 8) yet. And ModulatorInstance was split into a pure signal ({ id; kind; triggers }) plus a separate ResolvedRouting ({ modulatorId; targetObjectId; targetPort; amount }); routingsByPort buckets the latter. So one signal is evaluated once per frame and fanned out to every object its scopes resolved to. tagIndex (tag → objectIds) is built in resolve.ts exactly as drawn.

D.1 — resolve.ts & incremental caching

Because the store is immutable, "what changed" is a reference diff — the subscription already hands you (state, prev):

// core/engine/resolve.ts
const cache = new Map<string, ResolvedTrack>();   // trackId → resolved output

export function resolveProject(project, prev?) {
  for (const id in project.tracks)
    if (project.tracks[id] !== prev?.tracks[id]) cache.delete(id);  // dirty → drop cache
  const resolved = Object.keys(project.tracks).map(id =>
    cache.get(id) ?? resolveTrack(project.tracks[id], project));   // reuse or recompute
  return buildGraph(resolved);   // split objects/modulators, build routing + tag indices
}

resolveTrack runs the event-modifier combine chain from trackTypes.ts (its child modifiers — mute, suppress… — fold into the parent's event stream) and stamps the cache. One edited note → one track object changes identity → one resolveTrack call. O(change), not O(project) — the fix for the EDAW lag.

Dependency edges — which way they point. There are two relationships, and invalidation follows each. (1) A modifier child modifies its parent — you nest a mute / suppress track under the object track to modify the object, so editing that child (or adding / removing one) re-resolves the parent. (2) An object child inherits its parent's stream (the parent feeds it), so editing the parent re-resolves its children. Tyler's resolver states it directly: "modifier tracks affect the parent's output; regular tracks inherit it." Build a small dependents: Map<id, id[]> that captures both directions; start by invalidating the edited track plus its parent and subtree, then tighten.

D.2 — VisualEngine.ts (a module singleton, not a React store)

Deliberately not a zustand store: per-frame finalProps must never trigger React re-renders. It's a plain module holding mutable state, read imperatively from useFrame.

// core/engine/VisualEngine.ts
let graph: ResolvedGraph | null = null;
export function setProject(p, prev) { graph = resolveProject(p, prev); bumpObjectList(); }
export function computeAtBeat(beat) {
  const outs = evaluateModulators(graph, beat);   // modulators.ts + automation.ts
  runMatrix(graph, outs);                       // matrix.ts — fills finalProps
}
export const getObjectState = (id) => graph?.objectsById.get(id);

bumpObjectList() notifies VisualScene (a tiny external-store subscription) only when the set of objects changes — so React reconciles the scene tree on resolve, never per frame.

D.3 — VisualBeatSync.tsx — the only React-side glue

// core/engine/VisualBeatSync.tsx — mounted once inside <Canvas>
export function VisualBeatSync() {
  useFrame(() => computeAtBeat(useTimeStore.getState().currentBeat));  // runs first
  return null;
}
// module scope — set up once on import: deferred, debounced resolve
let t; useProjectStore.subscribe((s, prev) => {
  clearTimeout(t); t = setTimeout(() => setProject(s, prev), 80);  // never blocks the edit
});
Files this leans on, and why: TimeStore (the clock — already exists), ProjectStore (the document — subscribe only, no writes), and R3F's useFrame (already used by today's Cube). Nothing in the editor changes to support this; it's pure observation.

Phase 6 — Ability lanes  ·  Phase 7 — Event modifiers trackTypes midi editors

  • Abilities — shipped as 5 commits. Each ability = an entry in the parent's parallel lanes structure (decided — not a child track), rendered as an indented sub-row with its own MIDI editor. (Shipped in phase 7 already — modifier tracks fold into the parent's note stream.)
As-built (phase 6, 5 commits):
  1. Skeleton — AbilityLaneDef { key; label; editor?: EditorKind; color? } + abilities? on ObjectInstrumentDef (instruments/types.ts); Track.lanes?: Record<string, Block[]> (types.ts) — keyed per ability, each a full block+notes stream (not the plan's flat Block[]). editor declared; only pitched wired.
  2. Ability events — resolve.ts folds each lane into ResolvedObject.abilityEvents: Map<key, ResolvedNote[]>, threaded to ObjectState so the instrument's own render pulls it per frame.
  3. Track-row UI — added (opt-in) lanes render as uniform-height indented sub-rows via a new flattenVisualRows model (VisualRow = track | lane) in trackTree.ts; the shown lanes come from abilityLanesOf(track) = declared abilities filtered to those present in track.lanes. The object track just gets taller. computeDropTarget, nest-drag, library-drag, block vertical-move, and Alt-copy-drag were all made visual-row-aware so a taller track can't desync the drag math. New AbilityLaneRow.tsx + self-contained useLaneGestures.ts (lane draw/move/resize).
  4. Editor lane-targeting — editingBlock.laneKey? (UIStore); the store's block actions take an optional laneKey (writes route to blocks or lanes[laneKey]); the MIDI editor (PianoRollPanel / useMidiEditorState / useMidiBlockGestures / MidiEditor) threads it, so a lane block edits exactly like a track's.
  5. First real ability — the Cube's Shatter (Cube.tsx): a note on its Shatter lane bursts the cube into eight fragments that fly out and reassemble over the note, velocity setting the blast radius.
The behavior is code, not a scalar namespace. Per §4.4/§4.5: the instrument's own render consumes abilityEvents (the code escape hatch) — abilities are bespoke and instrument-intrinsic, not generic or attachable. Circle/Triangle stay passive RenderSpec specs (no abilities). The declarative onAbility grammar remains the long-term north star, to be grown from real abilities like Shatter, not designed up front.

Automation — parameter lanes as child tracks ✓ shipped (4 commits)

  • The third input layer (§"three layers"): params ← automation, ports ← modulators, abilities = signature moves. The plan draws automation as "a lane that drives a param value" and leaves the container open; decided: an automation track is a real CHILD TRACK (type: 'automation', no instrument, targetParam, interpolation) parented to the object — mirroring the modifier-track pattern (EDAW: "a child automates its parent").
As-built (automation, 4 commits):
  1. Scaffolding — core/engine/automation.ts: extractKeyframes(blocks, bpb, min, max) (each note's pitch → param value, pitch span AUTOMATION_PITCH_MIN..MAX = 36–84 → [param.min, param.max]) + sampleLane(keyframes, beat, mode) (binary-search + interpolate: step / linear / ease-in/out/in-out / exponential, endpoints held). Pure function of the beat → scrub == playback.
  2. Data model — ResolvedObject.automations; resolve.ts gathers each object's automation child tracks into keyframe lanes; computeAtBeat samples per frame and overrides obj.params[targetParam] (exactly how the matrix produces portValues). addAutomationTrack(parentId, paramKey, paramLabel).
  3. Right-click UI — TrackContextMenu.tsx on a track label: "Add ability track ▸" (parent's abilities) above "Add automation track ▸" (parent's params). This is also where abilities became opt-in — addAbilityLane + the centralised abilityLanesOf. An automation track can't be re-parented (locked to its object).
  4. Value editor — an automation track opens the MIDI editor in value mode: generateAutomationRows (rows labelled by value), the param name in the frozen corner, and an interpolation picker (setTrackInterpolation).
This revives the old dead stub (TrackType 'automation', targetParam, interpolation, generateAutomationRows) — now live. Long-term the declarative spec grows an automation vocabulary too; for now it's a child-track keyframe lane.

Phases 8–10 — Effects, RenderSpec, English later

  • Effects (plugins) — ✓ shipped. Transform (offset/rotate/scale), Clone (echo/tile/radialSymmetry/rotationalSymmetry/linearDuplicate), and Shader (kaleidoscope/pixelate/chromaticAberration/opacity), ported from Tyler. See the as-built note below.
  • RenderSpec — ✓ shipped: core/engine/renderSpec.ts (interpreter + expression vocabulary); ObjectRenderer dispatches code-vs-spec (Circle/Triangle are specs).
  • English → spec: an LLM endpoint that emits + validates a RenderSpec. Only after the spec is stable. not started
As-built (effects). Contract widened to Tyler's VisualPlugin { category: 'transform'|'clone'|'shader', params, applyTransform?, getClones?, fragmentShader? } (plugins/types.ts); chain per object in ObjectRenderer is transform ▸ clone ▸ shader. time = the beat (music-synced), not wall-clock.
  • Clones — CloneWrapper React-duplicates the object into N groups (not InstancedMesh — matches Tyler; fine for modest N), applying getClones().getTransform(i) + getOpacity(i) per frame. 5 plugins.
  • Shaders — Option A, per-object FBO (the plan's per-object model): ShaderWrapper renders the object (with its world transform) into an offscreen scene + FBO via createPortal, runs each shader as a ping-pong fullscreen pass, and draws the result as a clip-space fullscreen overlay (depth-test off) over the 3D scene. A shaded object becomes a full-frame post-processed layer; un-shaded objects are untouched. 4 plugins.
  • Deferred (Tyler has, we skipped): the multi-scene SceneCompositor + masks (not needed for per-object post-fx), instancing pools (lib/three/* — an instrument concern), and automating effect params (automation targets instrument params only so far). Enum/boolean plugin params render as numeric sliders for now.

H · Risks & open questions

  • risk Resolve dependency edges. Incremental invalidation must follow combine-chain + routing + loop dependencies, or a stale cache renders wrong. Mitigation: build a dependents map at resolve; start by invalidating the edited subtree, tighten with profiling. (§D.1)
  • risk Engine-as-singleton vs React. Per-frame state lives outside React on purpose; the only React-visible signal is the object list. Get the external-store subscription right (useSyncExternalStore in VisualScene) or you'll either over-render or miss new objects.
  • risk Cube refactor is load-bearing. It's both the first object def and the proof the engine drives rendering. Keep its mesh JSX; only move the value-production out. If phase 1 doesn't reproduce today's pulse exactly, the engine wiring is wrong — a good canary.
  • open Where do routings live — on the modulator track or a separate list? Plan assumes Track.targets: Routing[] on the modulator (matches "the user picks a modulator + a scope"). Revisit if a routing ever needs to outlive its modulator.
  • decided Abilities are a parallel lanes[] structure — not child tracks in the hierarchy. So phase 6 doesn't depend on phase 5's nesting data model (it renders the lanes array as indented sub-rows). Architecture §4.4/§7 still phrase this as open — sync them if you want one source of truth.
  • open Resolve debounce vs playback freshness. 80ms is fine while editing; force a synchronous resolve on play so visuals can't lag audio at the downbeat.
