Persistence & Schema Strategy

How to save projects to Supabase without hand-editing the schema every time the data model moves — and where the Terraform provider actually fits.

You want to save projects. You've used Supabase and want it again, fast. The scar tissue from last time is real: the data model changed constantly and you were in the dashboard hand-editing tables to keep up. You're asking whether there's a better way, and you lean toward the Supabase Terraform provider — "that's like CDK, right?". Short version: Terraform is in the same family as CDK, but it solves a different half of your problem than the half that actually hurt.

The one-line answer: your project is basically one big JSON document (tracks → blocks → notes, plus instruments/effects/automation/abilities). Store it as a JSONB blob in Supabase and almost all of your schema churn disappears — adding a field to Track is a TypeScript edit, not a migration. Keep a tiny relational skeleton (a projects row: id, owner, title, updated_at, data jsonb) with RLS, manage that skeleton with Drizzle migrations (schema follows your code), and reach for Terraform only when you want the Supabase project config itself reproducible. Terraform is great, but it is not primarily a table-migration tool — it won't eliminate the thing that burned you.

★ Internalize this first

Before the option-by-option detail, here is the whole recommendation in one view, and the three ideas everything hangs off.

Three things to hold in your head

  1. Your pain is "schema drift", not "no database". The fix isn't a fancier deploy tool — it's not having a wide relational schema to drift in the first place. A document blob makes 90% of your model changes require zero schema work.
  2. "Schema-as-code" and "infra-as-code" are different axes. Migrations tools (Drizzle/Prisma/Supabase CLI) keep your tables in sync with your code. Terraform/CDK keep your project config (auth, roles, storage, edge settings) reproducible. Terraform is weak at the first axis — that's the nuance under your "that's like CDK, right?".
  3. You can have both, cheaply. JSONB blob for the document + Drizzle for the few real columns gets you saving today with near-zero manual schema edits. Add Terraform later only if reproducible project config becomes worth it. Nothing here is one-way.

1 · The real problem — name it correctly

It's tempting to frame this as "pick a database" or "pick a deploy tool." Neither is the thing that hurt. The thing that hurt was schema drift: your app's data model kept changing, and every change meant your Postgres tables were now out of sync with your code, so you opened the Supabase dashboard and hand-edited columns to catch up. That is slow, error-prone, un-versioned, and un-reproducible.

There are two orthogonal problems hiding inside "I don't want to edit the schema by hand," and it's worth separating them because they have different best answers:

ProblemWhat it meansFamily of solution
Schema drift / migrationsYour tables need to change as your data model changes, without manual dashboard edits.Either avoid a wide schema (JSONB) or generate migrations from code (Drizzle/Prisma/Supabase CLI).
Infra / project config driftThe Supabase project itself — auth providers, roles, storage buckets, RLS policies, edge config — should be reproducible, not clicked together by hand.Infra-as-code: Terraform (or CDK for AWS). This is what the Terraform provider is for.
Your instinct toward Terraform is a good instinct aimed at the second problem. But the pain you described — editing columns every time the model changes — is the first problem. Terraform's table support exists but is deliberately thin (more in §3.3), so it won't be the thing that saves you here. The good news: the first problem has an even easier fix for a project shaped like yours.

2 · A visualizer project is one big JSON document

Look at what a Cabin project actually is. From src/editor/types.ts and src/editor/store/ProjectStore.ts, the whole document is a tree of plain, serializable objects:

// The entire persisted project, conceptually:
interface ProjectDocument {
  bpm: number; beatsPerBar: number; totalBars: number;
  rootTrackIds: string[];
  tracks: Record<string, Track>;      // each Track → blocks → notes,
                                       // plus params, tags, targets (routings),
                                       // visualPlugins, lanes, laneMeta, childIds…
}

Every field on Track, Block, Note, Routing, and PluginInstance is a string, number, boolean, array, or nested object. There are no foreign keys you query across, no "give me all notes with velocity > 100 across every project" reporting need, no partial writes from multiple clients. You load a whole project, edit it in memory in Zustand, and save the whole thing back. That is the textbook profile of a document, not a normalized relational dataset.

This reframing is most of the answer. The reason last time hurt is that a wide relational schema forces the database to know the shape of your data — so every time the shape changes, the database has to change too. A JSONB blob lets the database stay ignorant of the shape. For a single-writer, load-whole/save-whole creative document, that's not a hack; it's the correct model.

3 · The four options, honestly

Four candidates. For each: what it is, a small diagram, and a straight pros/cons pass.

3.1 — JSONB document column recommended core

Store the entire serialized project in a single jsonb column. The relational schema is a stub: one projects table with identity/ownership columns and the blob. Adding, removing, or restructuring anything inside the document is a code change, never a schema change.

create table projects (
  id          uuid primary key default gen_random_uuid(),
  owner       uuid references auth.users(id) not null,
  title       text not null default 'Untitled',
  data        jsonb not null,          -- the whole ProjectDocument
  updated_at  timestamptz not null default now()
);
alter table projects enable row level security;
create policy owner_rw on projects
  using (owner = auth.uid()) with check (owner = auth.uid());

✓ Pros

  • Near-zero schema churn — the entire class of problem you hit last time evaporates. Model changes are TS-only.
  • Fastest to ship — one table, one policy, a save()/load() pair. You could be persisting today.
  • Trivial serialization — your store is already plain objects; JSON.stringify(state) essentially is the payload.
  • Versioning is easy — add a schemaVersion field inside the JSON and migrate in app code on load. No DB migration to reshape old rows.
  • Still Supabase — RLS, auth, realtime, storage all work unchanged.

✕ Cons

  • No cheap cross-project queries — "every note above velocity 100" means scanning JSON. You don't need this today, and Postgres can index into JSONB if you ever do.
  • Whole-document writes — you save the entire project, not a delta. Fine at your sizes; not fine for Google-Docs-style multi-writer editing (not your case).
  • Weaker DB-level validation — Postgres won't enforce field types inside the blob. Mitigate with a Zod parse on load/save (also gives you migration hooks).
  • Big rows — very large projects become large JSON. Postgres handles multi-MB JSONB fine; revisit only if projects get huge.

3.2 — Supabase + a migrations tool recommended for the skeleton

Keep a real (small) relational schema, but never hand-edit it. A migrations tool turns schema changes into versioned, code-authored files you apply with a command. Two flavors:

  • Supabase CLI migrations — you write SQL migration files; supabase db push applies them; they live in your repo. The schema is versioned, but you still hand-author the SQL.
  • Drizzle (or Prisma) — you define the schema in TypeScript, and the tool generates the migration from the diff between your code and the DB. The schema follows your code. Drizzle is the lighter, closer-to-SQL choice and shares your TS types, so it pairs naturally with a Next.js app.

✓ Pros

  • No hand-editing — schema changes are code + a command, versioned in git.
  • Real relational power — proper columns, indexes, FKs, DB-level constraints for the things that are relational (users, project list, sharing).
  • Drizzle shares your TS types — one source of truth between app and DB for the relational parts.
  • Reproducible — apply the same migrations to dev/prod/CI.

✕ Cons

  • Still a migration per model change if you model the whole document relationally — you've automated the pain, not removed it.
  • More upfront wiring than the blob — a schema file, a migration workflow, a query layer.
  • Two type systems to reconcile if you also keep the app's own types — mitigated by generating one from the other.
The winning move isn't blob or migrations — it's both, on different data. Use Drizzle migrations for the thin relational skeleton (which changes rarely), and the JSONB blob for the document (which changes constantly). The migration tool then only ever touches the stable part, so migrations become rare instead of per-edit.

3.3 — Supabase Terraform provider right tool, different job

The Supabase Terraform provider is declarative infrastructure-as-code for the Supabase project itself. You describe the desired state of your project in HCL, and terraform apply reconciles the live project to match — creating what's missing, changing what drifted, deleting what you removed.

# HCL — declarative desired state of the PROJECT
resource "supabase_project" "cabin" {
  organization_id = var.org_id
  name            = "cabin-visuals"
  region          = "us-east-1"
  # … database password, instance size, etc.
}
resource "supabase_settings" "cabin" {
  project_ref = supabase_project.cabin.id
  auth = jsonencode({ /* providers, JWT expiry, redirect URLs … */ })
  api  = jsonencode({ /* exposed schemas, max rows … */ })
}

What it's genuinely good at: project creation, auth configuration, API/settings, environment reproducibility — the stuff you'd otherwise click through in the dashboard once and forget how you set up. If you ever want to stand up a staging project identical to prod, or onboard a collaborator's environment, this is exactly right.

The nuance behind "that's like CDK, right?" Terraform is declarative IaC like CDK (§4). But the provider is not primarily a table-migration tool. Managing your tracks/notes table shape — the exact thing that burned you — is not what it's built for; database schema is generally handled through migrations (Supabase CLI / Drizzle) applied over a connection, not through Terraform resources. So adopting Terraform would give you reproducible project config and still leave you needing a migrations story (or the blob) for the schema-drift problem. It solves the second problem in §1, not the first.

✓ Pros

  • Reproducible project config — auth, settings, org/project setup as versioned code.
  • Declarative reconcile — terraform plan shows the diff, apply converges to it. Same mental model you liked in CDK.
  • Multi-environment — spin up identical staging/prod projects.
  • Auditable — the whole project's config lives in git.

✕ Cons

  • Doesn't solve your actual pain — table/column drift is still a migrations job, not a Terraform job.
  • Overhead for a solo hobby project — state files, provider auth, tokens; real value shows up with multiple environments/people.
  • Learning curve + ceremony for something you'd run rarely.
  • Not where the fast path is — it's an add-on, not the thing that gets you saving projects sooner.

3.4 — AWS CDK heavier, not Supabase

AWS CDK defines cloud infrastructure in real TypeScript, which it synthesizes into CloudFormation templates that AWS provisions. It's powerful and it's the tool you'd reach for if your backend were AWS-native (RDS/Aurora + Lambda + API Gateway + Cognito, etc.).

✓ Pros

  • Write infra in TypeScript — loops, conditionals, real abstractions, shared types with your app.
  • Deep AWS coverage — if you want the full AWS toolbox, nothing beats it.
  • Battle-tested for serious production infra.

✕ Cons

  • It's not Supabase — you'd be rebuilding auth, RLS, the DB, and the client SDK you already know and want to reuse. Against your explicit goal.
  • Heaviest option by far — CloudFormation, IAM, VPCs, bootstrapping. Enormous overhead for saving a JSON document.
  • Slowest to first save — the opposite of "fast."
  • Still needs a migrations story for the DB — CDK provisions the database, it doesn't manage table shape either.
CDK is a fine tool aimed at a problem you don't have. You want persistence for a creative app you already picked Supabase for. Unless you're planning to leave Supabase for AWS-native infra, CDK is out of scope — included here only to answer the comparison you raised.

4 · Terraform vs CDK — the family resemblance

Your intuition — "Terraform is like CDK" — is correct at the level that matters: both are infrastructure-as-code that reconcile a desired state. You describe what you want; the tool figures out the create/update/delete steps to get there, and can show you a plan/diff first. The differences are in how you author and what runs:

TerraformAWS CDK
ModelDeclarative IaC — you state the end resultDeclarative IaC — but authored imperative-ish
Authoring languageHCL (a purpose-built config language)TypeScript (or Python/Java/Go) — real code with loops & abstractions
What actually executesTerraform core diffs state and calls provider APIs directlyCode synthesizes CloudFormation, which AWS then provisions
Cloud scopeMulti-cloud via providers (incl. a Supabase provider)AWS-only
StateExplicit state file (local or remote)State lives in CloudFormation stacks
Reconcile commandterraform plan → terraform applycdk diff → cdk deploy
So: yes, they're cousins. If you like the declarative-reconcile model, the Supabase Terraform provider gives you exactly that for your Supabase project config. Just don't expect it to manage your table shape — that's the one thing that's still a migrations job. It's a great tool pointed slightly to the side of your pain.

6 · Recommendation — the pragmatic hybrid

For a fast-moving creative project whose data model is essentially one document, the fastest path with near-zero manual schema work is:

The stack

  1. Store the project as a JSONB blob in Supabase, in a thin projects table (id · owner · title · updated_at · data jsonb) with RLS scoped to the owner. This absorbs the schema drift that hurt last time — model changes become TypeScript edits, not migrations.
  2. Wrap it in a thin typed client — a small saveProject()/loadProject() pair that JSON.stringifys the store and validates on load with Zod against your types.ts. The Zod parse is also where you run schemaVersion upgrades in app code.
  3. Use Drizzle migrations for the few real relational columns — the projects skeleton, and later any genuinely relational tables (users, sharing, a project index). The schema follows your TypeScript; you generate + apply migrations, never touch the dashboard. Because the document lives in the blob, these migrations are rare.
  4. Add the Terraform provider later, if and when you want reproducible project config — auth providers, settings, a staging environment. It's the right home for your "declarative like CDK" instinct, but it's an enhancement, not the critical path, and it won't reduce table-migration work by itself.
Why this and not "just Terraform": respecting your lean — Terraform is genuinely good and it is in CDK's family, so keep it in your back pocket for reproducible project config. But it doesn't touch the schema-drift pain, so leading with it would leave your actual problem unsolved while adding ceremony. The blob solves the pain outright; Drizzle handles the rare real migrations; Terraform slots in when reproducibility (not persistence) is what you want. That ordering gets you saving projects the soonest with the fewest manual schema edits — which is exactly what you asked for.

7 · Concrete first steps

  1. newCreate the projects table + RLS in Supabase (the SQL in §3.1). One migration, and you're storing documents.
  2. newAdd a ProjectDocument type + Zod schema mirroring src/editor/types.ts plus the top-level bpm/beatsPerBar/totalBars/rootTrackIds/tracks from ProjectStore.ts. Include a schemaVersion integer.
  3. newWrite saveProject() / loadProject() using supabase-js: save serializes the store to the blob; load parses with Zod and hydrates the store. Debounce autosave off store changes.
  4. setupAdd Drizzle pointed at the Supabase Postgres connection, with the projects table as its first schema definition, so future skeleton changes are code-generated migrations rather than dashboard edits.
  5. laterIntroduce Terraform only when you want a reproducible second environment or want auth/settings in git. Start with supabase_project + supabase_settings; leave table schema to Drizzle.

8 · Risks & open questions

  • riskBlob validation drift. Postgres won't police the JSON's shape, so a bad write could persist a malformed document. Mitigation: a Zod parse on every save and load, and a schemaVersion with in-app upgrade functions so old saves keep loading.
  • riskWhole-document writes & concurrency. Saving the entire project is fine for one editor but clobbers concurrent edits. Fine today (single-writer). If multi-writer ever matters, that's a different architecture (CRDT/delta sync), not a schema tweak.
  • openHow much to pull out of the blob. Title and updated_at clearly belong as real columns (for a project list without parsing every blob). Anything else you want to list or sort by should be promoted to a column. Decide the list-view fields early.
  • openAutosave cadence & history. Debounced autosave vs explicit save; whether to keep version snapshots (cheap with a blob — insert a new row or a project_versions table). Easy to add later precisely because it's a document.
  • riskTerraform scope creep. If you do adopt Terraform, resist trying to manage table schema through it. Keep a clean line: Terraform = project config, Drizzle = table schema, blob = document. Blurring that line reintroduces the exact friction you're trying to escape.
  • openRow size ceiling. Very large projects mean large JSONB rows. Postgres is comfortable into the multi-MB range, but if projects balloon, consider offloading the blob to Supabase Storage and keeping only metadata in the row.
