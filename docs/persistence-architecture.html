Persistence Architecture

How a Cabin project is saved, loaded, and versioned on Supabase — one JSONB document blob per project, a thin relational spine, local-first autosave, and a JSON shape that can evolve without breaking old saves.

Cabin already has Supabase Auth, a projects table, and a working project browser (create / list / delete). What it does not yet have is the part that matters most: the actual project — tracks, blocks, notes, instruments, effects, automation, abilities — never leaves the browser. This doc proposes the missing half: store the whole project as a single JSONB document blob, wire debounced autosave + load to the Zustand ProjectStore, protect rows with RLS, manage the schema as code with Drizzle, and give the JSON shape a version number with upgrade functions so the model can keep changing fast.

The one-line answer: the project is a document, not a relational graph — persist it as one data JSONB blob keyed by (id, owner), with only the columns you actually query (name, updated_at, schema_version) promoted to real columns. The client stays local-first (Zustand is the source of truth; Supabase is a debounced mirror), RLS makes "you can only touch your own rows" a database invariant, and a schemaVersion + pure upgrade functions let the JSON evolve without a migration for every field you add.

★ The big picture — internalize this first

Before the section-by-section detail, here is the whole persistence layer in one view, and the five mental models everything else hangs off. Hold these and the rest is elaboration.

Five things to hold in your head

  1. Document blob, not relational graph. The whole project serializes to one data JSONB value. You don't shred tracks/blocks/notes into rows — the editor's in-memory model is the storage model. One read rebuilds everything; one write saves everything.
  2. A thin relational spine alongside the blob. Only columns you actually query become real columns: id, owner, name, updated_at, schema_version. The project browser lists on those without ever parsing a blob.
  3. Local-first: Zustand is the truth, Supabase is a mirror. Edits hit the store instantly; a debounced observer flushes to Supabase off the critical path. The UI never blocks on the network, and a failed save never corrupts the in-memory doc.
  4. RLS makes ownership a database invariant. owner = auth.uid() on select/insert/update/delete — the client physically cannot read or write another user's project, even with a leaked anon key. Security lives in Postgres, not in app code.
  5. The JSON shape is versioned. A schemaVersion travels inside the blob; pure upgrade() functions run on load to bring an old doc up to current. The model keeps changing fast; old saves keep opening.

2 · The decisions that actually matter

Five forks. For each: the two camps, and the call.

ForkCamp ACamp BCall
Storage shape Normalize — tracks / blocks / notes each in their own tables, joined on load One JSONB document blob per project B — the editor already holds the whole doc in memory; shredding it into rows buys queryability you don't need and costs a schema migration per model change.
Relational columns Everything in the blob, zero real columns Promote only what a list/query needs B — name, updated_at, schema_version as columns so the browser sorts/filters without parsing JSON; the rest stays in data.
Source of truth Server-authoritative — every edit is a mutation round-trip Local-first — Zustand is truth, Supabase mirrors it B — a DAW edits at 60fps (drags, sliders). The network can never be in that loop; autosave is a debounced background flush.
Access control Filter by user_id in app queries RLS — enforced in Postgres B — app-side filtering is one forgotten .eq() away from a leak. RLS makes ownership an invariant the client can't violate.
Schema management Click tables/policies in the Supabase dashboard Schema-as-code via Drizzle migrations B — the model changes fast; migrations in git are reviewable, reproducible, and roll forward on every environment. RLS policies live in migrations too.
When the blob is right — and when it isn't. The document blob is the correct default because the access pattern is "open the whole project, edit it locally, save the whole project." If you later need server-side queries into a project (search all notes across projects, server-side render a thumbnail, multi-user live collaboration on sub-objects), that's when you selectively normalize — promote just those pieces out of the blob. Don't pre-normalize for hypothetical needs; the blob keeps you fast now and nothing about it blocks that later.

3 · The data model — document blob + relational spine

Two halves: the relational spine (a handful of columns, so the browser and RLS have something to key on) and the document blob (the entire serialized project). The spine is what the database queries; the blob is what the editor consumes.

// The serialized project — what lives in projects.data (JSONB).
// Mirrors ProjectStore's non-function fields, the same boundary HistoryStore snapshots.
interface ProjectDocument {
  schemaVersion: number;        // travels INSIDE the blob — drives upgrade() on load (§8)
  bpm: number;
  beatsPerBar: number;
  totalBars: number;
  tracks: Record<string, Track>;   // tracks · blocks · notes · params · tags · targets · lanes · effects
  rootTrackIds: string[];
  audioClip?: AudioClip;          // {ref, fileName, duration} — metadata only; bytes → Storage (deferred)
}

// The relational spine — the row, minus the blob. What the browser lists on.
interface ProjectRow {
  id: string;                    // uuid, primary key
  owner: string;                 // = auth.uid() — the RLS anchor (today's column: user_id)
  name: string;                  // duplicated out of the doc so a list query never parses JSON
  updated_at: string;            // bumped on every save — sort "recent", detect conflicts
  schema_version: number;        // mirrors data.schemaVersion — lets you find/upgrade docs in bulk
  data: ProjectDocument;          // the blob
}
Why duplicate name and schema_version out of the blob? Because the project browser must list, sort, and filter without deserializing every document — a column read is cheap, a JSON parse across N rows is not. The rule: a field becomes a real column the moment the database (not the editor) needs to read it. Everything else stays in data. On save you write both places (the repo copies doc.name → row.name); the blob's copy is canonical, the column is a queryable projection.

3.1 — The Supabase stack

Cabin is a Next.js App Router app, so the client story is the standard @supabase/ssr split — already in place. The persistence layer reuses it rather than introducing a new client.

  • Browser client — utils/supabase/client.ts (createBrowserClient). This is what the editor's autosave/load calls run through; it carries the user's session cookie automatically, so RLS sees auth.uid().
  • Server client — utils/supabase/server.ts (createServerClient + cookies). For server components / actions (the project browser can list here; the editor is a client island).
  • Middleware — utils/supabase/middleware.ts refreshes the session on every request and gates routes; the editor already sits behind it.
  • Storage (deferred) — the audio bytes belong in Supabase Storage (a bucket, RLS-scoped like the table), referenced by the doc's audioClip.ref. The document blob only ever holds the reference, never the bytes — keeping the blob small and JSON-friendly.

3.2 — The persistence module

All Supabase-touching code for the document lives in one place — a persistence/ module — so the editor never imports supabase-js directly and swapping the backend later is a one-directory change.

// persistence/projectRepo.ts — the only place document CRUD lives
export const projectRepo = {
  // list: spine columns only — never pulls a blob
  async list(): Promise<ProjectSummary[]> {
    const { data } = await supabase
      .from('projects').select('id, name, updated_at')
      .order('updated_at', { ascending: false });
    return data ?? [];
  },
  // load: one row → upgrade the blob to current shape → hand back a document
  async load(id): Promise<ProjectDocument> {
    const { data } = await supabase
      .from('projects').select('data').eq('id', id).single();
    return upgradeDocument(data.data);          // §8 — migrate old shapes forward
  },
  // save: one UPSERT — blob + the projected columns, in one write
  async save(id, doc): Promise<void> {
    await supabase.from('projects').update({
      data: doc, name: doc.name, schema_version: doc.schemaVersion,
      updated_at: new Date().toISOString(),
    }).eq('id', id);                          // RLS ensures it's YOUR row
  },
};
The repo is the seam. Everything above it (the editor, the autosave loop) speaks in ProjectDocuments; everything below it speaks Postgres. Because there's exactly one door, the eventual "add offline IndexedDB cache," "swap to a different backend," or "batch multiple saves" changes are all contained here — no editor code changes.

4 · The save/load lifecycle

The lifecycle has two independent flows that meet only at the store: load (Supabase → store, once, on open) and autosave (store → Supabase, debounced, forever). The editor between them is untouched — it edits the store and knows nothing about the network.

4.1 — Debounced autosave

Autosave is a subscription to ProjectStore, debounced so a continuous gesture (a slider drag, a block move) collapses to one write when it settles. It mirrors the exact mechanism HistoryStore already uses — a store subscription with a burst window — just aimed at Supabase instead of an undo stack.

// persistence/autosave.ts — one standing subscription, registered on load
let timer, saving = false, dirty = false;

export function startAutosave(projectId) {
  return useProjectStore.subscribe((state, prev) => {
    if (!documentChanged(state, prev)) return;   // reference diff — immutable store, cheap
    dirty = true;
    clearTimeout(timer);
    timer = setTimeout(flush, 1000);            // ~1s idle → save; coalesces a burst
  });

  async function flush() {
    if (saving) { timer = setTimeout(flush, 300); return; } // don't overlap writes
    saving = true; dirty = false;
    setSaveStatus('saving');
    try {
      await projectRepo.save(projectId, serialize(useProjectStore.getState()));
      setSaveStatus(dirty ? 'pending' : 'saved');   // edited again mid-save? still dirty
    } catch (e) {
      setSaveStatus('error'); dirty = true;         // keep dirty → retry on next tick / edit
    } finally { saving = false; if (dirty) timer = setTimeout(flush, 1000); }
  }
}
Autosave never blocks the editor. Three rules keep it invisible: (1) debounced — a slider drag writes the store every frame but saves once, ~1s after you let go; (2) coalesced & non-overlapping — one write in flight at a time, and if you edit mid-save the doc stays dirty and a follow-up save fires; (3) failure-safe — a network error never touches the in-memory doc, it just leaves the project dirty and retries. Also flush on visibilitychange/beforeunload and force a save on explicit "close project" so the last edit can't be lost to the debounce window.

4.2 — Load & hydrate

Load is the mirror image and simpler: the editor route already carries ?project=<id> (unused today). On mount, read the row, run the blob through the upgrader (§8), and setState the store — the same shape HistoryStore restores into on undo, so the editor renders it with zero special-casing.

  • One SELECT pulls data (the blob). The spine columns aren't needed to open — they're for the list.
  • Upgrade on the way in — upgradeDocument(blob) runs pure vN → vN+1 steps until the doc is current (§8). The editor only ever sees the latest shape.
  • Hydrate, then arm autosave — setState(doc), then startAutosave(id). Arming after hydrate avoids the hydrate itself triggering a redundant save.
  • New project — create inserts a row with an empty/default document (current schemaVersion) and routes into the editor, which loads it like any other.

4.3 — Optimistic, local-first

The whole design rests on one inversion: the store is authoritative, the server is a mirror. Every edit is applied optimistically to Zustand and is immediately real to the user; Supabase catches up asynchronously.

The honest tradeoff — last-write-wins & the multi-tab caveat. A single-blob, local-first model is simple precisely because it assumes one editor at a time per project. Two tabs (or two devices) editing the same project will clobber each other on save — last write wins, silently. Mitigations, cheapest first: (a) an updated_at conflict check on save (if the row moved since you loaded, warn instead of overwrite); (b) a Supabase Realtime channel that flags "this project is open elsewhere"; (c) full CRDT/OT only if real-time collaboration becomes a product goal. Start with (a) — it's a where-clause — and don't build (c) speculatively. This is the blob's one genuine cost, and it's the same cost Google Docs paid before it added OT.

5 · Auth

Auth is already built — this section is about how persistence uses it, not how to build it. The session's user.id is the single anchor everything else keys on: it's the owner column, it's what RLS compares against, and it's what the project list filters by.

  • Auth hooks — a small useAuth() in persistence/ exposes the current user + loading state (wrapping supabase.auth.getUser() / onAuthStateChange), so the editor can guard "save requires a session" and the list knows whose projects to fetch.
  • Owner on insert — new-project insert stamps owner: user.id. RLS also enforces this on write, so even a tampered client can't insert a row it doesn't own (§6).
  • Route gating stays as-is — the existing middleware already redirects unauthenticated users away from protected pages; the editor inherits it. No new gating needed for persistence.

6 · Row-Level Security

RLS turns "you can only touch your own projects" from an app-code convention into a database invariant. Without it, security is one forgotten .eq('user_id', …) away from leaking every user's work. With it, the anon key is safe to ship — the database refuses cross-owner reads and writes no matter what the client sends.

-- one policy per verb; auth.uid() is the logged-in user's id
alter table projects enable row level security;

create policy "read own projects"  on projects for select
  using (owner = auth.uid());

create policy "insert own projects" on projects for insert
  with check (owner = auth.uid());   -- can't insert a row you don't own

create policy "update own projects" on projects for update
  using (owner = auth.uid()) with check (owner = auth.uid());

create policy "delete own projects" on projects for delete
  using (owner = auth.uid());
USING vs WITH CHECK — the one subtlety. USING filters which existing rows a statement can see (read/update/delete). WITH CHECK validates the row a statement tries to write (insert/update). You need both on update: USING so you can only update rows you own, WITH CHECK so you can't update a row into someone else's ownership. Get these backwards and you either can't write your own rows or can hand rows to other owners.
RLS policies are schema-as-code too. These policies live in a Drizzle migration (§7), not the dashboard — so they're reviewed, versioned, and applied identically to every environment. A policy silently deleted in a prod dashboard is a security incident nobody can diff; a policy in a migration is a line in a PR.

7 · Schema-as-code (Drizzle migrations)

The relational spine is tiny — a handful of columns and four policies — but it will change (add thumbnail_url, add a project_shares table, tweak a policy). Managing that by clicking the Supabase dashboard is unreviewable and drifts between environments. Drizzle keeps the schema in TypeScript, generates SQL migrations, and applies them reproducibly.

// db/schema.ts — the spine, as code
export const projects = pgTable('projects', {
  id:            uuid('id').primaryKey().defaultRandom(),
  owner:         uuid('owner').notNull(),                // = auth.users.id
  name:          text('name').notNull().default('Untitled Project'),
  schemaVersion: integer('schema_version').notNull().default(1),
  data:          jsonb('data').notNull().$type<ProjectDocument>(),  // the blob, typed
  createdAt:     timestamp('created_at').notNull().defaultNow(),
  updatedAt:     timestamp('updated_at').notNull().defaultNow(),
});
// RLS enable + the four policies (§6) live in the generated migration alongside this.
Drizzle for the spine, not the blob. Drizzle types and migrates the columns — id, owner, name, updated_at, schema_version, and the data column's existence. It does not police the JSON inside data — that shape is governed by the app's schemaVersion + upgrade functions (§8). This split is the whole point: relational changes (rare, need a migration) are separate from document changes (frequent, need only a doc-version bump). Adding a field to a track is a code change with an upgrade function — zero database migrations. That's what "the model can change fast" buys you.

8 · Versioning the JSON document

The blob's freedom — change the model without a database migration — has a catch: old saved documents still have the old shape. A project saved last month doesn't know about the field you added yesterday. The answer is a schemaVersion stamped inside every document and a chain of pure upgrade functions that run on load to bring any old doc up to current.

// persistence/upgrade.ts — the JSON document's own migration system
const CURRENT_VERSION = 4;

// each step: pure, total, vN → vN+1. Never edit an old step; append new ones.
const upgrades: Record<number, (d: any) => any> = {
  1: (d) => ({ ...d, tracks: mapTracks(d, t => ({ ...t, tags: t.tags ?? [] })) }),   // v1→v2: add tags
  2: (d) => ({ ...d, tracks: mapTracks(d, t => ({ ...t, targets: toRoutings(t.targets) })) }), // v2→v3: reshape targets
  3: (d) => ({ ...d, tracks: mapTracks(d, t => ({ ...t, lanes: t.lanes ?? {} })) }),        // v3→v4: lanes record
};

export function upgradeDocument(raw): ProjectDocument {
  let d = raw, v = raw.schemaVersion ?? 1;
  while (v < CURRENT_VERSION) { d = upgrades[v](d); v++; }
  return { ...d, schemaVersion: CURRENT_VERSION };
}
Two rules keep doc-versioning safe forever. (1) Upgrade steps are append-only and immutable — once 1→2 ships and real users have v1 saves, you never edit it; a mistake there corrupts every old document. New shape change = new step at the tail. (2) Upgrade on read, save on current. A doc is upgraded when loaded and re-saved at CURRENT_VERSION, so the population drifts forward naturally — but you keep every historical step because a user who hasn't opened a project in a year still has that old version on disk. Optional: a one-off "upgrade all rows" backfill (find rows where schema_version < N, load-upgrade-save) to retire very old steps, but only after you're sure nothing dormant is left.

This maps cleanly onto Cabin's real history. The visual-engine work already reshaped the document repeatedly — tags arrived in phase 4, targets moved from a flat {targetTrackId, targetPort}[] to Routing[], lanes landed as a Record<string, Block[]>, automation/ability child tracks appeared. Every one of those would have been an upgrade step. Building doc-versioning now means the next such change (and there will be many — the RenderSpec, English authoring) costs one appended function instead of a data-loss incident.

9 · Risks & open questions

  • riskConcurrent edits (multi-tab / multi-device). Single blob + last-write-wins clobbers. Mitigation: updated_at optimistic-concurrency check on save; a Realtime "open elsewhere" flag; CRDT only if live collaboration becomes a goal. Decide the floor (a conflict warning) early. See §4.3.
  • riskBlob size at scale. A huge project (hundreds of tracks, dense automation) makes each autosave a big write. Mitigation: the debounce already coalesces; measure real doc sizes; if needed, split the largest sub-trees out of the blob (selective normalization) rather than abandoning it.
  • riskAutosave debounce vs data loss. An edit inside the debounce window that's followed by a crash/close is lost. Mitigation: flush on beforeunload/visibilitychange and on explicit close; keep the window short (~1s).
  • riskUpgrade-step correctness. A buggy or edited upgrade step silently corrupts old documents. Mitigation: steps are pure, append-only, and unit-tested against captured old-version fixtures; never mutate a shipped step.
  • openAudio bytes storage. The clip metadata rides in the blob; the bytes need Supabase Storage (RLS-scoped bucket, referenced by audioClip.ref). Deferred — but decide the bucket/RLS model before shipping audio persistence.
  • openWhere does the JSON schema live vs the DB schema? Two versioning systems (Drizzle for columns, schemaVersion for the blob) is deliberate but needs a clear rule: relational change → migration; document change → upgrade step. Document the boundary so contributors don't reach for the wrong one.
  • decidedLocal-first, blob-per-project. Zustand is authoritative; Supabase mirrors it debounced; the project is one JSONB document with a thin queryable spine. This is the recommended shape and everything above builds on it.

10 · Recommendation in one breath

Store the project as a document, mirror it debounced, version the shape.

  • Persist the whole project as one JSONB document blob in projects.data, with only id / owner / name / updated_at / schema_version promoted to real columns — the relational spine the browser lists on.
  • Keep the client local-first: Zustand's ProjectStore is the source of truth, and a debounced autosave subscription mirrors it to Supabase off the critical path — the editor never waits on the network.
  • Put every Supabase-document call behind one persistence/ module (client · projectRepo load/save/list · autosave · auth hooks) so the backend is swappable and the editor stays ignorant of storage.
  • Make ownership a database invariant with RLS (owner = auth.uid() on every verb), and keep the schema — tables and policies — as Drizzle migrations in git.
  • Give the JSON a schemaVersion + append-only upgrade functions so the model can keep changing fast without a migration per field and without ever breaking an old save.
  • Build on what's there: auth and the project list already work — the whole job is the document layer (load/save + versioning), and the stores were already designed for exactly this serialization boundary.
