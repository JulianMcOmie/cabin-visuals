🛠 Real bugs — medium effort

Visibility doesn't affect laser instruments

Tyler: disappointing✓ Fixed & verified 2026-07-22

Confirmed working by Julia. Took three passes — the shader fix was necessary but not sufficient, and the probe-driven hunt turned up two deeper bugs affecting every instrument: (1) a "hidden" object was still drawn at opacity 0 and kept writing depth, so its invisible silhouette carved holes out of anything behind it — fully faded objects are now un-rendered entirely; (2) shader-based instruments' opacity uniforms were written from a second site with its own timing — applyMaterialOpacity now syncs any material's declared uOpacity uniform itself, one write site, same frame as the render. Lasers (surface, bloom halo, and cast light), FilmStock, and preview ghosts all gate correctly now.

Adding/removing a scene makes objects stop rendering

Tyler: very annoying✓ Fixed 2026-07-22

Was: VisualScene.tsx keyed the whole scene map on the sorted set of scene ids, so adding one scene disposed every scene's render targets and remounted all portals. Now the map reconciles incrementally — surviving scenes keep their THREE.Scenes and GPU targets; only the added/removed scene's resources change, and disposal waits for the commit effect.

Export audio out of sync

Tyler: very annoyingInvestigateMedium once diagnosed

The pipeline's math is sound (video PTS and audio sample offsets both derive from the same clock), and a previous +44ms "AAC priming" compensation was removed because it made things worse. Remaining suspects: AAC encoder priming (2112 samples ≈ 44ms — i.e., the compensation may have had the right idea and the wrong sign or place), export-range anchoring, or player-dependent decoding. Needs a measured repro (export a click track, inspect the waveform against frames) before touching code.

Videos seem compressed (worse on busy projects)

Tyler: minor but long-term✓ Addressed 2026-07-22

Diagnosis: export bitrate was a fixed number per tier (12 Mbps at 1080p, 35–50 at 4K) regardless of content. Cabin Visuals frames are near-worst-case for a fixed-bitrate encoder — the final grade adds animated per-pixel grain to every frame, plus HDR bloom halos and particle fields — so busy projects blew the budget and got visibly quantized while simple ones fit and looked clean. That's exactly the "complicated projects look compressed at any resolution" symptom. (Export genuinely renders at target resolution, DPR 1 — resolution was never the issue.)

Change: tiers raised to motion-heavy values — 1080p 12→20 Mbps, 720p 8→12, 4K 35/50→60/80. Files grow proportionally (~60–70%), and the platform re-encode gets a much cleaner source. Remaining ceiling is Instagram/TikTok's own re-encode, which we can't control — exporting 4K gives their encoder the most to work with.

Camera not centered by default

Tyler: kinda annoying✓ Fixed 2026-07-22

Julia decided: centered. Default camera is now [0, 0, 5], and the Camera instrument's default Position Y matches (so adding one no longer jumps the view). Note this shifts the framing of every existing project down 1.2 units — expected and accepted.

🎨 Design-flavored changes

Size is applied after movers

Tyler: very annoyingMedium + back-compat risk

Confirmed: size is baked into the object's world matrix, which multiplies the mover transform (world × moverTransform) — so size scales mover orbits/offsets too, acting as a global zoom rather than per-shape size. Fix = decouple scale from the placement matrix so it applies per-copy. The math is contained, but it changes how every existing project with sized + moved instruments looks, so it likely needs a per-track or per-project opt-in rather than a silent swap.

Better motion curves for movers

Tyler: ehEasy per mover

The pattern already exists: rotation movers and bursts have full easing tables (expo/cubic/elastic/back…) exposed as an easing param. Extending the same pattern to the remaining movers (position, force-field push, oscillators) is mechanical — an afternoon per batch of movers, no research needed.

Change instrument type in place

Tyler: nice to haveAlready exists — tell Tyler

This shipped long ago and nobody knew: select the placed track, then double-click any library row — the track converts in place, keeping its children/movers, MIDI, targets and mute (params reset to the new instrument's defaults). Cycling = double-click library rows one after another; the library's hover preview even shows each instrument before you commit. A right-click "Change instrument" menu was briefly added on 2026-07-22 and removed the same day as redundant. Remaining gap is discoverability only — nothing hints that double-click means "convert selected track".

Live video preview per track

Tyler: "kinda a lot"Hard

The hover preview exists precisely because live previews are expensive — it keeps one warm WebGL canvas and only renders while hovered. N always-on live feeds (one per track, with effects applied) is a real rendering-budget and architecture project: shared render targets, low framerate/low-res scheduling, pausing off-screen rows. Very cool, genuinely 10×-professional-looking, and genuinely the most expensive item here.

A cheaper 80% version: periodically-refreshed static thumbnails (every few seconds / on param change) instead of live feeds.

💡 New idea — "3 laser orbs" instrument/template

Laser orb trio with MIDI-driven radius & spin

Tyler: "would be a great template"Medium

Tyler's spec: 3 laser orbs, each a different adjustable color, each with 2 radial splitters. MIDI notes select the radius per orb-per-splitter from 4 options (one of them 0/collapsed) — 3 orbs × 2 splitters × 4 radii = 24 notes. Radius changes trigger at note start and glide along an exponential curve from the previous radius to the new one. Additional MIDI notes control rotation per orb and per splitter — either 3 options (none / spin / reverse) or 5 (adding half- and full-speed in each direction). MIDI layout, extra splitters, and extra options are open to interpretation. Goal: makes his recent posted visuals easy to recreate; good tutorial-video candidate if the visual leads in the first 1–2 seconds.

Every ingredient already exists: laser spheres, radial splitters, mover definitions with midiRows, and exponential easing curves. The real work is composition and layout — a bespoke instrument (or a template wiring existing pieces) with ~30+ labeled MIDI rows, plus the note-start-triggered exponential radius glide, which is a new interpolation behavior between successive notes on the same row group. Fits squarely into the abilities model (bespoke, instrument-intrinsic MIDI lanes). Estimate: a couple of days for a solid v1 — and unlike most of this list, it's new content that doubles as a template + tutorial, which aligns directly with the templates/marketing push.
