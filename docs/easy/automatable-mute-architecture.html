Automatable Mute - Architecture

Tyler: "to get rid of an instrument I have to move it offscreen." The question isn't whether mute exists - three mute-shaped things already do - it's which of them removes the OBJECT over time, and whether the answer is documentation or a small feature.

1 · The three mutes that exist - and what each actually silences

2 · The verdict tree

Verification outcomeResponse
Mute modifier makes Tyler's actual instruments disappear (they're note-driven)Documentation gap, zero code: name the pattern in the README's editor tour and tell Tyler. Done.
It silences notes but the object body persists (expected for Cube-likes)Small feature: teach the Mute modifier's active region to also hide the parent OBJECT - one flag in resolve's object output (hiddenByLane), honored where the M-mute blackout is already applied in the renderer. Semantics: M button = static, Mute lane = timed, both land on the same renderer switch.
Tyler wants a fade, not a cutThat's the envelope modulator → opacity item (medium tier) - don't build a second fade mechanism here; hard-cut visibility now, fades arrive with the envelope work.

3 · Decisions & non-goals

DecisionWhy
Reuse the Mute modifier lane, don't invent a "visibility lane"The lane, its blocks, its timeline row, and its persistence all exist; extending its meaning from "mute events" to "mute events + hide body" matches what every DAW user expects Mute to do.
Hide via the existing blackout switchThe renderer already knows how to hide an object for M-mute; the lane just becomes a second, beat-dependent input to the same switch - pure function of beat, canary-safe.
No opacity ramps hereFades belong to the envelope modulator (one fade system, not two).
The one-sentence mental model: mute exists three ways but only statically for the object's body - verify that gap on a persistent instrument, and if it's real, let the Mute lane throw the same hide-switch the M button throws, just as a function of the beat.
