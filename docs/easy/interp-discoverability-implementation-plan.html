B · Phase-by-phase

Phase 0 - Never hide the control PianoRollPanel (unconditional)

  • Today: automation is only set when the parent instrument's param def resolves and is numeric; otherwise the whole automation treatment (value rows + Interp select) silently degrades to a note editor.
  • Change: for track.type === 'automation', ALWAYS build the automation view - if the param def is missing, fall back to { paramLabel: track.targetParam ?? 'value', paramMin: 0, paramMax: 1 } and show a small "param not found on parent" hint in the range gutter.
  • Verify: rename/swap the parent's instrument so the lookup fails → lane still opens as an automation lane, curve still editable.

Phase 1 - Promote in place PianoRollPanel (★ recommended)

  • Rename the label "Interp" → "Curve"; move the group to sit directly after the track-name cluster (before Snap/grid) when the lane is automation - the first thing an automation editor shows should be automation controls.
  • Slightly widen the select and title it "Animation curve between keyframes".

Phase 2 - Right-click Curve on the automation row timeline (★ optional)

  • Automation/ability rows currently have no own context menu (the add-lane menu belongs to the parent label). Add a minimal menu for automation rows: Curve ▸ the 7 modes (check the active one) - reusing the SubMenu clamping component from TrackContextMenu.
  • One store write (setTrackInterpolation) - the piano-roll select and this menu can't desync.
  • Wire: Track.tsx label onContextMenu branches on track.type === 'automation' to the new menu.

Phase 3 - Verify

  • Curve changes from either surface reflect in the other and audibly change the sampled motion (linear vs step on a position lane is unmistakable).
  • Undo steps the mode; persistence round-trips it (both already true - regression check only).
