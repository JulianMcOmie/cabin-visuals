A · Components that change — and one layer down

Badges: new edit.

new persistence/videoStorage.ts + core/video/videoSource.ts — bytes

Byte-for-byte mirrors of audioStorage.ts / audioSource.ts: upload to the private project-videos bucket at {userId}/{projectId}/{clipId}, signed URLs to read, session-only object-URL fallback when the project has no row yet, and an in-memory ref → URL cache.

new store/VideoStore.ts — the catalog

videoClips: Record<ref, { fileName, duration, width, height }>, persisted by serialize.ts exactly like audioClips. Not undoable (loading a file isn't an edit).

edit types.ts / resolve.ts / instrumentFrame.ts — plumbing

track.videoRefs?: string[] (ordered pad bank) flows resolve → ResolvedObject.videoRefs → ObjectState.videoRefs. The frame-gate signature gains one ref-compare entry (the array is a fresh identity per resolve, contents immutable — same argument as elementMatrices).

new core/video/videoTime.ts — the pure time model

activeVideoAt(notes, beat, baseNote, clipCount) → { clipIndex, noteBeat } | null and clipTimeAt(beat, noteBeat, secPerBeat, duration, loop). No DOM, no three.js — this is the testable heart.

new instruments/Video.tsx — the instrument

Def (params: baseNote, size, x/y/z, loop; midiRowLabels for the pad rows) + component: warm elements, one VideoTexture plane, per-frame reconcile (play + drift-correct vs pause + seek), invalidate() on seeked/loadedmetadata, block-gate hiding. Registers an export frame preparer on mount.

edit components/TrackEditor.tsx — the clip bank UI

A CLIPS section shown only for instrumentId === 'video': upload button (accept mp4/webm, ≤100 MB, ≤8 clips), ordered list with pad label (C3, C#3…), move up/down, remove. Writes setTrackVideoRefs (undoable — arranging the bank IS an edit) and the VideoStore catalog.

edit core/export/exportEngine.ts — frame preparers

registerFramePreparer(fn) module registry; walkFrames awaits all preparers with the frame's beat before renderFrame. Zero registered preparers = zero cost.

C · The time model — pure, tested

// The latest note-on at or before `beat` that falls inside block coverage.
// Latch: it stays active until a LATER note-on replaces it. null = hidden.
activeVideoAt(notes, beat, baseNote, clipCount):
  candidate = argmax over notes n of n.beat, where
    n.beat <= beat and beat is inside n's block span
  if none → null
  return { clipIndex: mod(candidate.pitch − baseNote, clipCount), noteBeat: candidate.beat }

clipTimeAt(beat, noteBeat, secPerBeat, duration, loop):
  t = (beat − noteBeat) × secPerBeat
  return loop ? t mod duration : min(t, duration − ε)
Block coverage nuance: the latch must not leak across block gaps — a note in bar 1 does not keep a clip alive in a block that starts at bar 9. The candidate's own blockEndBeat bounds its reign: past it, and with no later candidate, the instrument hides. This is the exact behavior the block-gated-visibility rule demands, and it's the reason the helper takes resolved notes (which carry block bounds) rather than raw ones.

Phase 2 — plumbing

  • edit editor/types.ts (videoRefs?), ProjectStore.ts (setTrackVideoRefs), core/visual/types.ts + resolve.ts (flow it), instrumentFrame.ts (gate signature entry).

Phase 3 — time model

  • new core/video/videoTime.ts + core/video/videoTime.test.ts (node --test, same harness as the engine tests): latch, restart-same-pitch, mod wrapping, loop on/off, block-gap hiding.

Phase 4 — the instrument

  • new instruments/Video.tsx; edit instruments/index.ts + LeftSidebar.tsx (OBJECT section entry, film-frame icon).
  • Element lifecycle: create/destroy on videoRefs identity change; warm all; only active plays. Texture swaps by pointing the plane's material map at the active element's texture (one VideoTexture per element — cheap, avoids rebind stalls).

Phase 5 — clip bank UI

  • edit TrackEditor.tsx: CLIPS section (upload / list with pad labels / reorder / remove). Duration/aspect probed via a throwaway element's loadedmetadata before the catalog entry is written.

Phase 6 — export seam

  • edit exportEngine.ts: preparer registry + one await in walkFrames; edit Video.tsx: register/unregister a preparer that seeks the active element to the exact clipTime and resolves on seeked (resolves immediately when already within half a frame).

F · Risks & open questions

  • risk Seek latency on live cuts — a note-on mid-playback seeks a warm element; keyframe-sparse encodes can take >100 ms. Mitigation: warm elements + accepting one late cut live (export is exact regardless). If it bothers in practice: WebCodecs decoder path later.
  • risk Memory — 8 warm decoded videos is real GPU/RAM pressure; caps enforced at upload. Watch for Chrome tab-kill on big projects.
  • risk Signed URL expiry (1 h) — an element erroring mid-session re-resolves its URL once before surfacing an error.
  • risk Autoplay policy — elements are muted, so Chrome permits programmatic play; Safari stays out of scope (export already is Chrome-first).
  • Open — should velocity map to anything (opacity? rate)? Deferred until a user asks.
