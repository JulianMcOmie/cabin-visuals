Video Export - Implementation Plan

Turning video-export-architecture.html into specific files, types, and functions: a core/export/ island that steps the beat frame-by-frame through the existing visual engine, encodes with WebCodecs, renders audio offline through the shared placement math, and muxes an MP4 in the browser.

This plan introduces one new directory - src/editor/core/export/ (frame stepper, WebCodecs encode, offline audio render, MP4 mux, capability gate) - plus a small refactor that extracts the beat⟷second placement math out of AudioEngine into a shared helper, an export dialog, and wiring for the header's existing (dead) Export button. No document version bump - export writes a file, not the project. Decisions baked in: recompute frame-by-frame, never record · borrow the live canvas (frameloop off, pinned size, DPR 1) · MP4 = H.264 + AAC via mp4-muxer · 1080p/60 default, fixed 16:9 · include-audio toggle (on by default) · whole project length · capability-gated (Chrome-first, confirmed) · encoder backpressure via encodeQueueSize.

A · Components that change - and one layer down

The highest level first, then a peek at the key function inside each. Files in monospace; status badges: new edit.

new core/export/exportEngine.ts - the loop

One async function runExport(settings, hooks): compute N = ceil(totalBeats · 60/bpm · fps); for each frame set the beat, ask the driver for one render, wrap the canvas in a VideoFrame (timestamp i·1e6/fps µs), feed the encoder, await when encodeQueueSize > 2. Reports onProgress(i, N); checks an AbortSignal every frame. Runs the audio render in parallel and finalizes the mux.

new core/export/audioRender.ts - offline mix

renderAudioTrack(blocks, clips, bpm, beatsPerBar, durationSec) → AudioBuffer: an OfflineAudioContext(2, durationSec·48000, 48000), one AudioBufferSourceNode per block placed with the shared blockPlacement() helper (extracted from AudioEngine.armBlock - the from-zero case), master gain 0.85 to match live playback. Then AAC via AudioEncoder.

edit core/audio/AudioEngine.ts - extract, don't duplicate

The three-case beat⟷second math moves to core/audio/placement.ts as a pure function; armBlock calls it with the live start beat, audioRender calls it with beat 0. One source of truth - live playback and export can't drift apart. Behavior change: none (verified by the phase-3 canary).

new components/visual/ExportDriver.tsx - inside the Canvas

Mounted next to VisualBeatSync; exposes { renderFrame(beat), pin(w,h), unpin() } to the engine via a ref/registry. renderFrame sets TimeStore.currentBeat and calls R3F's advance() once - so VisualBeatSync → computeAtBeat runs exactly as in live scrubbing. pin switches frameloop to never, sets renderer size to export dims, DPR to 1; unpin restores everything.

C · Settings model - no document change

Export is an action, not state: nothing about it belongs in the project document, so there is no schema bump (the v2 upgrade chain is untouched). Settings live in component state with a localStorage nicety, and the one shared type lives in the export island:

interface ExportSettings {
  width: number          // 1920 | 1280 (fixed 16:9)
  height: number         // 1080 | 720
  fps: 30 | 60           // default 60 (decided)
  includeAudio: boolean  // default true - off = video-only MP4
  videoBitrate: number   // default 12 Mbps @1080p60, 8 @720p
  fileName: string       // default: project name
}
Duration is derived, never entered: totalBeats = totalBars · beatsPerBar, durationSec = totalBeats · 60/bpm, frameCount = ceil(durationSec · fps). The project-length control (the BARS scrubber) is the export length control.

D · The export engine - core/export/

D.1 - the frame loop (the one place export timing lives)

for (let i = 0; i < frameCount; i++) {
  if (signal.aborted) break
  const beat = i * bpm / (60 * fps)          // pure arithmetic - no wall clock
  driver.renderFrame(beat)                     // setCurrentBeat + advance() - same path as scrubbing
  const frame = new VideoFrame(canvas, { timestamp: i * 1e6 / fps })
  encoder.encode(frame, { keyFrame: i % (fps * 2) === 0 })   // keyframe every 2s
  frame.close()
  while (encoder.encodeQueueSize > 2) await nextEvent(encoder, 'dequeue')
  if (i % fps === 0) onProgress(i, frameCount)
}
  • VideoFrame from the canvas, same task as the render - no pixel readback, no preserveDrawingBuffer. The GPU surface is still valid because nothing else rendered in between.
  • Backpressure, not buffering: the loop never holds more than a few frames; a 5-minute 1080p60 export stays flat on memory.
  • Config: avc1.640028 (High 4.0) at 1080p, latencyMode: 'quality', chunks streamed straight into the muxer's addVideoChunk.

D.2 - audio (audioRender.ts) and the shared placement helper

  • new core/audio/placement.ts: blockPlacement(block, clipDuration, bpm, beatsPerBar, fromBeat) → { when, offset, duration } | null - the exact three-case math lifted from armBlock. AudioEngine calls it with the live seek beat; export calls it with fromBeat = 0.
  • Decode through the existing core/audio/waveform.ts buffer cache (bytes via audioSource.ts) - export never fetches what playback already decoded.
  • Honor mute/solo the same way setBlocks does (audible = !muted && !(anySolo && !solo)) and the 0.85 master gain, so the file sounds like the editor.
  • Encode: AudioEncoder, mp4a.40.2 (AAC-LC) 48 kHz stereo 192 kbps, feeding addAudioChunk. No audio track in the project → video-only MP4, no special casing downstream.

D.3 - mux + save (mux.ts) and the gate (support.ts)

  • mp4-muxer (new dependency, ~zero-dep, tree-shakes small) with ArrayBufferTarget; finalize → Blob → anchor download (showSaveFilePicker when available, silent fallback otherwise).
  • isExportSupported(): checks VideoEncoder/AudioEncoder existence and isConfigSupported for the avc + aac configs at startup; the Export button tooltips "requires Chrome" and disables when false. The WebM fallback is a named seam behind this gate - not built now.

E · The export dialog & the frame driver

  • new components/ExportDialog.tsx - opened by the header button: resolution (1080p/720p), fps (60/30, default 60), an Include audio toggle (on by default; off skips the audio render entirely and produces a video-only MP4), filename; then a progress view (frame i/N, elapsed, est. remaining, encode-vs-realtime speed) with one Cancel button. Body scroll-locked; transport keys suppressed while open.
  • edit App.tsx - the Export button gets its first onClick (opens the dialog) and the disabled/tooltip state from isExportSupported().
  • new components/visual/ExportDriver.tsx - see A. The subtle part is pin/unpin: save {frameloop, size, dpr}, set frameloop='never' + export size + DPR 1, and restore in a finally so a thrown encode error can never leave the editor frozen at 1920×1080.
  • While exporting, the canvas behind the dialog shows the frames as they render - free progress feedback, since the playhead genuinely moves through the project (setCurrentBeat per frame, exactly like a fast scrub).
Why borrow the live canvas instead of a hidden one: the visual engine is a module singleton (computeAtBeat/getObjectState) and VisualBeatSync assumes one scene. A second <Canvas> would double-drive the engine and double GPU memory. Pinning the existing canvas behind an overlay is one seam, zero new engine semantics - and the pause canary keeps watching for free.

Phase 1 - Deterministic frame driver ExportDriver

  • new components/visual/ExportDriver.tsx (renderFrame/pin/unpin); new core/export/exportEngine.ts first half - the beat stepper walking frames with a no-op sink.
  • Verify - the determinism canary: render frames 0/100/200 twice via canvas.toDataURL() hashes - byte-identical across runs; pin to 1920×1080 on a small window and confirm aspect-aware instruments (FractalTunnel et al.) compose for the export aspect, not the window's; unpin restores the editor pixel-perfect.

Phase 2 - Video encode + mux ★ keystone videoEncode · mux

  • new core/export/videoEncode.ts (encoder config + backpressure) and mux.ts; engine loop feeds real frames; download lands.
  • Verify: a 10-second project exports to an .mp4 whose duration is exactly totalBeats·60/bpm (±1 frame), plays in Chrome + QuickTime, no dropped/duplicated frames (count via ffprobe once, manually); memory flat across a 3-minute export; cancel mid-way leaves no dialog, no pinned canvas, no orphaned encoder.
The keystone canary: export a project with a note at beat 0 and a note on every bar line, then step through the file frame-by-frame (comma/period in mpv or QuickTime) - every visual onset must land on the exact expected frame (bar · beatsPerBar · 60/bpm · fps). If onsets drift or double, the beat(i) map or the timestamp math is wrong; fix before audio lands on top.

Phase 3 - Audio offline render + AAC audioRender AudioEngine (extract)

  • new core/audio/placement.ts (extracted three-case math) + core/export/audioRender.ts; edit AudioEngine.ts to call the shared helper (behavior-neutral - live playback must sound identical before/after, same canary as the audio plan's phase 2).
  • Verify: export with the audio block at bar 0 - waveform of the file's audio aligns with the source clip sample-exactly at t=0; move the block to bar 8 - audio enters at exactly 8·beatsPerBar·60/bpm seconds; trim honored; mute silences it in the file; no audio track → valid video-only mp4; Include audio off → identical video-only file even when an audio track exists.

Phase 4 - Dialog polish ExportDialog

  • Progress (i/N, speed ×realtime, est. remaining), cancel via AbortSignal, error surface (encoder/mux failures land as a readable message, never a stuck overlay), transport + shortcuts locked during export, settings remembered in localStorage.
  • Verify: cancel at 5% / 50% / 95%; force an encoder error (absurd bitrate) and confirm clean recovery; export twice in a row without reload.

Phase 5 - Later deferred

  • Range/region export (loop region when that exists) · WebM/VP9+Opus fallback behind the same gate · image-sequence export (PNG per frame, for external encoding) · alpha/transparency · custom resolutions/aspects · server render for non-Chrome users.

The one-sentence mental model: you're adding a core/export/ island that walks the beat through the untouched visual engine one frame at a time and streams the canvas into WebCodecs, mixes the audio track once through placement math now shared with the live engine, muxes both into an MP4 in memory, and hands you the file - the transport, the document, and every instrument never learn export exists.

H · Risks & open questions

  • risk WebCodecs availability. Chrome/Edge: full. Safari/Firefox: partial-to-none for this config. Mitigation: the phase-0 gate (button disabled with a reason) and a named WebM fallback seam. Decided: Chrome-first is acceptable.
  • risk AAC encoder missing where H.264 exists. isConfigSupported both at gate time; if only AAC fails, offer video-only export rather than blocking.
  • risk VideoFrame-from-canvas timing. The frame must be constructed in the same task as the render (before anything else touches the GL context). The loop is structured render → capture → encode sequentially, so this holds by construction; a regression shows up as black/stale frames in the phase-1 hash canary.
  • risk Per-frame React cost. setCurrentBeat re-renders beat readouts every frame. Fine at scrub speed today; if it caps export speed, throttle the UI subscription during export (engine correctness is unaffected - it reads the store directly).
  • risk Memory on long projects. Chunks stream into the muxer, but ArrayBufferTarget holds the finished MP4 in RAM (~1 GB/10 min at 12 Mbps - fine). If projects grow, swap to FileSystemWritableFileStreamTarget; the seam is one constructor argument.
  • risk Pin/unpin leaving the editor wedged. Everything restorative lives in one finally; phase 4 tests cancel and forced-error paths explicitly.
  • open Default fps. Decided: 60 (the visuals are motion-heavy); 30 stays in the dialog for smaller/faster exports.
  • open Export aspect vs window aspect. Fixed 16:9 is the decision; instruments that read viewport re-compose to it (that's the point of the aspect-aware work). Worth one visual pass over the fullFrame instruments at 16:9 during phase 1.
  • later Ranges, stems, WebM, image sequences, alpha, server render - named in the architecture doc §5; none designed here.
