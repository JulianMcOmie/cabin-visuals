Undo & Redo

A snapshot-based history that records the project document, coalesces drags automatically, and never asks a single action to know how to reverse itself. Roughly seventy lines, zero dependencies, zero changes to existing code.

01  The decision

DimensionChoiceWhy
RepresentationSnapshot (memento)The store is already immutable - snapshots structurally-share and cost ~the diff. Bulletproof; restore is one assignment.
GranularityThe project store, wholeOne history for the whole document, not per-action reverse logic. Actions stay oblivious.
What's capturedEvery non-function fieldGeneric - instrument settings (in tracks) and future effects come along for free, no history edits.
RecordingStore subscriptionNo middleware, no zustand-mutator type gymnastics. Fires on every change; a cheap filter ignores no-ops.
Coalescing~200ms debounceA drag's frames are <16ms apart → one entry; deliberate edits are spaced → separate. No gesture-hook changes.
SurfaceKeyboard only⌘/Ctrl Z · ⌘/Ctrl ⇧ Z · Ctrl Y. No buttons.

Considered & rejected for v1: immer forward/inverse patches (Tyler's approach). Strictly more machinery - an immer dep, a typed set middleware, and a rewrite of all ~15 actions into draft-mutation style just to get granular patches - for a benefit (memory at huge scale, collab/OT) you don't need yet. And the public API is identical, so it's a low-regret swap later if you ever do.

02  What's undoable

The rule that makes the whole thing hold: the ProjectStore is the serializable project document. Undo snapshots it wholesale; anything ephemeral stays out.

StateLives inUndoable?
Tracks, blocks, notes, hierarchyProjectStore.tracks / rootTrackIdsyes
Instrument settingstrack.params - inside tracksyes already covered
Effects (when added)put on track, like track.effects[]automatic
Selection · zoom · scrollUIStoreno ephemeral
Playhead positionTimeStore.currentBeatno
BPM · total bars · beats/barTimeStorelater move into ProjectStore to include
Already covered, for free: instrument settings live at track.params and setTrackParam writes them immutably (new track → new tracks). Because the snapshot grabs every non-function field, the day you add track.effects it's captured with no change to the history code.

03  How it works

edit → ProjectStore → subscribe fires
capture the pre-edit doc → debounce 200ms
burst → one entry → push to past, clear future

Three mechanics carry the whole design:

  • Capture the previous document, not the next. Undo means "go back," so each entry stores the state as it was before a burst of edits began. The store subscription hands you (state, prev) - prev is exactly that.
  • One burst = one entry. The first change in a burst stashes its prev as the restore point; subsequent changes within 200ms only reset the timer. When it goes quiet, that single restore point is committed. A 60fps drag collapses to one undo step without any gesture-hook involvement.
  • Guard the restores. Undo/redo write the store too - an applying flag makes the subscription ignore those writes so they don't record themselves.

04  The build

  1. 1The history engine - one new file. The store, the subscription, the debounce.
src/editor/store/HistoryStore.tsnew
import { create } from 'zustand'
import { useProjectStore } from './ProjectStore'

type Snapshot = Record<string, unknown>
const LIMIT = 100          // cap the stack; oldest entries fall off
const DEBOUNCE_MS = 200   // burst window: drags coalesce, deliberate edits don't

// The undoable document = every non-function field of the project store.
// Generic on purpose: add track.effects / a project name later and it's covered.
const pick = (s: Record<string, unknown>): Snapshot => {
  const out: Snapshot = {}
  for (const k in s) if (typeof s[k] !== 'function') out[k] = s[k]
  return out
}
const snapshot = () => pick(useProjectStore.getState() as Record<string, unknown>)
const changed = (a: Snapshot, b: Snapshot) => {
  for (const k in a) if (a[k] !== b[k]) return true   // reference compare - immutable store
  return false
}

interface HistoryState {
  past: Snapshot[]; future: Snapshot[]
  undo: () => void; redo: () => void; clear: () => void
}

// Module-level transient state (not reactive - it's plumbing).
let applying = false                  // true while a restore is in flight
let pendingBase: Snapshot | null = null // pre-burst restore point
let timer: ReturnType<typeof setTimeout> | null = null

export const useHistoryStore = create<HistoryState>((set, get) => {
  const flush = () => {            // commit the in-flight burst as one entry
    if (timer) { clearTimeout(timer); timer = null }
    if (!pendingBase) return
    const base = pendingBase; pendingBase = null
    const past = [...get().past, base]
    if (past.length > LIMIT) past.shift()
    set({ past, future: [] })
  }

  const restore = (snap: Snapshot) => {
    applying = true
    useProjectStore.setState(snap)   // shallow-merges data back; actions untouched
    applying = false
  }

  // Record: the subscription is the only writer of `past`.
  useProjectStore.subscribe((state, prev) => {
    if (applying) return
    const a = pick(state as Record<string, unknown>)
    const b = pick(prev  as Record<string, unknown>)
    if (!changed(a, b)) return
    if (!pendingBase) pendingBase = b        // stash PRE-edit doc at burst start
    if (get().future.length) set({ future: [] }) // any edit kills redo
    if (timer) clearTimeout(timer)
    timer = setTimeout(flush, DEBOUNCE_MS)
  })

  return {
    past: [], future: [],
    undo: () => {
      flush()                              // commit any in-flight edit first
      const { past, future } = get()
      if (!past.length) return
      const current = snapshot()
      restore(past[past.length - 1])
      set({ past: past.slice(0, -1), future: [...future, current] })
    },
    redo: () => {
      flush()
      const { past, future } = get()
      if (!future.length) return
      const current = snapshot()
      restore(future[future.length - 1])
      set({ past: [...past, current], future: future.slice(0, -1) })
    },
    clear: () => set({ past: [], future: [] }),
  }
})
The subscription is wired inside the store initializer, so it activates the moment anything imports useHistoryStore - there's no separate "init" call to forget. canUndo/canRedo aren't methods because you don't need them (no buttons); if you ever do, they're useHistoryStore(s => s.past.length > 0).
  1. 2The shortcuts - one small hook. Guards text fields; handles both redo conventions.
src/editor/hooks/useUndoRedoKeys.tsnew
import { useEffect } from 'react'
import { useHistoryStore } from '../store/HistoryStore'

// ⌘/Ctrl+Z = undo · ⌘/Ctrl+Shift+Z = redo · Ctrl+Y = redo (Windows). Mount once.
export function useUndoRedoKeys() {
  useEffect(() => {
    const onKey = (e: KeyboardEvent) => {
      const t = e.target as HTMLElement | null
      if (t && (t.tagName === 'INPUT' || t.tagName === 'TEXTAREA' || t.isContentEditable)) return
      if (!(e.metaKey || e.ctrlKey)) return
      const { undo, redo } = useHistoryStore.getState()
      if (e.key === 'z' || e.key === 'Z') { e.preventDefault(); e.shiftKey ? redo() : undo() }
      else if (e.key === 'y' || e.key === 'Y') { e.preventDefault(); redo() }
    }
    window.addEventListener('keydown', onKey)
    return () => window.removeEventListener('keydown', onKey)
  }, [])
}

That's the whole feature. Two new files, one import line. No middleware, no immer, no changes to any of the ~15 ProjectStore actions, and no edits to the gesture hooks - the debounce does the drag-grouping that explicit transactions would otherwise require.

05  Edge cases - already handled

  • Undo with the MIDI editor open. Undo restores the store's block; useMidiEditorState already detects a block change that isn't its own save and re-syncs the local notes - the piano roll updates on undo with no extra work.
  • Drags. Timeline block move/resize/draw and the MIDI block-header drag write the store every frame; the 200ms debounce folds each into a single entry. Note edits are already debounced to one store write by useMidiEditorState (500ms), so they're one entry too.
  • Undo mid-burst. undo() flushes the pending burst first, so an edit you just made (still within its debounce) commits and then immediately undoes - no lost or doubled step.
  • Re-recording on restore. The applying flag makes the subscription skip undo/redo's own writes.
  • Typing. The shortcut bails inside inputs/textareas, and preventDefault stops the browser's native undo from fighting it.

06  Deliberately out of scope - and the upgrade paths

Not nowIf you want it later
Per-action precision - fast back-to-back discrete edits (<200ms) can merge into one undo.Add explicit begin()/end() transactions around the two continuous drag flows. Slots in behind the same API; no call-site churn elsewhere.
BPM / timeline settings undo.Relocate those few fields into ProjectStore (they're "project" data) - then they're captured automatically. Or have the snapshot read both stores.
Patches (memory at huge scale, collaborative editing / OT, compact persisted history).Swap the entry representation from snapshot to immer patch-pairs. undo/redo/clear and the coalescing seam are unchanged - internal-only.
Undo labels / a history panel.Store a label alongside each entry (derive from which fields changed); the stacks are already in a reactive store.
