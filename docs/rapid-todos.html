Automation for effect parameters

Automation currently targets instrument params (track.targetParam → engine overlay). Effect params live in track.effects[i].settings, so lanes need a richer target (effectInstanceId + settingKey), resolve/engine need to sample and overlay them, and the wrappers (Transform/Clone/Shader) need to read overlaid values. Medium-hard; touches the automation model end to end — and it's the enabler for the symmetric-effects item.

🔴 Hard / long-term — plan before touching

Per-instrument "plugin" UI (the Logic-style instrument window)

The real problem you named: param-adjustment friction. This is an architecture piece — a per-instrument custom editor surface (draggable/resizable manipulation, immediate feedback), a registry hook like def.editorComponent, and probably direct-manipulation overlays in the 3D view. Worth a design doc of its own; start by picking ONE instrument (Cube?) as the pilot. Interim cheap win: better slider UX (type-in values, fine-drag modifier).

MIDI vim

A modal editing language for the piano roll. Big, and it deliberately overlaps with row-hiding/scaling and fixed-distance note entry — design them together (the row-model work above is the foundation). Needs its own doc + keymap design before any code.

Incremental resolve

Known and deliberately deferred: today an edit re-resolves the whole project (debounced 80ms, params fast-path is separate) and the docs sketch the incremental design (per-track caching, reference dirty-diffing). Don't build it until a real project makes edits feel slow — full resolve is cheap at current scale, and incremental invalidation across modifier chains + routings + (now) loops is genuinely fiddly.
