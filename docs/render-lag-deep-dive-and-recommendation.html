Render Lag — Deep Dive & Recommendation

Why certain instruments freeze the whole editor — worse on Windows — and the staged fix. The short version: rendering is not on a separate thread, and the heaviest instruments re-upload a full-screen bitmap to the GPU every frame down a path that is cheap on Metal and expensive on Windows' Direct3D.

1 · The one assumption to unlearn

The mental model going in was "the rendering happens on a different thread than the editor." It does not. There is no OffscreenCanvas and no Web Worker anywhere in the codebase — the single R3F <Canvas> (App.tsx:44) runs its render loop on the browser's main thread, the exact same thread as React, the timeline, and your clicks and drags.

What is genuinely parallel is only the final GPU rasterization (separate silicon). Everything leading up to each frame — computing per-beat state, drawing to 2D canvases, uploading textures, issuing WebGL draw calls — is JavaScript on the main thread, serialized with the UI. So a frame that takes 40 ms is a 40 ms hole in your scrubbing. That is the entire reason a heavy instrument lags the whole editor and not just the canvas.

2 · What runs every frame

The loop is frameloop="always" (unset → R3F default): it renders every frame forever, whether or not anything changed. Each frame, on the main thread, in order:

  1. computeAtBeat(beat) (VisualBeatSync.tsx:24 → VisualEngine.ts:63) — resolves every object's per-frame state: modulators, automation samples, world-matrix composition, and an active-notes filter that scans the track's full note array (VisualEngine.ts:91). Arithmetic; scales with tracks/notes. The expensive structural re-resolve is correctly debounced 80 ms off this path.
  2. Per-object placement (ObjectRenderer.tsx:28) — one useFrame per object writes its transform. Cheap.
  3. Each instrument's callback (instrumentFrame.ts:19) — where the cost lives. A positioned cube updates a material (cheap). A full-frame Canvas-2D instrument redraws a big bitmap and re-uploads it (expensive — §3).
  4. The GL render — draw calls plus every dirty texture/buffer upload. The uploads are the Windows-expensive part.
There are zero performance guards. No DPR cap, no adaptive quality, no render-on-demand, no throttle — grepped and confirmed. The only place frameloop/dpr are ever touched is the export path, which pins them and restores them. During editing, the loop always runs at full device pixel ratio (up to 2×) with every mounted instrument active.

3 · The heavy path — a full-screen bitmap, re-uploaded every frame

The expensive instruments are the full-frame Canvas-2D → CanvasTexture ones. Each frame they (a) draw an effect into a 2D canvas of up to ~2 million pixels, then (b) hand the whole canvas to the GPU as a texture via texImage2D. Both steps are pure main-thread cost, and step (b) is the platform splitter.

The upload is unconditional: every one of these sets texture.needsUpdate = true every frame regardless of whether the pixels changed (e.g. FractalTunnel.tsx:453, CrtScanlines.tsx:207, WindowsXP.tsx:1114, SilkSymmetry.tsx:376). FractalTunnel draws its canvas up to three times per frame when note-pulses are active (main + two offscreen compositing passes), then uploads.

4 · Why Windows and not Mac

The instrument code is identical on both platforms; the divergence is in the browser's WebGL backend, specifically the texImage2D canvas-upload step:

  • Mac runs WebGL on Metal. Uploading a 2D canvas into a texture is cheap and tightly integrated.
  • Windows Chrome runs WebGL through ANGLE, which translates every call into Direct3D 11. Re-uploading a multi-megabyte 2D canvas to a D3D texture every frame is a known slow path — format conversion plus system-memory-to-GPU copies that cost multiples of the Metal path. Same code, the same 7–8 MB, many times the per-frame cost.

Cross the ~16.7 ms budget (60 fps) on that upload and the frame overruns; because it is on the main thread, the overrun is editor lag. Mac has more headroom before it crosses, so it usually doesn't — but it will, with enough heavy instruments stacked, high particle counts, or a large retina canvas. Same mechanism, higher threshold.

Confidence, stated honestly. The single-thread loop, unconditional uploads, canvas sizes, particle counts, and absence of perf guards are verified in the code. The ANGLE/Direct3D upload-cost explanation is the well-established mechanism for exactly this symptom, but it has not been measured on the specific machines. Before investing in the biggest fixes, confirm with a Chrome DevTools → Performance capture on the Windows machine while a laggy instrument plays: you want to see long texImage2D / GPU-commit bars and the instrument's frame callback eating the budget. A cheap corroborating tell: does the lag get worse when you enlarge the visual panel? If yes, it is upload-bound, which nails the diagnosis. The tier-1 fixes below are worth doing regardless of what the trace shows.

5 · The recommendation — three tiers

The mechanisms below stack. Ship tier 1 first; it fixes the worst-felt symptom (editing lag) cheaply and buys time to measure. Tier 2 is the direct, self-correcting fix for the Windows playback gap. Tier 3 is the durable structural payoff.

Tier 1 — Make editing instant, regardless of instruments cheap · ship first

Render on demand while paused. The pause invariant guarantees a paused editor is a static frame (every visual is a pure function of the beat; a frozen beat is a frozen picture — the dev canary already enforces this). So when the transport is paused there is nothing to render. Flip frameloop to "demand" and invalidate() only on the events that actually change the picture: play, scrub, edit/param-drag, panel resize. While you edit a paused project the render loop goes idle and the main thread is free — the editor feels instant even with DiamondLattice and FractalTunnel both mounted. This does not speed up playback; it removes the "the whole app is unusable while I'm working" feeling, which is the most infuriating part.

Plus two static caps that cost nothing: clamp the canvas DPR (dpr={[1, 1.5]} — fewer framebuffer pixels, negligible visual change at editor size), and gate needsUpdate so a texture only re-uploads when its canvas actually changed (with render-on-demand, a paused editor then uploads exactly zero textures).

Tier 2 — Make playback self-correct on slow machines medium

An adaptive quality governor. During playback, measure a rolling average frame time. When it exceeds the budget, step a global quality scalar down (1.0 → 0.75 → 0.5); when there's comfortable headroom, step back up. The scalar feeds three knobs: canvas DPR, the full-frame instruments' canvas resolution, and particle counts. Result: a heavy scene on a Windows laptop automatically downscales until it fits the frame budget, while the identical scene on a fast Mac stays at full quality. This is standard dynamic-resolution-scaling and it is the direct answer to "lags on Windows but not Mac" — the app adapts to the machine instead of assuming one. R3F ships the building block (the performance/regress() API) to hang this on.

Pair it with a hard resolution cap on the fixed-1080p offenders (DiamondLattice, WindowsXP) — there is no reason those draw at 1920×1080 for a viewport that's a few hundred pixels tall; aspect-scaling them like FractalTunnel already quarters their cost.

Tier 3 — Kill the cost structurally big · durable

(A) Port the worst Canvas-2D instruments to GPU fragment shaders. Compute the effect in a shader on a fullscreen plane — no per-frame 2D draw, no texImage2D upload at all. This deletes the entire Windows-specific cost for that instrument. FractalTunnel, CrtScanlines, DiamondLattice, and SilkSymmetry are all procedural and shader-portable, and the codebase already runs GLSL (the effects ShaderWrapper). The pause invariant survives (the shader reads beat as a uniform). Cost: a real per-instrument rewrite — do the top four, biggest-first.

(B) Move the whole render loop to an OffscreenCanvas in a Web Worker. Transfer the canvas to a worker and run R3F there; a heavy frame then stalls the worker, not the main thread, so the editor UI stays responsive even during heavy playback. This is the architecturally correct answer to "why does the whole editor lag" — but it is a large lift (stores/engine/instruments assume main-thread + DOM access; Canvas-2D instruments need OffscreenCanvas-2D in the worker; the export path borrows the live canvas and would need rework). Only pursue B if playback responsiveness during heavy visuals is still unacceptable after tiers 1–3.

The one-paragraph verdict: the render loop shares one thread with the editor, and the heaviest instruments re-upload a full-screen bitmap to the GPU every frame down a path (ANGLE→Direct3D) that is far pricier on Windows than Mac — so those instruments blow the frame budget on Windows first and, because it's the same thread, take the whole editor down with them. Fix it in order: render-on-demand-while-paused makes editing instant on any machine for almost no code; an adaptive quality governor makes playback self-correct on slow ones; and shader-porting the top Canvas-2D offenders removes the Windows cost at the source.
