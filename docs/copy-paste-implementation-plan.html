1 · Scope & interpretation

Four related editing affordances, all of which revolve around selection → act on selection → place relative to the playhead:

FeatureTriggerOperates onContext
Copy / paste notes⌘/Ctrl + C · ⌘/Ctrl + VSelected MIDI notesMIDI editor open
Copy / paste blocks⌘/Ctrl + C · ⌘/Ctrl + VSelected blocksTrack timeline
Duplicate trackAlt + drag a trackOne track (+ contents)Track timeline
Delete trackDelete / BackspaceSelected trackTrack timeline
Reading "ctrl p ctrl v": interpreted as copy = ⌘/Ctrl + C and paste = ⌘/Ctrl + V (⌘ on macOS, Ctrl on Windows/Linux via e.metaKey || e.ctrlKey). If you actually want a different chord for copy, say so - only one line changes.

The unifying idea is "paste at the playhead": pasted content is repositioned so its earliest element lands at currentBeat, preserving the internal time/pitch offsets between copied items.

2 · Data model & current state

The shapes we'll be cloning and placing (src/editor/types.ts):

interface Note  { id; startBeat; durationBeats; pitch; velocity }   // startBeat is block-LOCAL (beats)
interface Block { id; startBar; durationBars; loop; notes: Note[] }  // startBar is BAR-based (int)
interface Track { id; name; instrumentId; params?; color; muted; solo;
                blocks: Block[]; parentId?; childIds: string[]; targets? }

What already exists (reuse)

  • useNoteGestures.ts - local selectedNoteIds: Set<string>, alt-drag-duplicate of notes, and a window keydown that handles Delete/Backspace/Esc for notes.
  • UIStore.ts - selectedBlockIds: Set<string>, selectedTrackId, and editingBlock (which block is open in the MIDI editor - our context signal).
  • ProjectStore.ts - addTrack, addBlock, addNote, updateBlockNotes, deleteBlock, reorderRootTracks.
  • useTransportKeys.ts - the window-keydown pattern (guards INPUT/TEXTAREA/contentEditable) we'll mirror.
  • Track reorder via dnd-kit sortable (App.tsx handleTrackDragEnd, Track.tsx useSortable; the label column carries the drag listeners).
  • IDs everywhere via crypto.randomUUID().

What's missing (build)

  • No clipboard abstraction.
  • No deleteTrack and no duplicate action in ProjectStore.
  • No undo/history - every op here is destructive and irreversible (see §8).

3 · Shared infrastructure

3.1 - A clipboard store new file

src/editor/store/ClipboardStore.ts - a tiny zustand store holding a tagged union, so paste knows what it's pasting:

type Clip =
  | { kind: 'notes';  notes: Note[] }
  | { kind: 'blocks'; blocks: Block[]; sourceTrackId: string }

interface ClipboardState { clip: Clip | null; setClip: (c: Clip | null) => void }

Stored snapshots are normalized at copy time (earliest element shifted to 0) so paste math is "add the playhead offset" with no per-paste bookkeeping.

3.2 - Cloning helpers co-located, no new file

Deep-clone with fresh IDs, one per level. These live next to their only consumer rather than in a shared file: cloneBlock + cloneTrack as module-private fns in ProjectStore.ts (used by pasteBlocks / duplicateTrack); the note clone is a one-liner inlined in useNoteGestures.ts. The chain is track → block → note:

const cloneNote  = (n: Note)  => ({ ...n, id: crypto.randomUUID() })
const cloneBlock = (b: Block) => ({ ...b, id: crypto.randomUUID(), notes: b.notes.map(cloneNote) })
const cloneTrack = (t: Track) => ({ ...t, id: crypto.randomUUID(), blocks: t.blocks.map(cloneBlock),
                              childIds: [], name: t.name + ' copy' })
Nested tracks: cloneTrack above drops childIds (clones the track shallowly w.r.t. hierarchy). Recursively cloning a subtree - and remapping parentId/targets to the new IDs - is a follow-up; the common case (a root track with blocks) works as written. Flagged again in §6 & §8.

3.3 - Keyboard layering & context

The same chords mean different things depending on whether the MIDI editor is open. Each gesture hook owns its own shortcuts, co-located with the selection state it manages - both are single-mount and already mirror each other, so this is symmetric (no dedicated keyboard file):

ContextOwnerHandles
editingBlock != null (MIDI editor mounted)useNoteGestures.ts (extend its existing listener)C/V/Delete for notes
Otherwise (timeline)useTrackGestures.ts (extend it - mounted once in TimelineArea)C/V for blocks; Delete for blocks or track

The timeline handler must bail when editingBlock is set so it never competes with the note handler. Both reuse the INPUT/TEXTAREA/contentEditable guard from useTransportKeys.

Why the split (answering an earlier question): it's driven by where selection lives - note selection is local to useNoteGestures; block/track selection is owned by useTrackGestures. A single top-level handler couldn't reach the note selection without lifting it into a store.

4 · Copy/paste MIDI notes at the playhead

edit useNoteGestures.tsedit MidiEditor.tsxClipboardStore

Copy

Snapshot the selected notes, normalized to the earliest startBeat:

const sel = notes.filter(n => selectedNoteIds.has(n.id))
const base = Math.min(...sel.map(n => n.startBeat))
setClip({ kind: 'notes', notes: sel.map(n => ({ ...n, startBeat: n.startBeat - base })) })

Paste

Notes are block-local; the playhead is global. Convert, place, re-ID, select:

  1. Resolve the active block from editingBlock; its global start in beats is block.startBar * beatsPerBar.
  2. playheadLocal = currentBeat − block.startBar * beatsPerBar - used as-is, even if negative or past the block end (no clamp, no "is the playhead inside the block" check).
  3. For each clip note: cloneNote, then startBeat = playheadLocal + clipNote.startBeat. No quantize snap on paste - land exactly at the playhead.
  4. onNotesChange([...notes, ...pasted]) and set selectedNoteIds to the pasted IDs (so an immediate drag/delete targets the paste).

Edge cases

  • Playhead before the block → notes get a negative local startBeat; allowed by decision. Heads-up: the grid may render them off the left edge / clip them - visual only.
  • Notes past the block end → allowed to overflow, no durationBeats clamp.
  • Repeated ⌘/Ctrl + V pastes at the same spot each time (playhead-anchored) - acceptable; a "paste then nudge" stack is out of scope.

5 · Copy/paste MIDI blocks at the playhead

extend useTrackGestures.tsClipboardStoreUIStore.selectedBlockIdsProjectStore.pasteBlocks

Copy

Gather blocks across tracks by selectedBlockIds, remember the source track, normalize to the earliest startBar:

const picked = allBlocksWhereSelected()              // scan tracks for ids in selectedBlockIds
const base = Math.min(...picked.map(b => b.startBar))
setClip({ kind: 'blocks', sourceTrackId, blocks: picked.map(b => ({ ...b, startBar: b.startBar - base })) })

Paste

  1. Target track = selectedTrackId. If nothing is selected → no-op.
  2. Playhead → bar with no snap: targetBar = currentBeat / beatsPerBar (fractional startBar is fine - a block can start mid-bar; it renders at startBar × barWidth).
  3. For each clip block: cloneBlock (fresh block and note IDs), then startBar = targetBar + clipBlock.startBar.
  4. addBlock(targetTrackId, pasted) for each; set selectedBlockIds to the new IDs.
Add a pasteBlocks(trackId, blocks) action on ProjectStore so the multi-add is atomic (one set). cloneBlock lives here too.

Edge cases

  • Overlapping blocks on a track - the model allows it today; leave as-is unless you want collision handling.
  • loop flag and all notes ride along via cloneBlock.
  • Multi-track copy pasting into a single target track collapses everything onto that track (relative bar offsets preserved, track identity lost) - acceptable for v1; true multi-track paste is a later enhancement.

5b · Playhead snap toggle new

UIStore.timelineSnapedit useScrub / TimelineArea

Separate from paste (paste never snaps). A toggle that only governs playhead scrubbing, mirroring the MIDI editor's snap. "Paste on a bar" then becomes: snap-on → drag the playhead to a bar → paste reads that exact position.

  • Add timelineSnap: boolean (+ setter) to UIStore.
  • A small toggle button in the timeline toolbar/ruler area.
  • In useScrub (or its computeBeat in TimelineArea): if snap on, round the resolved beat to the nearest bar before writing currentBeat / seeking.
Open: snap the playhead to bars or beats? Defaulting to bars (matches block granularity).

6 · Alt-drag to copy tracks

edit App.tsx (DndContext)ProjectStore.duplicateTrackdnd-kit interaction

Track rows are already a dnd-kit sortable list. Goal: the original stays exactly where it is, and a clone is produced at the drop location (store write happens at dragEnd). The cleanest way without rebuilding drag is to change the drop behavior and add copy feedback:

  • Normal drag → reorder (today's handleTrackDragEnd → reorderRootTracks).
  • Alt-drag → keep the original row static (don't apply the sortable shift to it), show a DragOverlay "+copy" ghost following the cursor, and on drop insert a clone at the drop index. Original untouched; a copy appears.
function handleTrackDragEnd(e: DragEndEvent) {
  const alt = (e.activatorEvent as PointerEvent)?.altKey   // modifier at drag start
  const toIndex = rootTrackIds.indexOf(e.over.id as string)
  if (alt) {
    const src = tracks[e.active.id]
    duplicateTrack(src.id, toIndex)          // clone + splice into rootTrackIds at toIndex
  } else { /* existing reorder */ }
}

New store action ProjectStore

duplicateTrack(srcId, atIndex?) {
  const copy = cloneTrack(tracks[srcId])              // fresh track + block + note ids
  tracks[copy.id] = copy
  rootTrackIds.splice(atIndex ?? end, 0, copy.id)
}

Decisions & risks

  • risk Modifier capture. dnd-kit's DragEndEvent doesn't carry live modifiers; activatorEvent.altKey reads the state at drag start. If you want "press Alt mid-drag," track Alt via a window keydown/keyup ref instead.
  • Copy-feel. The DragOverlay "+copy" ghost + static original is what sells it as a duplicate rather than a move - worth doing, not just a drop-time mutation.
  • Nested children deferred. cloneTrack drops childIds for now (root tracks only). Recursion + parentId/targets remap is a marked TODO for when nesting ships.
  • Most-faithful version (later): a ghost that literally peels off the original and follows the cursor means bypassing dnd-kit's sortable - more work. The overlay approach above gets ~90% of the feel for far less.

7 · Delete blocks & tracks via Delete / Backspace

ProjectStore.deleteTrackextend useTrackGestures.ts

New store action

deleteTrack(trackId) must clean up every reference, not just the map entry:

  1. Remove from tracks.
  2. Remove from rootTrackIds (if root) or from its parent's childIds.
  3. Recursively delete childIds - deferred with nesting; root tracks have none today.
  4. Strip this id from any other track's targets[] (automation/routing references).
  5. Clear UIStore.selectedTrackId if it was the deleted one; drop its blocks from selectedBlockIds.

Keyboard wiring (locked precedence)

Inside useTrackGestures, on Delete/Backspace (not in a text field, editingBlock == null):

if (selectedBlockIds.size) deleteSelectedBlocks()    // blocks win - track they sit on stays
else if (selectedTrackId)   deleteTrack(selectedTrackId)  // no blocks selected → delete track + its blocks
Backspace + browser back: call e.preventDefault() so a stray Backspace outside inputs can't navigate away. The MIDI editor's existing note-delete already shows the pattern.

Edge cases

  • Deleting a parent - confirm the children policy (cascade delete vs orphan/reparent).
  • No confirmation dialog (matches the app's current directness); given no undo, consider whether a guard is wanted for tracks with content.
  • After delete, selection is cleared so a second keypress is a no-op rather than deleting a "neighbor."

8 · Cross-cutting concerns & risks

ConcernNotes
No undo (deferred)Paste/duplicate/delete are irreversible until history lands. Not a per-action retrofit: a snapshot-based middleware on the project store (record whole-project state before each mutating action) is written once and covers every action - existing and future - automatically, à la the sibling DAW's historyStore/historyMiddleware. The only per-action nuance is choosing undoable "units" (a drag = one entry). Safe to ship these features first and add history later.
Mac vs Windowse.metaKey || e.ctrlKey for copy/paste; Delete and Backspace both bound.
Focus & contextSingle source of truth = editingBlock. Every handler guards text inputs and checks this flag to avoid double-firing.
System clipboardThis is an in-app clipboard (zustand), not the OS clipboard - so it won't collide with the browser and persists only for the session. Fine for v1; OS-clipboard JSON is a later option.
Selection feedbackAlways re-select the pasted/duplicated result so the next action targets it and the user sees what happened.
SnappingPaste never snaps (lands exactly at the playhead). Snapping is a property of the playhead via the new toggle (§5b) - turn it on to scrub onto bars, then paste there.

9 · Suggested build order

  1. Infra - ClipboardStore (§3.1). Clone helpers land with the actions that use them, not as a separate file.
  2. Delete blocks & tracks (§7) - deleteTrack action + extend useTrackGestures' keydown. Highest value, lowest surface area; establishes the keyboard plumbing.
  3. Copy/paste blocks (§5) - same useTrackGestures handler + clipboard + pasteBlocks.
  4. Playhead snap toggle (§5b) - small, and makes block paste feel right.
  5. Copy/paste notes (§4) - self-contained inside the MIDI editor; reuses existing note selection.
  6. Alt-drag duplicate track (§6) - most interaction risk (dnd-kit); do last (overlay + drop-index, or Alt+click fallback).
  7. (Later today) Undo/history middleware covering all of the above at once.
