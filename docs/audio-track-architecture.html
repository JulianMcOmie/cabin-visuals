Audio Track Architecture

Turning the single global backing clip into a first-class audio track — a block you can drag around the timeline, offset, trim, and stack — while keeping the beat clock the sole source of truth.

Contents

Right now audio is a single global clip pinned to the transport: one AudioClip in AudioStore, one Tone.Player inside the playback engine, hard-wired so file 0:00 == timeline beat 0. It is not a track, it has no position, and it can't be moved. This doc reshapes it into an audio block on an audio track — a clip reference plus a startBar/offset — so the player is scheduled at the block's timeline position, and moving the block reschedules the audio. The clock stays exactly where it is.

The one-line answer: add an 'audio' track type carrying audio blocks (each = a clipRef + startBar + in-clip trimStart/trimEnd), teach the existing core/playback.ts engine to schedule one Tone.Player per block at its beat offset instead of one global player at beat 0, and render a waveform inside the block from the decoded buffer. The beat clock (currentBeat in TimeStore) never changes — audio is scheduled relative to it.

★ The big picture — internalize this first

Before the section detail, here is the whole change in one view, and the four mental models everything hangs off. Hold these and the rest is elaboration.

Four things to hold in your head

  1. The clip is the bytes; the block is the placement. An AudioClip (already exists — ref · fileName · duration) is the decoded material. An audio block is a reference to a clip plus where it sits (startBar) and how much of it plays (trimStart/trimEnd). Many blocks can point at one clip.
  2. The beat clock is the sole timebase — audio is scheduled relative to it. Today's hard-wired "file 0:00 == beat 0" becomes "block plays when currentBeat enters its window, at an in-clip offset computed from beat − blockStartBeat." Moving the block just changes that window → reschedules the audio. The clock code doesn't move.
  3. One Tone.Player per audio block, armed on a beat window. The engine keeps a small pool of players keyed by block. On play()/seek() it (re)computes, for each block, whether it's live at the current beat and where inside the clip to start — the same start(when, offset) the code already uses, generalized from one player to N.
  4. Seeking is already sample-accurate — extend it per-block. The seek() path already re-arms the audio at beat * 60 / bpm. Generalize it: for each block, restart it at its own offset if the new beat lands inside its window, stop it if it doesn't. Scrub-while-paused is a design choice (§7).

Audio is deliberately minimal and global. Three files own it, and there is no notion of a track, a position, or an offset.

2 · The goal & the one hard problem

The user wants audio to behave like a block: drag it along the timeline, offset its start, trim its ends, and drop more than one — exactly like the MIDI blocks that already exist (Block with startBar/durationBars, positioned at startBar * barWidthPx). Everything in this doc reduces to one hard problem:

The hard problem in one sentence: audio is scheduled in seconds on an audio clock, but the timeline is measured in beats — so placing a clip at bar 12 means "when currentBeat reaches bar 12's beat, start this player at second 0 of the clip; while the clip plays, its in-clip second = (beat − blockStartBeat) × 60/bpm." Get that beat⟷second conversion right, per block, at play/seek/tempo-change, and the feature is done. Everything else — the data model, the waveform, the UI — is scaffolding around that conversion.

3 · Data model — audio track & audio block

Two design choices, and a recommendation. The audio material (the AudioClip) already exists and is the right shape — it just needs to become a library (many clips) instead of a single slot. The new thing is a placement: a block that references a clip and knows where it sits.

3.1 — Which container: a new track type vs. an audio-typed block

OptionShapeVerdict
A · Audio track type Add 'audio' to TrackType; the track carries audioBlocks: AudioBlock[] (or reuses blocks with an audio payload). Mirrors the MIDI-track model exactly. Recommended. A track already owns a lane, a color, a name, mute/solo, and a row in the timeline — everything an audio lane needs. It slots into the existing track tree, gesture system, and renderer with the least new surface.
B · Audio block on any track A Block variant (kind: 'audio') that can live on an existing track alongside MIDI blocks. Rejected as primary. Mixing audio + MIDI in one lane muddies resolve (audio isn't a note stream) and the piano-roll gesture math. Keep audio in its own track kind; the block itself is where the audio payload lives.

3.2 — The types

// store/AudioStore.ts — the clip stays as-is, but the store holds a LIBRARY
interface AudioClip {          // unchanged — already serializable
  ref: string;                  // opaque handle → audioSource.ts bytes
  fileName: string;
  duration: number;             // seconds (full file)
}
interface AudioState {
  clips: Record<string, AudioClip>;   // was: clip | null — now a library keyed by ref
  addClip; removeClip;
}

// types.ts — the placement lives in the project document (undoable)
interface AudioBlock {
  id: string;
  clipRef: string;              // → AudioClip.ref (the material)
  startBar: number;             // timeline position — the whole point (like Block.startBar)
  trimStart: number;            // seconds into the clip where playback begins (default 0)
  trimEnd: number;              // seconds into the clip where it ends (default = clip.duration)
  gain?: number;                // per-block volume (dB or linear)
  fadeIn?: number; fadeOut?: number; // seconds — optional edge fades
}

type TrackType = 'base' | 'add' | … | 'audio';   // + one member

interface Track {                // an audio track reuses the existing Track shell
  … name; color; muted; solo; parentId; childIds;
  audioBlocks?: AudioBlock[];   // present only when type === 'audio'
}
Why startBar not startBeat? To match the MIDI Block exactly (startBar/durationBars, drawn at startBar * barWidthPx). The beat position and the duration in beats are derived at schedule time (startBar × beatsPerBar, and the trimmed clip length converted through the current bpm). Storing beats would duplicate what bpm already implies; storing bars keeps the block's on-screen width honest as the ruler zooms.
Audio duration is fixed seconds; the timeline is beats — they drift with tempo. A 4-second clip is 4 seconds no matter the bpm, but its width in bars changes when you change tempo (8 beats at 120bpm, 4 beats at 60bpm). Decide the contract early: (a) the block anchors at startBar and its visible width is recomputed from (trimEnd−trimStart) seconds ÷ current beat-length (audio never time-stretches; the block just gets wider/narrower on the ruler), or (b) time-stretch the audio to fit a bar count (out of scope — needs a rubberband-style DSP). Recommendation: (a) — no stretching, width follows tempo. This is the audio analogue of §5's beat⟷second bridge and the single most important semantic call in the doc.

4 · Placement on the timeline

An audio block is drawn and dragged exactly like a MIDI block, because the timeline already speaks bars. The MIDI Block component positions itself at left = startBar * barWidthPx, width = durationBars * barWidthPx (timeline/Block.tsx); an AudioBlock does the same, but its width comes from the trimmed clip length rather than a stored durationBars.

  • Reuse the block gesture system. Move/drag, marquee-select, copy/paste, alt-duplicate already operate on Blocks. An AudioBlock needs the same move gesture (writing startBar) and edge-drag handles for trim (§9). Because it lives on its own track type, its gestures can branch cleanly without touching MIDI-block math.
  • Width follows content. Unlike a MIDI block (author-set durationBars), an audio block's width is derived from (trimEnd − trimStart) seconds at the current tempo — you don't type a bar count, you drag the trim edges (§9).
  • Snapping. Same bar/beat snap the ruler already enforces. Free-positioning (no snap, with a modifier) is a natural later add since audio start rarely lands on an exact beat.

5 · How it plugs into resolve + the clock

There are two engines in this codebase and the audio work touches only one of them. The visual engine (core/engine/, resolve → compute → render) is a read-only observer of the project and cares nothing about audio. The playback/transport engine (core/playback.ts) owns the Tone transport, the RAF beat clock, and — today — the single audio player. Audio-as-a-track lives entirely in the second engine.

Does audio need to "resolve"? No heavy resolve like the visual graph. On play()/seek() the playback engine gathers the audio blocks from the project store and computes, for each, its beat window (startBeat…endBeat). That's a cheap O(blocks) pass done at transport events, not per frame. There's no per-frame audio work at all in the steady state — once players are armed with start(when, offset), Tone's clock carries them; the RAF only reads the transport position to advance currentBeat and to detect when a block window closes.

6 · Scheduling audio at a timeline offset

This is the mechanical core: generalize today's single-player start(when, startBeat*60/bpm) into one player per block, each started at its in-clip offset. The existing engine already does the hard audio-clock alignment (a shared when = Tone.now() + AUDIO_LOOKAHEAD so transport and audio are sample-aligned). We keep that and loop it over blocks.

// core/playback.ts — generalized play(), sketched
async play(startBeat: number) {
  const bpm = this.callbacks.getBpm();
  const beatsPerBar = this.callbacks.getBeatsPerBar();
  const transport = Tone.getTransport();
  transport.stop();
  transport.bpm.value = bpm;
  transport.position = beatToPosition(startBeat, beatsPerBar);

  const when = Tone.now() + AUDIO_LOOKAHEAD;   // shared audio-clock anchor (unchanged)
  transport.start(when);

  for (const b of this.audioBlocks) {              // was: one global player
    const startBeat_b = b.startBar * beatsPerBar;
    const clipLenBeats = (b.trimEnd - b.trimStart) * bpm / 60;
    const endBeat_b = startBeat_b + clipLenBeats;
    const player = this.playerFor(b);           // pooled Tone.Player, buffer already loaded
    player.stop(when);
    if (startBeat >= endBeat_b) continue;         // past — leave idle
    if (startBeat <= startBeat_b) {                // future — schedule ahead
      const delaySec = (startBeat_b - startBeat) * 60 / bpm;
      player.start(when + delaySec, b.trimStart);
    } else {                                    // mid-clip — join with an offset
      const offset = b.trimStart + (startBeat - startBeat_b) * 60 / bpm;
      player.start(when, offset);
    }
  }
  this.startBeatTracking();
}
The whole feature is this loop. Today's engine schedules one player with start(when, startBeat*60/bpm) — which is exactly the mid-clip branch for a block anchored at beat 0. Adding startBar just shifts the anchor and splits into three cases. The AUDIO_LOOKAHEAD alignment, the beatToPosition conversion, and the RAF tracker all carry over unchanged.
Player pooling & buffer loading. loadAudio(ref) today loads one buffer and no-ops if the ref is unchanged. Generalize to loadClips(refs): dispose players for refs no longer referenced, create/keep a Tone.Player per block (two blocks of the same clip can overlap, so key players by block id, sharing a decoded ToneAudioBuffer per ref). usePlayback.play() awaits this before starting, exactly as it awaits loadAudio now.

7 · Seeking & scrubbing with audio

The engine already has a correct seek(beat): it sets transport.position and, if playing, re-arms the single player at beat*60/bpm. Generalize it to the three-case loop from §6 — for each block, restart it at its own offset if the new beat lands in its window, stop it if it doesn't.

  • Seek while playing — re-window every block (the figure). Reuse a shared armBlock(block, atBeat, when) helper so play and seek can't diverge.
  • Scrub while paused — a design choice. Today, dragging the playhead while paused updates currentBeat (visuals follow) but plays no audio. Two options: (a) keep it silent (simplest, matches today), or (b) "audio scrubbing" — play a short grain of the clip at the scrub position (a DAW nicety; more work, needs a throttled short-lived player). Recommendation: ship (a); treat (b) as a later polish.
  • Tempo change while playing — setBpm currently just sets transport.bpm. With positioned audio, a live bpm change moves every block's beat window (seconds are fixed, beats aren't — §3.2's warning). Simplest correct behavior: on a live bpm change, re-window all blocks (a seek(currentBeat)-style re-arm) so they realign. Audio doesn't stretch; it just re-anchors.
  • End-of-project — the RAF tracker already stops the transport + player at maxBeat. Generalize "stop the player" to "stop all block players."

8 · Waveform rendering pipeline

A waveform inside the block turns the abstract clip into something you can align by eye. The pipeline is: decode the clip once → compute min/max peaks per pixel column → draw to a canvas sized to the block → redraw on resize/zoom/trim.

  • Decode once, off the buffer you already have. AudioBar already decodes the file to read duration (ctx.decodeAudioData). Keep that buffer (cache it per ref) so both the waveform and the Tone.Player reuse it — no second decode.
  • Peaks are size-independent. Compute a fixed-resolution min/max peak array per clip (e.g. a few thousand buckets) once; the canvas draw samples from that array to the current pixel width. Zoom/resize is a redraw, not a re-analysis.
  • Draw on a <canvas> inside the block. Cheap, and it composites under the block's chrome. Trimming renders only the [trimStart, trimEnd] slice (or dims the rest, as drawn).
  • Where the code goes. A small core/waveform.ts (decode-cache + peaks) feeding a <AudioBlockView> component — the audio analogue of the MIDI Block.tsx.

9 · Multiple clips & trimming

Once a block is a positioned reference, "multiple clips" and "trim" fall out almost for free — they're just more blocks and two more numbers.

9.1 — Multiple clips / blocks

  • Many blocks per track, many tracks. An audio track holds an AudioBlock[]; you can have several audio tracks. The §6 scheduling loop already iterates all blocks, so N clips is the same code path as one.
  • Overlap is fine. Two blocks whose windows overlap = two players sounding at once (Tone mixes them at the destination). This is why players are keyed by block, not by ref.
  • The clip library. AudioStore.clip becomes clips: Record<ref, AudioClip> — a small library the block picker draws from. AudioBar's load path adds to the library instead of replacing the single slot; the existing library-drag-to-add-track gesture (recent commit) is the natural way to drop a clip onto the timeline.

9.2 — Trimming

  • Trim end — drag the right edge → decrease trimEnd. Block gets shorter; start unchanged.
  • Trim start — drag the left edge → increase trimStart and increase startBar by the same beat amount, so the surviving audio stays aligned to its original beats (the classic DAW "trim from the left" behavior).
  • Scheduling honors trim automatically — the §6 math already uses trimStart as the base offset and trimEnd to clamp duration. No new scheduling code; trim just feeds numbers into the existing loop.
  • Split (later) — cutting a block at the playhead = two blocks sharing the ref, with complementary trimStart/trimEnd/startBar. Free once trim exists.

10 · Tradeoffs & open questions

  • semantics Seconds vs. beats under tempo change. The single most important call (§3.2). Audio is fixed seconds; blocks are placed in bars. Decide "no time-stretch, width follows tempo, re-window on bpm change" up front — it's cheap and correct, and it keeps the beat clock the sole timebase.
  • risk Player pool lifecycle. One Tone.Player per block (sharing decoded buffers per ref). Create/dispose as blocks are added/removed; don't leak players on delete. The current single-player dispose() generalizes to a keyed map.
  • risk Scheduling drift on many blocks. All players share one when anchor, so they stay aligned; but scheduling far-future blocks with Tone's own timeline (vs. re-arming on each seek) is cleaner than polling. Prefer start(when + delay, offset) and let Tone hold them.
  • open Audio scrubbing while paused. Ship silent-scrub first (matches today). Grain-scrub is a real feature with real cost — defer.
  • open Does audio belong in the undoable document? The placement (audio blocks, startBar, trim) clearly should — it's an edit, like moving a MIDI block, so it lives in ProjectStore. The clip library (bytes/refs) is more like AudioStore's current ephemeral role; decide whether adding a clip is undoable.
  • open Loading latency at play(). usePlayback.play() awaits buffer load before starting. With many clips, pre-decode on insert (when the block is dropped) rather than at play, so the downbeat isn't gated on decoding.
  • later Audio-reactive visuals. Out of scope here, but positioned audio is the substrate for it: once a clip sits at a known beat window, its analyzed amplitude/FFT could feed a modulator port in the visual engine. Named, not designed — keep the audio track independent of the visual engine for now.

Recommendation in one breath:

  • Add an 'audio' track type carrying audio blocks — each a clipRef + startBar + trimStart/trimEnd. The clip (already the right shape) becomes a small library.
  • Generalize core/playback.ts from one global player to one player per block, armed on its beat window via the same start(when, offset) alignment the code already uses — three cases: mid-clip, future, past.
  • Keep the beat clock the sole timebase; audio is scheduled relative to currentBeat, never the reverse. Moving a block = changing startBar = rescheduling that player.
  • Draw a waveform in the block from a decode-once, size-independent peak cache; trim shows a slice of it.
  • Prove it on the smallest loop first: one audio block you can drag → it plays at its position. Multiple clips and trimming are additive once that lands.
