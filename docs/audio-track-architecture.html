Audio Track Architecture

Turning the single global backing clip into a first-class audio track — blocks you can drag, offset, trim, and stack — scheduled by a dedicated audio engine alongside the transport and visual engines, with the beat clock as the sole timebase.

Right now audio is a single global clip pinned to the transport: one AudioClip in AudioStore, one Tone.Player inside the playback engine, hard-wired so file 0:00 == timeline beat 0. This doc reshapes it into audio blocks on audio tracks — a clip reference plus a startBar/trim — scheduled by a new, dedicated audio engine that mirrors how the visual engine already sits beside the transport. The beat clock stays exactly where it is.

The one-line answer: add an 'audio' track type carrying audio blocks (each = a clipRef + startBar + in-clip trimStart/trimEnd), stand up core/audio/ — an audio engine owning a per-block Tone.Player pool, the decode/peak cache, and the one beat⟷second helper — and have the transport engine hand it (beat, when) at every transport event. Audio is never resampled: a clip is fixed seconds, its width in bars follows the tempo, and the beat clock (currentBeat in TimeStore) never changes.

★ The big picture — internalize this first

Before the section-by-section detail, the whole change in one view, and the five mental models everything else hangs off.

Five things to hold in your head

  1. The clip is the bytes; the block is the placement. An AudioClip (already exists — ref · fileName · duration) is the material. An audio block is a reference to a clip plus where it sits (startBar) and how much of it plays (trimStart/trimEnd). Many blocks can point at one clip.
  2. The beat clock is the sole timebase — audio is scheduled relative to it. "File 0:00 == beat 0" becomes "block plays when currentBeat enters its window, at an in-clip offset computed from beat − blockStartBeat." Moving the block changes the window → reschedules the audio. The clock code doesn't move.
  3. Audio is never resampled. A clip is fixed seconds no matter the bpm. Change the tempo and the block just occupies more or fewer beats — its on-screen width recomputes, its sound doesn't. (Decided — §7.)
  4. The audio engine is a sibling of the visual engine, not a clone of it. Same position in the architecture (a project-reading engine beside the transport), different rhythm: visuals recompute per frame; audio arms players at transport events and lets Tone's clock carry them. Don't force per-frame symmetry — audio must be scheduled ahead on the audio clock.
  5. One Tone.Player per audio block, armed on a beat window. On play/seek/bpm-change the engine computes, per block, whether it's live at the current beat and where inside the clip to start — the same start(when, offset) the code already uses, generalized from one player to N sharing one decoded buffer per ref.

1 · How audio works today

Audio is deliberately minimal and global. Since the persistence work shipped, the bytes already live in Supabase Storage — that seam is done and untouched by this work.

2 · The goal & the one hard problem

The user wants audio to behave like a block: drag it along the timeline, offset its start, trim its ends, drop more than one — exactly like the MIDI blocks that already exist. Everything reduces to one hard problem:

The hard problem in one sentence: audio is scheduled in seconds on an audio clock, but the timeline is measured in beats — so placing a clip at bar 12 means "when currentBeat reaches bar 12's beat, start this player at trimStart seconds into the clip; while it plays, its in-clip second = trimStart + (beat − blockStartBeat) × 60/bpm." Get that beat⟷second conversion right, per block, at play/seek/tempo-change — in one shared helper — and the feature is done. Everything else is scaffolding around that conversion.

3 · Data model — clips, tracks, blocks

Two kinds of state: the material (clips — bytes in the bucket, descriptor in AudioStore) and the placement (audio blocks — position + trim, in the project document, undoable). The beat window each block occupies is derived, never stored — recomputed from startBar + trim + tempo, exactly like the visual engine's resolved graph.

3.2 — The types

// store/AudioStore.ts — the clip stays as-is; the store becomes a LIBRARY
interface AudioClip {              // unchanged — already serializable
  ref: string;                    // opaque handle → bucket bytes (audioSource/audioStorage)
  fileName: string;
  duration: number;               // seconds (full file)
}
interface AudioState {
  clips: Record<string, AudioClip>;  // was: clip | null
  addClip; removeClip;
}

// editor/types.ts — the placement lives in the project document (undoable)
interface AudioBlock {
  id: string;
  clipRef: string;                // → AudioClip.ref (the material)
  startBar: number;               // timeline position (mirrors Block.startBar)
  trimStart: number;              // seconds into the clip where playback begins (default 0)
  trimEnd: number;                // seconds where it ends (default = clip.duration)
  gain?: number;                  // per-block volume
  fadeIn?: number; fadeOut?: number; // seconds — later phase
}
type TrackType = 'base' | 'add' | … | 'audio';  // + one member
interface Track {                  // existing shell, one optional field
  … name; color; muted; solo; parentId; childIds; blocks;
  audioBlocks?: AudioBlock[];      // present only when type === 'audio'
}

Why startBar, not startBeat? To match the MIDI Block exactly (startBar/durationBars, drawn at startBar * barWidthPx). The beat position and beat-duration are derived at schedule time. Storing bars keeps the block's on-screen behavior consistent with everything else on the ruler.

Contrast with Excellent DAW (deliberate): E-DAW stores decoded audioData on the block itself and gives blocks an author-set durationBars with loop-wrapping inside. Cabin's model — block = ref + position + trim, width derived — keeps bytes out of the document (they live in the bucket; the doc carries refs) and keeps "how long is this clip" a fact about the audio, not an author decision. See §10.

4 · Placement on the timeline

  • Drawn and dragged like a MIDI block. Position at left = startBar * barWidthPx; dragging writes one number, startBar; the engine reacts by rescheduling that block. Bar/beat snapping comes free from the ruler.
  • Width follows content — and tempo, reactively. An audio block's width is (trimEnd − trimStart) × bpm/60 ÷ beatsPerBar bars, recomputed live: change the bpm and every audio block visibly widens or narrows on the spot. You never type a bar count — you drag trim edges (§9).
  • Free-positioning (no snap, with a modifier) is a natural later add, since real audio rarely starts on an exact beat.

5 · Three engines, one clock

The original draft put the player pool inside core/playback.ts. Decided instead: a dedicated audio engine, so the codebase has three engines with one clock — the same separation the visual engine already proves out.

EngineOwnsConsumes the beat…
core/playback.ts — transportTone transport, RAF beat clock, play/pause/seek/setBpm. Sheds its Tone.Player. Sole producer of the beat and of the shared audio-clock when anchor.—
core/audio/ — audio ★ newPlayer pool (keyed by block id), decode/buffer cache (shared with waveforms), the armBlock beat⟷second helper, gain/fades later.Discretely: armAll(beat, when) at transport events; Tone's clock carries the players in between. No per-frame work.
core/engine/ — visualResolve → compute → render. Untouched.Continuously: computeAtBeat(beat) every frame, a pure function.
The guardrail — don't chase full symmetry. The audio engine has no clock of its own and an event-shaped API (setBlocks · loadClips · armAll(beat, when) · stopAll), not a computeAtBeat-shaped one. It never touches the transport; the when anchor is computed in exactly one place (the transport engine) and passed in — that single anchor is what keeps N players and the transport sample-aligned. If the audio engine ever grows its own Tone.now() calls for scheduling, the alignment story is broken.

Does audio need to "resolve"? No heavy resolve like the visual graph. At transport events the audio engine gathers the audio blocks from the project store and computes each block's beat window — a cheap O(blocks) pass. In steady state there is no per-frame audio work at all; the RAF only advances currentBeat.

6 · Scheduling audio at a timeline offset

The mechanical core: generalize today's single-player start(when, startBeat*60/bpm) into one player per block, each started at its in-clip offset. The existing hard part — the shared when = Tone.now() + LOOKAHEAD alignment — carries over; it just moves behind the audio engine's door and loops over blocks.

// core/audio/AudioEngine.ts — the one place beat⟷second math lives
armBlock(b: AudioBlock, atBeat: number, when: number, bpm: number, beatsPerBar: number) {
  const startBeat = b.startBar * beatsPerBar;
  const endBeat   = startBeat + (b.trimEnd - b.trimStart) * bpm / 60;
  const player = this.players.get(b.id);
  if (!player) return;
  player.stop(when);
  if (atBeat >= endBeat) return;                       // past — idle
  if (atBeat <= startBeat)                              // future — schedule ahead
    player.start(when + (startBeat - atBeat) * 60 / bpm, b.trimStart);
  else                                                  // mid-clip — join with an offset
    player.start(when, b.trimStart + (atBeat - startBeat) * 60 / bpm);
}
  • Mute/solo at arm time (decided — taken from E-DAW, §10). Before arming, skip blocks on muted (or soloed-out) audio tracks — the same semantics the visual engine applies to hide objects. Toggling mute/solo while playing triggers a re-arm pass. Retrofitting this later is annoying; it ships with the engine.
  • Player pool lifecycle. loadClips() reconciles a Map<blockId, Tone.Player> against the current block set — dispose orphans, create newcomers — with one decoded ToneAudioBuffer shared per ref (two blocks of one clip can overlap, so players are keyed by block, buffers by ref).
  • Headroom (taken from E-DAW). Overlapping players sum at the destination and can clip; give the engine a master gain (or a modest per-player default) rather than routing raw to destination.
  • Lookahead: 50ms (decided). Today's 20ms is tight for arming N players in a loop; E-DAW runs 50ms happily. Cheap insurance against downbeat jitter.

7 · Seeking, scrubbing & tempo — the decided semantics

  • decided Seek while playing = re-arm every block. A seek is "re-run the §6 windowing at the new beat" — each block independently stops, restarts mid-clip, or is scheduled ahead. play and seek share armBlock so they cannot diverge.
  • decided Scrub while paused is silent. Matches today's behavior and composes with the pause invariant (paused = static and silent). Grain-scrub — a throttled short-lived player at the scrub position — is a real feature with real cost; explicitly deferred, not designed here.
  • decided Tempo never changes the sound — audio is never resampled. A clip is fixed seconds; a bpm change re-derives every block's beat window and on-screen width ("more or fewer beats"), never its playback rate. Three implications:
    1. Block width reads bpm reactively — change tempo, blocks visibly resize at once.
    2. setBpm in the transport engine (today: just sets transport.bpm) must also trigger an audio re-arm pass while playing, since every window just moved.
    3. Known, accepted consequence: changing bpm while playing over a block makes the audio audibly jump position — the same beat now maps to a different in-clip second, and the beat clock wins. Changing bpm while paused (the normal case) is artifact-free.
  • End-of-project — the RAF tracker already stops the transport at maxBeat; "stop the player" generalizes to audioEngine.stopAll().

8 · Waveform rendering pipeline

A waveform inside the block turns the abstract clip into something you can align by eye. Decode once → peak arrays → canvas draw sized to the block → redraw on resize/zoom/trim.

  • Decode once, in the audio engine's cache. The same decoded buffer feeds the Tone.Player pool and the peak extraction — one decode per ref, ever. Pre-decode when a block is inserted (not at play()), so the downbeat is never gated on decoding.
  • Adaptive peak resolution (taken from E-DAW). Extract a base min/max peak array (~1000 buckets) per ref; when a block is drawn wider than the base resolution serves (zoomed in), re-extract at max(1000, widthPx / 2) buckets. Zoom/resize is a redraw; deep zoom is one cheap re-extraction — never a re-decode.
  • Draw on a <canvas> inside the block, compositing under the block's chrome. Trimming renders only the [trimStart, trimEnd] slice of the same peak data.
  • Where the code goes: core/audio/waveform.ts (decode-cache + peaks, shared with the player pool) feeding timeline/AudioBlockView.tsx — the audio analogue of the MIDI Block.tsx.

9 · Multiple clips & trimming

  • Many blocks per track, many tracks. The §6 loop already iterates all blocks; N clips is the same code path as one. Overlap is fine — two live windows = two players, mixed at the destination (hence the master-gain headroom).
  • The clip library. AudioStore.clips is the library the block picker draws from; AudioBar's load path adds to it instead of replacing a slot. The existing library-drag-to-add-track gesture is the natural way to drop a clip onto the timeline.
  • Trim end — drag the right edge → decrease trimEnd. Shorter block, start unchanged.
  • Trim start — drag the left edge → increase trimStart and startBar by the same beat amount, so the surviving audio stays aligned to its original beats (classic DAW left-trim).
  • Scheduling honors trim automatically — armBlock already uses trimStart as the base offset and trimEnd to clamp the window. Trim feeds numbers into existing code.
  • Split (later) — cutting at the playhead = two blocks sharing the ref with complementary trim/startBar. Free once trim exists.

10 · Lessons taken from Excellent DAW

E-DAW (the codebase Cabin's instruments are ported from) has a working positioned-audio implementation inside its 655-line core/playback.ts. What we take, and what we deliberately don't:

MechanismWhat E-DAW doesVerdict for Cabin
Mute/solo at arm timeChecks track mute + sibling-solo before scheduling and on every seek-restart.Take. Ships with the audio engine (§6) — cheap now, painful to retrofit.
50ms lookaheadAUDIO_LOOKAHEAD_SECONDS = 0.05 configured at context init.Take. Cabin's 20ms is tight for arming N players.
Adaptive waveform peaksBase 1000-bucket peaks; re-extracts at max(1000, widthPx/2) when drawn wider.Take. Same cache, one extra parameter (§8).
Player headroomClip players default to −6dB so overlaps don't clip.Take the idea — a master gain node in the audio engine.
Loop-boundary retriggerWhen a loop region starts mid-audio-block, a special retrigger Part restarts the player each loop pass.File away. Cabin has no loop regions yet; when it does, this is the known trap + fix.
Decoded audio on the block; author-set durationBars + loop-wrapBlocks carry audioData and a fixed bar width; audio loops/clips inside it.Don't take. Bytes belong in the bucket, not the document; width belongs to the clip + tempo, not the author (§3, §7).
Playback-speed scalingbpm × speed and player.playbackRate = speed (pitch shifts).Don't take. Violates "audio never sounds different"; out of scope.

11 · Persistence — the document goes to v2

The original draft predates the persistence layer. Audio tracks are its first real schema evolution — and the upgrade chain built for exactly this moment does the work:

  • What changes in the document: audioClip (single descriptor) becomes clips (the library, keyed by ref). audioBlocks on tracks ride into the blob for free — persistence/serialize.ts picks every non-function field of ProjectStore generically, and blocks live on Track.
  • The v1 → v2 upgrade step (persistence/upgrade.ts, first real entry in the chain): move audioClip into the clips library and synthesize an audio track with one block at bar 0 (trim = full clip). That reproduces the old "global clip pinned to beat 0" behavior exactly — old saves keep sounding identical. Bump CURRENT_VERSION to 2.
  • Bytes are already handled: saveAudio uploads to the project-audio bucket on add; clip refs are bucket paths; nothing new to build there.
  • Undo: placement (blocks, startBar, trim) lives in ProjectStore → undoable automatically via the generic snapshot boundary. The clip library lives in AudioStore → not undoable (loading a file isn't an edit), but it is persisted (serialize reads it explicitly, as it does the single clip today).

12 · Decisions & open questions

  • decided Separate audio engine (core/audio/): event-shaped API, no clock of its own, when passed in from the transport. (§5)
  • decided No time-stretch, ever. Width follows tempo reactively; bpm change re-arms; mid-block live bpm change audibly jumps (accepted). (§7)
  • decided Silent scrub while paused. Grain-scrub deferred. (§7)
  • decided Mute/solo honored at arm time, from day one. (§6)
  • decided Document v2 with a v1→v2 upgrade that synthesizes a bar-0 block from the old global clip. (§11)
  • risk Player pool leaks. Reconcile the keyed map against the block set on every loadClips; never leak a player on block delete.
  • risk Downbeat decode latency. Pre-decode on block insert via the shared cache, not at play().
  • open Parallel audioBlocks vs the shared block gesture plumbing. Start parallel (isolation); revisit after trim lands — the abilities story suggests unified plumbing may win long-term.
  • later Audio-reactive visuals. Positioned audio is the substrate: a clip at a known beat window could feed analyzed amplitude/FFT into a visual-engine modulator port. Out of scope; must eventually respect the pause invariant (analysis as a pure function of beat), which is a design problem of its own.
Recommendation in one breath: add an 'audio' track type carrying audio blocks (clipRef + startBar + trim); stand up core/audio/ — player pool, decode/peak cache, and the one armBlock beat⟷second helper — fed (beat, when) by the transport engine at play/seek/bpm events; never resample audio (width follows tempo); keep scrub-while-paused silent; take E-DAW's mute/solo-at-arm, 50ms lookahead, adaptive peaks, and headroom; and ship the document's first upgrade step so yesterday's global clip becomes a bar-0 block that sounds identical. Prove it on the smallest loop first: one audio block you can drag → it plays at its position.
