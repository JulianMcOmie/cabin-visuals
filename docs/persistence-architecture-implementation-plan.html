This plan introduces two new directories — src/persistence/ (the document repo + autosave) and db/ (Drizzle schema + migrations) — plus a small edit to editor/App.tsx and one new column-and-policies migration. A · Architectural components — what gets added, and one layer down

The highest level first, then a peek at the key function / effect inside each. Files in monospace; status badges: new edit delete.

1 · The project repo new persistence/projectRepo.ts

Adds the one door for document CRUD. It's the only file that names the projects table for the document. One layer down:

  • a list() that selects the spine columns only (id, name, updated_at) — never pulls a blob;
  • a load(id) that selects data, runs it through upgradeDocument() (§8 of the architecture doc), and returns a current-shape ProjectDocument;
  • a save(id, doc) that UPSERTs the blob plus the projected columns (name, schema_version, updated_at) in one write — RLS guarantees it's the caller's row;
  • create(name) / remove(id) that reuse the flow already in app/projects/page.tsx (moved here so both the browser and the editor share it).

2 · The autosave loop new persistence/autosave.ts

Adds a debounced subscription that mirrors the store to Supabase. Deeper:

  • startAutosave(projectId) — registers useProjectStore.subscribe, diffs by reference (immutable store, so state !== prev is exact), and schedules a debounced flush (~1s idle);
  • flush() — non-overlapping (one write in flight), failure-safe (a network error leaves the doc dirty and retries, never touches memory), and status-emitting (saving / saved / error) for a UI indicator;
  • flush-on-exit — a beforeunload/visibilitychange handler forces a final save so the debounce window can't eat the last edit.

Deliberately the same mechanism HistoryStore already uses (a store subscription with a burst window) — aimed at Supabase instead of an undo stack.

3 · Serialize + upgrade new persistence/serialize.ts · persistence/upgrade.ts

  • serialize(state) → ProjectDocument — picks the non-function, non-ephemeral fields of ProjectStore (the exact set HistoryStore snapshots) and stamps schemaVersion; hydrate(doc) is the inverse (setState);
  • upgradeDocument(raw) — walks the append-only vN → vN+1 chain until the blob matches CURRENT_VERSION, so the editor only ever sees the latest shape.

4 · The Supabase client re-export new persistence/supabase.ts

A one-line re-export of the existing browser client (utils/supabase/client.ts), so the persistence module has a single import point and nothing here re-implements auth. The session cookie rides along automatically, so RLS sees auth.uid().

5 · Auth & list hooks new persistence/hooks/useAuth.ts · useProjectList.ts

  • useAuth() — wraps supabase.auth.getUser() + onAuthStateChange; returns { user, loading }. Guards "save requires a session" and gives the owner id;
  • useProjectList() — projectRepo.list() as a hook with create/delete actions, so app/projects/page.tsx stops calling Supabase inline and goes through the repo.

6 · The schema, as code new db/schema.ts · db/migrations/* · drizzle.config.ts

  • schema.ts — the projects table in Drizzle, including the new data JSONB, schema_version, and updated_at columns; typed with .$type<ProjectDocument>();
  • migrations/0001_add_project_document.sql — the generated migration that adds those columns, enables RLS, and creates the four owner policies (§6);
  • drizzle.config.ts — points drizzle-kit at the Supabase connection string.

7 · Editor wiring edit editor/App.tsx · new editor/hooks/useProjectPersistence.ts

  • useProjectPersistence.ts — reads ?project=<id>, calls projectRepo.load(id), hydrates the store, then arms startAutosave(id) (in that order, so hydrate doesn't trigger a redundant save);
  • App.tsx — mounts the hook and a small save-status indicator in the header; otherwise untouched.

8 · Project browser goes through the repo edit app/projects/page.tsx

  • Swaps its inline supabase.from('projects')… calls for projectRepo.list/create/remove — same behavior, single source of truth. Create now inserts a default ProjectDocument at CURRENT_VERSION so the editor can load it like any other.

B · Scope & the phase ladder

This plan is incremental and always-shippable: each phase leaves a working app. The order mirrors the architecture doc — but here every phase is pinned to files. Phase 1 is the schema, phase 2 the repo, phase 3 the autosave/load wiring (which proves the whole thesis: reload and your project is still there); 4–5 are additive polish and safety.

In scope here: the schema migration, the persistence module, autosave + load wiring, routing the browser through the repo, doc-versioning, and conflict safety — with file lists for all phases. Out of scope (named, not detailed): audio-byte storage (phase 6, deferred — the clip metadata already rides in the blob), real-time multi-user collaboration (needs CRDT/OT — a product decision, not a persistence one), and any change to the auth flows, which already ship.

C · Data model — the types and where they live

Two kinds of types: the document (what the user edits, in ProjectStore, serialized into the blob) and the row (the relational spine the database queries). Keeping them separate is what lets the blob change freely while the columns stay stable.

C.1 — Document type new persistence/types.ts

// persistence/types.ts — the serialized project (= projects.data)
interface ProjectDocument {
  schemaVersion: number;               // stamped by serialize(); drives upgrade() on load
  bpm: number; beatsPerBar: number; totalBars: number;   // (from ProjectStore)
  tracks: Record<string, Track>;          // Track already carries blocks·notes·params·tags·targets·lanes·effects
  rootTrackIds: string[];
  audioClip?: AudioClip;                 // {ref, fileName, duration} — metadata only (§phase 6)
}

Track, Block, Note, AudioClip are reused verbatim from editor/types.ts / editor/store/AudioStore.ts — the document type is a thin envelope over the model that already exists. Nothing about the editor's model changes; persistence just adds the envelope and the schemaVersion.

C.2 — Row type new db/schema.ts edit the projects table

// db/schema.ts — the spine (Drizzle). The columns Postgres/RLS/the browser touch.
export const projects = pgTable('projects', {
  id:            uuid('id').primaryKey().defaultRandom(),   // (exists)
  owner:         uuid('owner').notNull(),                    // today: user_id — the RLS anchor
  name:          text('name').notNull().default('Untitled Project'), // (exists)
  schemaVersion: integer('schema_version').notNull().default(1), // NEW
  data:          jsonb('data').notNull().$type<ProjectDocument>(),  // NEW — the blob
  createdAt:     timestamp('created_at').notNull().defaultNow(),  // (exists)
  updatedAt:     timestamp('updated_at').notNull().defaultNow(),  // NEW — bumped every save
});
Mental model: persistence/types.ts = "what the user owns" (the blob's shape, versioned by schemaVersion); db/schema.ts = "what the database queries" (the columns, versioned by Drizzle migrations). The dependency points one way — db/schema.ts imports ProjectDocument to type the data column, but the document type knows nothing about Postgres. That's what keeps the editor free of any storage concept.
Naming reality: today's column is user_id, not owner; the architecture doc uses owner generically. Either rename it in the migration (cleaner) or keep user_id and alias it in the schema (zero-risk). Pick one and be consistent in the RLS policies — the policy compares whatever the column is against auth.uid().

D · Infrastructure: the persistence module

D.1 — serialize.ts — the exact boundary HistoryStore already draws

The set of fields to persist is already defined in the codebase: HistoryStore's pick() keeps "every non-function field of the project store." Persistence reuses that boundary, adds a schemaVersion, and drops nothing UI-ephemeral (there's nothing ephemeral in ProjectStore by design — transport lives in TimeStore, view state in UIStore).

// persistence/serialize.ts
export function serialize(s: ProjectState): ProjectDocument {
  const { tracks, rootTrackIds, bpm, beatsPerBar, totalBars } = s;   // non-function fields
  return { schemaVersion: CURRENT_VERSION, tracks, rootTrackIds, bpm, beatsPerBar, totalBars };
}
export function hydrate(doc: ProjectDocument) {
  const { schemaVersion, ...fields } = doc;
  useProjectStore.setState(fields);   // same shape HistoryStore restores into on undo
}
Keep the serialization boundary honest as the store grows. The reason HistoryStore uses a generic pick() (not a hand-listed set of keys) is precisely so a new field is covered automatically. Persistence should do the same — a field added to ProjectStore is part of the document by default, and part of the undo snapshot by default, with no extra wiring. If a field ever should not persist (a genuinely transient one), that's the exception that justifies moving it to UIStore/TimeStore instead.

D.2 — autosave.ts — a module singleton, not React state

Autosave is plumbing, not UI — a module-level subscription and timers, exactly like HistoryStore's recorder. The only React-visible surface is a save-status signal for an indicator.

// persistence/autosave.ts (elided — full version in the architecture doc §4.1)
export function startAutosave(projectId) {
  return useProjectStore.subscribe((state, prev) => {
    if (state === prev) return;               // reference diff — immutable store
    schedule(() => projectRepo.save(projectId, serialize(state)));  // ~1s debounce, non-overlapping
  });
}
Files this leans on, and why: ProjectStore (the document — subscribe only, no writes), projectRepo (the one door to Supabase), and a browser lifecycle event (beforeunload) for the final flush. Nothing in the editor's edit path changes — autosave is pure observation, the same relationship HistoryStore already has with the store.

E · Wiring autosave & load to the editor

editor/hooks/useProjectPersistence.tseditor/App.tsx

// editor/hooks/useProjectPersistence.ts
export function useProjectPersistence() {
  const id = useSearchParams().get('project');
  useEffect(() => {
    if (!id) return;
    let stop;
    (async () => {
      const doc = await projectRepo.load(id);   // SELECT + upgradeDocument()
      hydrate(doc);                             // setState the store
      stop = startAutosave(id);                  // arm AFTER hydrate
    })();
    return () => stop?.();                       // unsubscribe + final flush on unmount
  }, [id]);
}
  • edit editor/App.tsx — call useProjectPersistence() near the top of the editor tree; add a small save-status chip to the header (reads the autosave status signal). The Canvas, timeline, piano roll — untouched.
  • edit app/editor/page.tsx — already loads the editor ssr:false; no change needed beyond the query param it already receives.

F · Phase-by-phase build (with files)

Phase 1 — Schema: data JSONB + RLS + Drizzle db/

Stand up the storage shape with zero app behavior change: the editor still runs in-memory, but the column, the policies, and the migration tooling exist.

  • new db/schema.ts (the projects table, incl. new data, schema_version, updated_at), drizzle.config.ts, db/migrations/0001_add_project_document.sql (generated: add columns + enable row level security + the four owner policies).
  • edit package.json — add drizzle-orm, drizzle-kit and db:generate / db:migrate scripts.
  • Reconcile user_id vs owner (rename or alias — §C.2). Existing rows get a default empty document at schema_version = 1.

Phase 2 — persistence module + projectRepo persistence/

  • new persistence/supabase.ts (re-export the existing browser client), persistence/types.ts (ProjectDocument), persistence/serialize.ts, persistence/upgrade.ts (CURRENT_VERSION = 1, empty chain), persistence/projectRepo.ts (list/load/save/create/remove).
  • No editor wiring yet — the repo is exercised by a throwaway script / test (save a doc, load it back, assert equality). Proves round-trip serialization before touching the UI.

Phase 3 — Autosave + load wiring ★ keystone autosave App.tsx

  • new persistence/autosave.ts, editor/hooks/useProjectPersistence.ts.
  • edit editor/App.tsx — mount the hook + a save-status chip. First real "edit, reload, it's still there." This is the phase that proves the whole thesis; everything after hardens it.
The keystone test: open a project, add a track and some notes, wait ~1s (chip flips to saved), reload the tab — the track and notes are back. If that round-trips, the document layer works; if not, it's the serialize boundary or the RLS policy (a save silently filtered to zero rows). Those two are the usual first bugs.

Phase 4 — Project browser through the repo app/projects

  • new persistence/hooks/useAuth.ts, persistence/hooks/useProjectList.ts.
  • edit app/projects/page.tsx — replace inline supabase.from('projects')… with projectRepo.list/create/remove; create now inserts a default ProjectDocument. edit src/components/ProjectsDisplay.tsx if the summary shape changes (add updated_at for a "last edited" sort). Behavior identical; source of truth unified.

Phase 5 — Doc-versioning + conflict safety upgrade chain

  • edit persistence/upgrade.ts — as the model changes, append vN → vN+1 steps (each pure, immutable, fixture-tested). Bump CURRENT_VERSION.
  • edit persistence/projectRepo.ts — add an optimistic-concurrency check on save: pass the loaded updated_at, and if the row moved since, surface a conflict instead of clobbering (§4.3 of the architecture doc). Optional: a Supabase Realtime "open elsewhere" flag.

Phase 6 — Audio bytes → Storage deferred

  • new persistence/audioRepo.ts — upload/download the clip bytes to a Supabase Storage bucket (RLS-scoped like the table), referenced by the doc's audioClip.ref. edit editor/audioSource.ts to resolve ref through the bucket instead of in-memory. The clip metadata already persists in the blob today — this only moves the bytes. Deferred until audio persistence is a priority.

H · Risks & open questions

  • risk RLS policy is the silent failure. A wrong or missing policy makes every save/load return zero rows with no error — the classic "why isn't it saving?" Mitigation: policies ship in the phase-1 migration and are tested (a save-then-load round-trip in phase 3 is the canary). Watch user_id vs owner naming. (§6, §C.2)
  • risk Serialization boundary drift. If persistence hand-lists fields instead of reusing HistoryStore's generic pick(), a new store field silently stops saving. Mitigation: reuse the same generic boundary; a persisted field and an undoable field are the same set by construction. (§D.1)
  • risk Hydrate triggers a save. Arming autosave before hydrating fires a redundant (or looping) save on load. Mitigation: arm strictly after hydrate; the reference diff also short-circuits an identical state. (§E)
  • risk Upgrade-step correctness. A buggy/edited vN → vN+1 corrupts old blobs on load. Mitigation: steps are pure, append-only, and fixture-tested against captured old-version documents; never mutate a shipped step. (§phase 5)
  • open Concurrency floor. Two tabs clobber (last-write-wins). Decide the minimum: an updated_at conflict check (a where-clause) vs a Realtime "open elsewhere" flag. Ship the check; defer collaboration. (§4.3 arch)
  • open Autosave debounce vs data loss. An edit in the debounce window before a crash is lost. Mitigation: flush on beforeunload/visibilitychange and explicit close; keep the window ~1s.
  • open Audio bytes. The clip metadata persists today; the bytes need a Storage bucket + RLS. Decide the bucket model before shipping phase 6.
  • decided Local-first blob, thin spine, RLS + Drizzle. Zustand authoritative; one JSONB document mirrored debounced; ownership enforced in Postgres; schema (columns + policies) in git. This is the recommended shape and everything above builds on it.
