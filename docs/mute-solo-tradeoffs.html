Mute & Solo in a Nested Hierarchy

What "solo" should mean when objects can nest inside objects and carry automation / ability / modifier / modulator children. The design space, the DAW conventions, and a recommended rule set.

The one-line answer: keep object solo global (only soloed objects render), but make it hierarchy-aware - soloing an object also keeps its ancestors and its own shaping children (automation / ability / modifier / modulators targeting it) audible, so it renders in context and fully shaped. Mute hides a whole subtree. Child-track solo stays local (among siblings). This is DAW "solo = only soloed tracks play" plus folder-solo semantics, mapped onto a scene graph where "audible" means "contributes to the render."

In a DAW, "solo" is simple because tracks are a flat list: solo one, the rest go silent. Cabin's tracks are a tree - an object can nest under another object, and each object carries automation, ability, modifier, and modulator children that don't render on their own but shape the object. "Silence everything else" is now ambiguous: does soloing a cube also silence the automation that drives its size? The modulator pulsing it? Its parent group it inherits transforms from? This doc picks answers that stay predictable.

1 · What "audible" means here

Port the audio metaphor cleanly and the rest falls out. A track is one of two things:

  • An object - renders something (a cube, a circle). "Audible" = it contributes pixels to the frame.
  • A shaper - an automation, ability, modifier, or modulator child. It renders nothing; it only changes how some object renders. "Audible" for a shaper means its effect is applied.
This split is the whole game. Muting an object removes something you can see. Muting a shaper removes an influence on something else you can see. So "solo this object" and "solo this automation lane" are fundamentally different verbs - and conflating them is the trap. Solo among objects is about which things appear; solo among a shaper's siblings is about which influence wins.

2 · What the code does today

The resolver (src/editor/core/engine/resolve.ts) already makes a defensible set of choices. Worth stating them plainly, because the recommendation is mostly "keep these, add hierarchy-awareness."

RuleWhereBehavior
Object solo is global anyObjectSolo, objectOff If any object anywhere is soloed, every non-soloed object is muted. Solo scans the whole flattened forest, not siblings. DAW-standard
Muted/soloed-out objects are hidden objectOff → blackedOut (VisualEngine.ts:82) blackedOut = obj.muted || …blackout span…. A muted object stops contributing pixels.
Solo doesn't touch a soloed object's own shapers comment above anyObjectSolo "Modulators + children keep their own mute, so soloing an object never disables its own automation, its modifiers, or a modulator that targets it." correct
Ability-child solo is per-object resolveAbilityEvents Among one object's ability children: if any is soloed, its non-soloed siblings go silent. Scoped to that object.
Automation-child solo is per-object resolveAutomations (anyAutoSolo) Same pattern, among an object's automation children.
Modifier-child solo is per-object applyModifiers (anyModSolo) Same pattern, among an object's mute/add/suppress/… modifier children.
A mute modifier blacks out its span applyModifiers → blackouts Different mechanism from track mute: it hides the object only during the modifier note's time range.
Where it's debatable. Two gaps, both about hierarchy, because the resolver flattens the tree (flattenTree) and solo currently ignores nesting entirely:
  • Solo a nested object → its parent group disappears. If a cube nested under a "Scene" object group is soloed, the group is a non-soloed object, so objectOff mutes it. But the cube inherits the group's world transform (composeLocal, parent-before-child). Muting the parent can move/hide the very object you soloed. The transform still composes off worldMatrices, but a hidden parent is conceptually "not playing," which is a smell.
  • Solo a parent group → its child objects disappear. Soloing the group solos only the group; its child objects aren't soloed, so they mute. You soloed a container and got an empty container. Almost never what anyone wants.
These are the two cases the recommendation fixes with an ancestor + descendant pass.

3 · The solo design space

When you solo a track, "what gets muted" has four candidate answers. They differ only in scope.

OptionSolo mutes…ProsCons
(a) Global everything else, everywhere Dead simple, matches Ableton/Logic default. "Show me only this." One mental model. Naïvely applied, kills a nested object's parent context and a parent's children (the two smells above).
(b) Siblings only tracks with the same parent Local, composable; solo inside a group doesn't blank the rest of the show. Not what "solo" means to anyone from a DAW. Soloing a leaf still leaves unrelated top-level objects on screen - confusing "why is that still here?"
(c) Global, but keep ancestors + descendants everything except the soloed track's chain up and its subtree down Global clarity ("only this thing") plus renders in context and fully populated. Matches folder-solo in DAWs. Slightly more to explain; needs an ancestor/descendant pass in resolve. (Cheap - the tree is tiny.)
(d) Solo-in-place only within the soloed track's own subtree Powerful for auditioning one branch of a big scene without disturbing others. A second, different solo verb. Great as a modifier (Alt-solo), bad as the default - two behaviors on one button is a footgun.
Pick (c) as the default - global solo that spares the soloed object's ancestors and descendants. It's the only option that is both "only this thing" (what solo means) and "this thing rendered correctly" (in its parent's transform, with its own shapers intact). Keep (d) solo-in-place available later as a modifier gesture for auditioning a branch.

4 · Children of a soloed track - keep the shapers

Soloing an object must not silence the children that shape it. An automation lane driving its size, an ability lane it fires, a modulator pulsing it - these are the object's voice, not separate voices. Silencing them would "solo" a hollow version of the thing.

The code already gets this right: the comment on anyObjectSolo is explicit that modulators and children keep their own mute regardless of object solo. So a soloed object keeps its automation (resolveAutomations runs unconditionally on it), its modifiers (applyModifiers), and any modulator whose routing targets it (modulators are resolved independently, gated only by their own muted flag).

5 · Ancestors of a soloed nested object - keep the chain

An object nested under a group inherits that group's world transform (composeLocal multiplies parent world × local, parent-before-child in VisualEngine.ts). If solo mutes the parent group, you're soloing an object while yanking the coordinate frame it lives in. The soloed object might jump to origin, or - if the parent itself renders geometry that positions it - lose its context entirely.

Fix: when computing global solo, treat a soloed object's ancestor chain as implicitly soloed (kept audible). They stay on screen so the soloed descendant renders where it belongs.

Subtlety - does a kept ancestor render its own geometry? If the "Scene group" is a pure container (no visible mesh, only a transform), keeping it is free. If it also draws something, keeping it means that geometry shows too. That's acceptable and honest: the soloed object lives inside that thing, so seeing the container is context, not clutter. If you later want "transform-only, don't draw the ancestor," that's a per-object renders-geometry flag, not a solo concern.

6 · Mute down a subtree

Mute is the easy half, and its rule is a hard invariant: a muted parent hides its entire subtree. If a group is muted, nothing inside it can render - a child object has no visible effect when its coordinate frame is gone. This is the exact dual of the ancestor rule: solo keeps the chain up, mute cuts the chain down.

Today this transitivity isn't enforced. objectOff only reads a track's own muted flag; it doesn't walk ancestors. Because transforms compose off worldMatrices and a muted parent still gets a world matrix computed, a child of a muted parent will currently still render (just possibly mis-placed). The fix is a one-line ancestor check in the mute test: an object is hidden if it or any ancestor is muted.

8 · Recommendation

The rule set - five lines, no exceptions

  1. Object solo is global. If any object is soloed, un-soloed objects are muted. (Keep today's anyObjectSolo.)
  2. Solo spares the chain up and the subtree down. A soloed object's ancestors stay audible (so it renders in its inherited frame) and its descendant objects stay audible (so soloing a group isn't an empty box). to build
  3. Solo never touches shapers. A soloed object keeps its own automation, ability lanes, modifiers, and every modulator targeting it. (Already true.)
  4. Mute is transitive downward. An object is hidden if it - or any ancestor - is muted. to build
  5. Child-track solo stays local. Solo among an object's automation / ability / modifier children mutes only its non-soloed siblings. (Already true, per-object.)

What actually changes in code

Small, and localized to resolve.ts - the tree is tiny, so an extra walk is free.

// In resolveProject, replace the flat objectOff with a hierarchy-aware version.

// 1. Solo set = soloed objects, PLUS their ancestor chains, PLUS their subtrees.
const soloRoots = objectTrackIds.filter(id => p.tracks[id]?.solo)
const keptBySolo = new Set<string>()
for (const id of soloRoots) {
  addAncestors(id, keptBySolo)   // walk parentId up - render in context
  addSubtreeObjects(id, keptBySolo) // walk childIds down - don't solo an empty box
}
const anySolo = soloRoots.length > 0

// 2. Mute is transitive: hidden if self or any ancestor is muted.
const mutedByAncestor = (id: string) => {
  for (let t = p.tracks[id]; t; t = t.parentId ? p.tracks[t.parentId] : undefined)
    if (t.muted) return true
  return false
}

// 3. The new object-off test.
const objectOff = (id: string) =>
  mutedByAncestor(id) || (anySolo && !keptBySolo.has(id))
Everything else - shaper children keeping their own mute, per-object child solo, the mute-modifier blackout spans - stays exactly as written. The change is purely: solo and mute now respect nesting. Ports over cleanly because flattenTree already exposes parentId/childIds, and computeAtBeat already composes parent-before-child, so a kept ancestor's world matrix is ready when its soloed descendant renders.
In one breath: global solo (only the soloed thing), but spare its parent chain so it renders in context and its subtree so a group-solo isn't empty; never silence a soloed object's own shapers; mute hides the whole subtree beneath it; child-lane solo stays a local "which influence wins" among siblings. It's DAW solo/mute with folder rules - mapped onto "audible = contributes to the frame."
