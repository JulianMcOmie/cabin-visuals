Render Performance — Implementation Plan

Turning render-lag-architecture.html into specific files and functions: a render governor that idles the loop when the beat is static, an adaptive quality scalar that scales the frame to the machine, and — behind them — a shader-port of the worst Canvas-2D instruments. Each phase ships alone and helps on its own; nothing changes the instrument authoring contract.

This plan adds one small module — core/visual/renderGovernor (frameloop control + the quality scalar) — plus a shared markTextureIfChanged helper, a DPR cap on the <Canvas>, and a per-instrument quality read in the full-frame/particle instruments. No document change, no store-shape change, no instrument-contract change — instruments remain pure functions of the beat; the heaviest ones additionally read a global quality scalar to size their canvas/particle budget. The export path is untouched (it pins its own frameloop/DPR). Decisions baked in from the architecture doc: render-on-demand while paused (safe under the pause invariant) · DPR clamp ~1.5 · gate needsUpdate on change · adaptive quality on R3F's performance API · aspect-cap the fixed-1080p canvases · shader-port the top four Canvas-2D instruments, deferred structural worker option.

A · Components that change — and one layer down

Highest level first, then the key function inside each. Files in monospace; badges: new edit.

new core/visual/renderGovernor.ts — the brain

A tiny module (no React) holding two pieces of global state: the current frameloop intent and a quality scalar (1.0…0.5). Exposes getQuality() for instruments and requestRender() for the demand path. Subscribes to nothing itself — it's driven by the component below.

new components/visual/RenderGovernor.tsx — inside the Canvas

Mounted in <Canvas> next to VisualBeatSync. Reads R3F's invalidate/setFrameloop/setDpr. Watches TimeStore.isPlaying (→ always while playing, demand while paused) and subscribes to the change events (scrub, edit, resize) to fire one invalidate(). When playing, runs the quality feedback loop (measure → adjust scalar → setDpr).

edit App.tsx — the Canvas props + mount

Add dpr={[1, 1.5]} and frameloop="demand" to <Canvas> (the governor flips it to always on play). Mount <RenderGovernor />. One-line each.

new core/visual/textureUpdate.ts — the upload gate

markTextureIfChanged(texture, signature) — sets needsUpdate only when the frame's content signature (beat + relevant params) differs from last call. The Canvas-2D instruments call this instead of texture.needsUpdate = true.

edit the full-frame & particle instruments — read quality

Each heavy instrument multiplies its canvas resolution (or particle count) by getQuality(), and routes its texture-dirty flag through the gate. Mechanical, per-file. The fixed-1080p ones (DiamondLattice, WindowsXP) also switch to aspect-scaled sizing.

B · The phase ladder

Each rung is independently shippable and independently valuable; stop at any rung and nothing is broken. Rungs 1–2 are the editing win (cheap), rung 3 is the playback win (self-correcting), rungs 4–5 are the durable source-level fixes.

C · The render governor — frameloop control

The governor's demand path is the whole phase-1 payoff. The shape:

// RenderGovernor.tsx — mounted inside <Canvas>
const invalidate = useThree((s) => s.invalidate)
const setFrameloop = useThree((s) => s.setFrameloop)

useEffect(() => {
  // play/pause drives the loop mode; while playing the beat moves
  // every frame so "always" is correct, otherwise "demand".
  return useTimeStore.subscribe((s) => s.isPlaying, (playing) => {
    setFrameloop(playing ? 'always' : 'demand')
    if (!playing) invalidate() // paint the final resting frame once
  }, { fireImmediately: true })
}, [])

useEffect(() => {
  // any change to the *picture* while paused → one frame
  const unsubs = [
    useTimeStore.subscribe((s) => s.currentBeat, () => invalidate()), // scrub
    useProjectStore.subscribe(() => invalidate()),                    // edit / param
    useUIStore.subscribe((s) => s.topPanelFraction, () => invalidate()), // resize
  ]
  return () => unsubs.forEach((u) => u())
}, [])
The one correctness question: can a frame be dropped that should have painted? Only if something moves the picture without one of these events firing. The pause invariant forbids beat-independent motion (the dev canary trips on it), and every scene input — beat, params, viewport — has a subscription above. A resize triggered by ResizeObserver rather than the panel-fraction store is the one to watch; R3F already invalidates on its own canvas resize, so this is covered, but it's the first thing to check if a stale frame ever appears.

D · The quality scalar & the knobs

Phase 3. Build the measurement on R3F's built-in performance regression signal (or a rolling performance.now() average around the render) and map it to a coarse scalar so it doesn't oscillate:

// renderGovernor.ts
let quality = 1          // 1.0 → 0.75 → 0.5, hysteresis on the way back up
export const getQuality = () => quality
export function reportFrameMs(avgMs: number) {
  if (avgMs > 22 && quality > 0.5) quality = Math.max(0.5, quality - 0.25)
  else if (avgMs < 13 && quality < 1) quality = Math.min(1, quality + 0.25)
}

The scalar feeds three knobs, each already a single number in the code:

  • Canvas DPR — setDpr(baseDpr * quality) from the governor (cheapest, whole-scene).
  • Full-frame canvas resolution — the texH/texW formula (and the fixed 1920×1080 ones) multiply by quality; halving texH quarters both the 2D draw and the upload.
  • Particle counts — particleCount * quality in the points instruments (ParticleStreams' 92k, ParticleRiser's 9k, BeatParticleKit's 24.9k).
Why a scalar and not per-instrument tuning: one global number keeps the governor simple and the behavior legible ("the machine is at 75%"), and it composes — three heavy instruments share one budget instead of each guessing. Instruments opt in by reading getQuality(); those that don't are simply unaffected.

E · Phase-by-phase build (with files)

Phase 1 — Render-on-demand while paused renderGovernor · RenderGovernor App.tsx

  • new core/visual/renderGovernor.ts (state + requestRender) and components/visual/RenderGovernor.tsx (the frameloop/invalidate wiring in §C).
  • edit App.tsx: frameloop="demand" on the Canvas; mount <RenderGovernor />.
Phase-1 canary: pause the transport on a project with a heavy instrument (FractalTunnel). The GPU/CPU should go quiet (Task Manager / Activity Monitor), and dragging a timeline block or a slider should feel instant. Then scrub — the visual must track the playhead frame-for-frame (no stale frame). Then play — smooth resumes. If a scrub leaves a stale frame, a subscription in §C is missing.

Phase 2 — DPR clamp + upload gate textureUpdate App.tsx · Canvas-2D instruments

  • edit App.tsx: dpr={[1, 1.5]}.
  • new core/visual/textureUpdate.ts: markTextureIfChanged(texture, signature).
  • edit the Canvas-2D instruments (FractalTunnel, CrtScanlines, DiamondLattice, WindowsXP, SilkSymmetry, …): replace texture.needsUpdate = true with the gated call. Signature = the beat plus the params that affect the draw.

Phase 3 — Adaptive quality scalar ★ keystone quality loop heavy instruments

  • edit renderGovernor.ts + RenderGovernor.tsx: the measure→scalar→setDpr loop (§D).
  • edit the full-frame & particle instruments: multiply canvas resolution / particle count by getQuality().
The keystone canary: on the Windows machine, play a project heavy enough to drop frames. The visual should visibly soften (lower resolution) and the frame rate should recover toward 60; stop playback and it should sharpen back. On the Mac, the same project should stay at quality 1.0 (never regress). If the Mac regresses or the Windows machine never recovers, the thresholds in reportFrameMs need tuning — do this before phase 5.

Phase 4 — Aspect-cap the fixed-1080p canvases DiamondLattice · WindowsXP

  • edit DiamondLattice.tsx, WindowsXP.tsx: replace the hard 1920×1080 with the aspect-quantized texH/texW formula FractalTunnel already uses (then it also gets the phase-3 quality multiplier for free).

Phase 5 — Shader-port the worst Canvas-2D instruments large · staged

  • One instrument at a time, biggest-first: FractalTunnel → DiamondLattice → CrtScanlines → SilkSymmetry. Each becomes a fullscreen-plane fragment shader reading beat (and its params) as uniforms — no 2D canvas, no upload. Reuse the effects pipeline's GLSL plumbing (ShaderWrapper).
  • Keep the Canvas-2D version until the shader version passes the pause canary and a visual A/B at several beats; instruments are independent, so a partial port is fine.

G · Risks & open questions

  • risk A dropped frame while paused. Demand-mode is only correct if every picture-changing input invalidates. Mitigation: the subscriptions in §C cover beat/params/viewport; the pause canary already guarantees no beat-independent motion. First suspect if a stale frame appears: a resize path that bypasses both R3F's own canvas-resize invalidation and the panel-fraction store.
  • risk Quality-scalar oscillation. A too-tight threshold band makes the resolution pump up and down visibly. Mitigation: coarse steps (0.25) + hysteresis (regress at >22 ms, recover only under 13 ms) + a debounce so it adjusts a few times a second, not per frame.
  • risk Interaction with export. Export pins its own frameloop='never' + DPR 1 + fixed size and restores them in a finally. The governor must not fight this — it should no-op while the export driver holds the pin. Mitigation: the governor checks the same "pinned" flag before touching frameloop/dpr.
  • risk Instruments that animate on a held note "while paused." None should — everything is a pure function of the beat, and a held note's visual is a function of beat − note.beat, frozen when the beat is. If the canary ever trips under demand-mode, that instrument has a latent pause-invariant bug independent of this work.
  • open DPR clamp value. Proposed 1.5 (or 1 detected on Windows). Worth an eyeball on a retina Mac that the softer canvas is acceptable; export is unaffected (pins DPR 1 itself).
  • open Per-instrument quality opt-out. Some instruments may look bad at reduced particle counts; a per-instrument minQuality floor is a cheap escape hatch if needed — not designed here.
  • later OffscreenCanvas worker. The full decoupling — named in the architecture doc §4B; built only if heavy-playback responsiveness is still unacceptable after phases 1–5. Not designed here; the render governor is the seam it would eventually move into a worker.
The one-sentence mental model: add a render governor beside the visual engine that flips the loop to demand-mode when the beat is static (editing goes instant, free), runs a measure→scale→setDpr feedback loop when it isn't (playback self-corrects to the machine), and hand the heaviest instruments a quality scalar and an upload gate — then, only if a Windows profile still shows texImage2D dominating, port the four worst Canvas-2D instruments to shaders so the upload stops happening at all.
