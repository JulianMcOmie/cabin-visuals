Video Export — Architecture

Turning a project into an .mp4, entirely in the browser: step the beat frame by frame, render the same pure visuals playback renders, encode with WebCodecs, mix the audio offline, mux, download. No recording, no server.

1 · The pipeline

Export is a fourth consumer of the same clock math — after transport, visuals, and audio. A new core/export/ island steps a beat cursor through the project at exactly one frame per step, drives the existing visual engine at each beat, captures the canvas into a hardware video encoder, renders the audio track through an offline audio graph, and muxes both into an MP4. Nothing plays in real time; a 3-minute project exports as fast as the GPU and encoder can go.

2 · Why the output is exact — the pause invariant pays off

Every instrument is a pure function of the beat (the pause invariant, enforced by lint and the canary). So the frame at beat 12.5 is the same pixels whether you played there, scrubbed there, or exported there. Export doesn't record what happened on screen — it recomputes it, perfectly, at any resolution and frame rate, with zero dropped frames.

3 · The export loop

The live editor canvas is borrowed for the duration: its render loop is paused (frameloop="never"), it's pinned to the export resolution, and the stepper drives it one deliberate frame at a time — set the beat, advance React-Three-Fiber once, hand the canvas to the encoder, wait if the encoder queue is full, repeat. A progress overlay covers the app; on finish (or cancel) the canvas and render loop are restored.

4 · The audio path

Audio doesn't step frame-by-frame — the whole track renders in one offline pass. Each audio block schedules its clip slice into an OfflineAudioContext using the same (when, offset, duration) placement the live AudioEngine uses (the beat⟷second math is extracted into one shared helper so the two can never disagree). The rendered buffer is AAC-encoded and interleaved with the video by the muxer; sync is arithmetic — both timelines derive from the same bpm.

5 · Decisions & non-goals

DecisionWhy
Client-side, WebCodecsHardware encoders in the browser; no server, no upload, project data never leaves the machine. Chrome-first (WebCodecs); a capability gate greys the button elsewhere.
MP4 · H.264 + AACPlays everywhere she'll post it. Muxed in-browser (mp4-muxer). WebM/VP9+Opus is the named fallback if an AAC encoder is missing.
Audio is optionalAn "Include audio" toggle in the dialog (on by default) — off skips the offline mix entirely and produces a video-only MP4.
Recompute, don't recordMediaRecorder/capture rejected: real-time only, drops frames, non-deterministic. The pause invariant makes exact offline recompute free.
Fixed export resolution1080p (16:9) default, independent of the window — same principle as the aspect-aware instruments: output is unified regardless of how big the editing window was.
Borrow the live canvasOne visual-engine singleton, one WebGL context. Pin its size during export behind a progress overlay; restore after. A second hidden canvas would double-drive the engine.
Whole project, one fileRange/region export, image sequences, alpha, and server rendering are all explicitly later.
The one-sentence mental model: export walks the beat from 0 to the end at one frame per step, asks the same pure visual engine "what do you look like here?", feeds the answers to a hardware encoder, renders the audio track once through the same placement math, and zips the two streams into an MP4 — determinism the pause invariant already guarantees, pointed at a file instead of the screen.
