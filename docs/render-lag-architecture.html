Render Performance — Architecture

Keeping the editor responsive no matter how heavy the visuals get: a render governor that idles the loop when the picture is static, scales quality to the machine when it isn't, and — for the worst offenders — moves the per-frame bitmap upload off the CPU entirely. Everything hangs off the pause invariant that already makes the pixels deterministic.

1 · The problem, in one loop

The editor renders through a single R3F <Canvas> whose loop runs on the main thread — the same thread as React, the timeline, and input. The loop is frameloop="always": it renders every frame forever, whether or not anything changed, at up to 2× device pixel ratio, with no quality throttle (all verified — there are zero perf guards outside the export path). So the cost of one frame is charged directly against UI responsiveness: a 40 ms instrument frame is a 40 ms input stall.

2 · The lever the pause invariant hands us

Every instrument is a pure function of the beat (enforced by lint and the dev canary). The corollary the render loop never exploited: a paused editor is a static frame. If the beat isn't moving, the picture isn't either — so there is nothing to render. Today the loop redraws that identical frame 60 times a second anyway, uploading megabytes of unchanged canvas each time.

That makes render-on-demand not just an optimization but a provably correct default while paused: switch frameloop to "demand" and render exactly one frame per real change — play, scrub, an edit, a resize. The main thread goes quiet the instant you stop the transport, so editing is responsive on any machine no matter what's mounted. This is the same determinism the export pipeline leans on, pointed at idleness instead of a file.

3 · Scaling the frame to the machine

Render-on-demand fixes editing but not playback — during playback the beat moves every frame, so the loop is legitimately always-on and a heavy scene still overruns on a slow machine. The answer is to make the frame's size adapt: a governor measures a rolling average frame time and drives a single quality scalar that feeds three knobs — canvas DPR, the full-frame instruments' canvas resolution, and particle counts. Over budget → scale down; comfortable headroom → scale back up.

This is the direct answer to "lags on Windows but not Mac": the same scene downscales on the Windows laptop until it fits 16.7 ms and stays full-resolution on the fast Mac — the app adapts to the machine instead of assuming one. R3F provides the primitive (the performance/regress() API) to build the governor on.

4 · Removing the cost at the source

The two tiers above manage the symptom brilliantly but the underlying operation — draw a full-screen 2D canvas, then texImage2D it to the GPU every frame — is inherently wrong for animated content, and it's exactly the operation ANGLE makes expensive on Windows. Two structural options delete it, independently:

OptionWhat it removesCostWhen
A · Shader-port the worst Canvas-2D instrumentsThe per-frame 2D draw AND the upload — the effect is computed in a fragment shader on a fullscreen plane, reading beat as a uniform. No canvas, no texImage2D.A real rewrite per instrument (FractalTunnel, CrtScanlines, DiamondLattice, SilkSymmetry). B · OffscreenCanvas in a Web WorkerThe coupling itself — the render loop runs on a worker, so a heavy frame stalls the worker, not the editor UI.Large: stores/engine/instruments assume main-thread + DOM; Canvas-2D instruments need OffscreenCanvas-2D in the worker; the export path borrows the live canvas.Only if playback responsiveness during heavy visuals is still unacceptable after A.
A and B are not either/or. A attacks the cost of the frame; B attacks the coupling of the frame to the UI. Shipping A first is almost always right — it's cheaper, it helps every platform (not just the responsiveness of the main thread), and it shrinks whatever B would eventually have to move. B is the architecturally pure answer but the last-resort one.

5 · Decisions & non-goals

DecisionCall
Render-on-demand while pausedYes — provably safe under the pause invariant; the highest-leverage cheap win. The governor owns frameloop.
Default DPR capYes — clamp to ~1.5 (or 1 on Windows) for the editor canvas; export already pins DPR 1 independently.
Gate needsUpdate on changeYes — a shared helper in the instrument-frame layer; belt-and-suspenders with demand-mode.
Adaptive quality scalarYes — the self-correcting fix for the platform gap; built on R3F's performance API.
Fixed-1080p canvas offendersAspect-scale + cap them (DiamondLattice, WindowsXP) — no reason to draw 1080p for a small viewport.
Shader-port the top Canvas-2D instrumentsYes, staged — the durable source-level fix; four candidates named.
OffscreenCanvas workerDeferred — designed as a seam, built only if 1–3 leave heavy playback unacceptable.
Change instrument authoring contractNo — instruments stay pure functions of the beat; the governor and caps are invisible to them (they read a quality scalar at most).
Touch the export pathNo — export already pins its own frameloop/DPR/size; the governor is a live-editing concern only.
The one-sentence mental model: a render governor sits on top of the untouched instruments and does three things the pause invariant makes safe — idles the loop when the beat is static so editing is instant, scales the frame's resolution to whatever budget the machine can hit so playback self-corrects, and (for the four worst offenders) trades a per-frame CPU canvas upload for a GPU shader so the Windows-expensive operation stops happening at all.
