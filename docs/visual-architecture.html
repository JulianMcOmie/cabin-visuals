There are three artifacts in play: the year-old visualObjectEngineDocs.md (a declarative, code-level instrument-authoring API), Tyler's Excellent-DAW (a complete, working, performant engine), and the current Cabin Visuals (a single hand-wired Cube). None of them is the answer by itself. This doc reconciles them against your vision and proposes a concrete path.

The one-line answer: borrow Tyler's engine shape (stateless resolve → per-frame compute → pull; hybrid R3F + imperative pools) but reshape the conceptual model around your vision — first-class object vs modulator instruments connected through a port-based modulation matrix, targeted by hierarchy + tags, with instruments authored as declarative specs so "design in English" becomes generate-a-spec, not generate-arbitrary-code.

★ The big picture — internalize this first

Before the section-by-section detail, here is the whole engine in one view, and the five mental models everything else hangs off. Hold these and the rest is elaboration.

Five things to hold in your head

  1. Two instrument kinds. Objects render something; modulators (= shapers) render nothing and emit a signal into ports. (Two non-instrument layers sit alongside: effects = output plugins; modifiers = MIDI-transform tracks like mute / suppress / transpose — §4.7.)
  2. An object has three layers: params (public knobs, automatable), ports (hidden jacks modulators target), abilities (each its own editor). Ports are never user-visible.
  3. The modulation matrix is the new core. Modulators → ports → params, scoped to one object / a tag / a subtree. Automation drives params directly; everything time-varying meets in the render spec.
  4. One pipeline: resolve once → (per frame) compute @beat → modulate → render + effects. Borrowed from Tyler; the modulate pass is the only net-new stage.
  5. Authoring is declarative-first: instruments are serializable specs (so an LLM can write them) with a code escape hatch (so Tyler's instruments import wholesale).

2 · The three reference points

2.1 — The year-old VisualObjectEngine doc

A declarative, chainable API for authoring an instrument's generative behavior in code: engine.defineObject('cube').when(…).forEachInstance(…).withScale(…).applyADSR(…).applyPhysicsEnvelope(…). A synth, given MIDI + time + its own properties, returns a flat list of VisualObjects per frame.

✓ Worth keeping

  • Declarative property mapping — withScale(ctx => …) separates "what to draw" from "how it reacts." The right spirit for a spec.
  • Envelopes as first-class — ADSR + damped-spring physics envelopes baked in. These are exactly "modulator" math.
  • Nesting / generators — forEachInstance for structured replication (one note → a ring of children).

✕ Doesn't fit anymore

  • It's a code closure API (mappers capture this.properties) — not serializable, not LLM-safe.
  • No modulators, ports, or tags — modulation is hard-wired inside each instrument's mappers, not a routable signal.
  • Synth-centric, one object kind — no object/modulator split, no cross-instrument modulation.

2.2 — Tyler's Excellent-DAW engine (the working reference)

A real, performant engine. Three layers, cleanly separated:

  • Data model — one unified Instrument (instruments/types.ts:29) with hasAudio/hasVisual flags and an optional VisualComponent. A Track (core/types.ts:129) carries instrumentId, visualPlugins[], an optional automationConfig, blocks[], and parentId/childIds for nesting.
  • Resolution + state engine — resolveProject() flattens the track tree once into ResolvedTrack[] (events + automation lanes + blackout regions); computeStatesAtBeat(beat) binary-searches per frame into a VisualInstrumentState per track (visualTypes.ts:6: activeNotes, params, bloom, colorShift, noteOnCount, palette). Stateless & pull-based — components call getTrackState(trackId).
  • Renderer — @react-three/fiber tree (VisualScene → TrackRenderer → instrument) wrapped by a plugin chain: TransformWrapper (offset/rotate/scale), CloneWrapper (echo/tile/radial), ShaderChain (FBO post-fx), composited through SceneCompositor + masks. Performance via managed instancing pools (ManagedBlobPool/DotSet/LineSet) and imperative three.js (FBOs, group.visible, material swaps). A one-line VisualBeatSync calls computeStatesAtBeat(currentBeat) first each frame.

✓ Steal this

  • resolve-once → compute-per-frame → pull. Clean, fast, testable. The clock is just currentBeat.
  • Plugin chain (transform/clone/shader) with a uniform contract + a registry, and per-param settingsSchema that auto-builds inspector UI and bounds automation.
  • Managed instancing pools — the answer to "hundreds of objects."
  • Hierarchy + modifier tracks — child tracks that transform a parent's event stream (transpose / scale / rhythm / suppress / mute…). See §4.7.

△ Gaps vs your vision

  • No object/modulator distinction — instruments are uniform; "modulation" is keyframe automation lanes + event-transform, not reusable modulator instruments.
  • No first-class ports — any settingsSchema key is automatable, but there's no curated "scale / energy / density" input contract.
  • No tags — targeting is tree-bound (a child automates its parent). Cross-cutting "pulse these 18" isn't expressible.
  • Instruments are bespoke React components — infinitely expressive, but not serializable / not LLM-generable.

2.3 — Current Cabin Visuals (the starting line)

One Cube (editor/instruments/Cube.tsx): a useFrame loop reads currentBeat, scans the project for cube notes, computes a pulse inline, and mutates the mesh. Params live on track.params; the schema lives beside the component (InstrumentDef). The Tone transport is already the clock and a RAF playhead already exists. Good bones, but everything is hand-wired and single-instrument.

Net read: Cabin is behind Tyler mechanically but ahead conceptually — your notes already name the model (objects, modulators, ports, tags) that Tyler's engine never formalized. The job is to build that model on top of Tyler's proven mechanics.

4 · Recommended architecture

A four-layer engine. Layers 1–2 are borrowed shape from Tyler; layer 3 (the modulation matrix) is the new core that encodes your vision; layer 4 is the rendering you mostly borrow.

4.1 — Data model

Two instrument defs (the reusable templates) and the per-track instances that reference them. Defs are where "object vs modulator" becomes real.

// ── Reusable instrument definitions (the "library" items) ──

interface ObjectInstrumentDef {
  id: string; name: string; kind: 'object';
  tags?: string[];                 // default tags applied on insert
  params: ParamDef[];             // the knobs: size, hue, position… — as many as possible, ALL automation-targetable
  ports: PortDef[];               // shared modulation vocabulary modulators target; each maps onto params
  abilities: AbilityLaneDef[];    // each = its own nested MIDI editor (a "lane" = a full editor, not a row — §4.4)
  render: RenderSpec              // declarative (see §4.5) …
        | { component: React.ComponentType<{ trackId: string }> }; // … or code hatch
}

interface PortDef {                // a modulation jack, NOT a knob — bound to params in the render spec
  key: string;                    // 'scale' — from a small, shared vocabulary across instruments
  label: string;
  type: 'scalar' | 'color' | 'vec';
  range?: [number, number];
  combine: 'add' | 'multiply' | 'max' | 'replace'; // how multiple modulators stack on this port
  default: number;                // resting value with no modulation
}

interface ModulatorInstrumentDef {
  id: string; name: string; kind: 'modulator';
  outputType: 'scalar' | 'color';
  trigger: AbilityLaneDef;        // its own MIDI lane (what fires it)
  shapeProps: PropSchema[];       // decay, curve, amount, frequency…
  // declarative envelope/LFO spec, OR a pure evaluate() for built-ins:
  evaluate?: (ctx: ModCtx) => number | Color;
}

// ── Per-track instance (what lives in the project) ──

interface Track {                  // extends today's Track
  id; name; parentId?; childIds;  // nested hierarchy — TO BUILD (today: top-level tracks only)
  tags: string[];                // cross-cutting groups
  instrumentId: string;         // → ObjectInstrumentDef | ModulatorInstrumentDef
  params?: Record<string,number>;   // per-track param values (override def defaults; automation drives these)
  lanes: Block[];                 // one block/stream per ability lane; each renders as a nested sub-track (§4.4)
  targets?: Routing[];            // modulator-only: ports this drives — may be MANY (one modulator → several ports)
}

interface Routing {                // "this modulator → that port, over that scope"
  port: string;                  // 'scale'
  scope: { kind: 'track'; id } | { kind: 'tag'; tag } | { kind: 'subtree'; id };
  amount: number;                // routing depth
}
Parameters vs ports — and why both. Two layers, like a synth's knobs vs its CV jacks. Parameters are the concrete knobs (size, hue, position…): as many as possible, user-editable, and all automation-targetable (Tyler/EDAW-style — an automation lane drives any param's value over time). Ports are a small, shared modulation vocabulary (scale, rotation, hue, opacity, energy, density…) that modulator instruments target. The payoff is reuse: a "Rotate" modulator drives the rotation port of any object, without knowing that object names its param spin or angle. Each instrument maps its ports onto its params in the render spec (param.size * (1 + port.scale)).

The "won't I see a scale port and a size param?" worry dissolves: ports are never user-visible at all. Params are the knobs in the inspector; ports are internal plumbing — a modulator has its target port(s) baked into its definition (a Size Pulse targets scale), so the user only ever picks a modulator and a target scope, never a port. Automation drives param bases directly; modulators drive ports under the hood; a param can take both (base curve + modulation, like an envelope plus an LFO). The alternative — "every param is modulatable, no separate ports" — is simpler but discards the shared vocabulary, so modulators stop being generic. Keep the middleman; just keep the port set small and standard.

4.2 — The modulation matrix

This is the part neither Tyler nor the old doc has, and it's the heart of your design. A routing is "modulator ⟶ port, scoped to a track / a tag / a subtree." The scope is what makes "pulse the lead-vocal group" a one-click operation instead of 18 checkboxes.

// Per frame, after each modulator's output(beat) is computed:
for (const obj of objects) {
  for (const port of obj.def.ports) {
    let v = port.default;
    for (const r of routingsHitting(obj, port)) {   // direct + via obj.tags
      const out = modulatorOutput[r.modulatorId] * r.amount;
      v = combine(port.combine, v, out);            // add | multiply | max | replace
    }
    obj.portValues[port.key] = clamp(v, port.range);
  }
  obj.finalProps = obj.def.resolveProps(obj.params, obj.portValues);
}
  • Resolution scope is precomputed: build a tag → [objectId] index and a port → [routing] index once at resolve-time, so the per-frame loop is cheap even with many routings.
  • Combine semantics live on the port (not the routing) so the object author controls how stacking behaves (energy adds; opacity multiplies; a "freeze" replaces). One modulator can drive several ports at once — that's just several routings.
  • Two paths into a property: a parameter's base value (which automation lanes drive directly, any param) and the ports (which modulators drive). They meet in the render spec — param.size * (1 + port.scale) — so a knob can be automated and modulated at the same time.
Recommendation: route every modulator through ports — pulses, LFOs, envelopes, audio-reactive signals — so "many modulators" stays one matrix with many sources rather than many special cases. Keep automation as its own direct-to-parameter path so every param is automatable, not just the port-backed ones; the render spec is where the two combine.
How automation works — borrow Tyler's trick. An automation lane isn't a separate curve-drawing tool; it's a MIDI editor where a note's pitch encodes the parameter value (high note = high value) and its time is the keyframe time. You "play" the automation as notes and the engine interpolates between them per a chosen mode (step / linear / eases / exponential / smooth-step) — so the MIDI editor doubles as the automation editor, and freehand curve-drawing can come later as an optional second input. Each automation lane is a child track targeting one param (an instrument param or an effect/plugin param), exactly like Tyler's automationConfig + pitch→value remap.

4.3 — Resolve → compute → modulate → render

  1. Resolve (once, on project edit) — adopt Tyler's resolveProject almost verbatim: flatten the tree, expand looping blocks, apply modifier children to event streams (mute / suppress / transpose…, §4.7), build per-instance event lists. Add: split instances into objects vs modulators, build the routing & tag indices.
  2. Compute @beat (per frame) — Tyler's computeStatesAtBeat: binary-search active notes / note-on counts per track. For modulators, this yields the trigger events feeding their envelopes.
  3. Modulate (per frame, new) — evaluate each modulator's output(beat), then run the matrix (§4.2) to fill every object's portValues and finalProps.
  4. Render (per frame) — R3F reads finalProps and draws. Effects (plugins — transform/clone/shader/mask, Tyler's chain) then wrap the rendered output. A VisualBeatSync-style component runs steps 2–3 first each frame, exactly like Tyler.
Integration with Cabin today is small: you already have currentBeat from the Tone transport and a RAF. A <VisualBeatSync/> reading useTimeStore.getState().currentBeat drives steps 2–3; the current Cube becomes the first ObjectInstrumentDef and its inline pulse becomes a built-in Pulse modulator.
Editing stays instant — resolve is incremental and deferred. Two rules keep big projects responsive (EDAW lagged on every note edit because it re-ran a full resolve synchronously). (1) Deferred: resolve is an asynchronous observer of the project, never a synchronous step inside an edit handler — subscribe to store changes and schedule it, debounced, so a burst of edits triggers one resolve; the scene keeps rendering the previous resolved graph until the new one lands. (2) Incremental: never re-resolve the whole project — a note edit only changes the edited track's output (plus anything downstream of it in the modifier chain / routings), so cache each track's resolved output and recompute only the dirty set. Because the store is immutable, finding the dirty set is a cheap reference diff — tracks[id] !== prev.tracks[id] picks out exactly the edited track — so one note re-resolves one subtree instead of everything. That's the difference between O(change) and O(project) per edit: it keeps a 500-track project as snappy as a 5-track one.

4.4 — Abilities as lanes (one full editor each)

Your 04/09 note: an object's special abilities each get their own MIDI lane — and here a "lane" means a whole MIDI editor, not a row. An object instrument has several independent ways to play it (a circle's "spawn burst", "ripple", "flash"…), and each is its own full piano-roll, surfaced as its own track-like row nested one level under the instrument's track in the tracks section. Triggering notes inside an ability's editor fires that ability over time.

Lanes and editor-kind are orthogonal — keep both. The clarification is only about lanes: each special ability gets its own editor (its own nested sub-track), rather than being a row in a shared grid. It is not a rejection of Tyler's drum-style rows — those stay, as an editor kind (editorType/rangeLabels) you choose per lane. A drum-like instrument can absolutely use multi-row pads inside one of its editors. Err unrestrictive: an ability is its own editor; what kind of editor (pitched, drum-rows, trigger…) is a separate, free choice.
interface AbilityLaneDef {        // a "lane" = a full sub-editor, not a row
  key: string;        // 'spawnBurst'
  label: string;      // shown as the nested sub-track's name
  editor?: EditorKind;  // each lane can choose its own editor (pitched / drum / trigger…)
  // what a trigger does — declarative effect, or handled by the render spec
}
  • Each entry in the object def's abilities[] becomes its own nested sub-track in the timeline (indented under the instrument track), with its own MIDI editor and its own block/stream — not a relabelled row in a shared grid.
  • A modulator's trigger is the degenerate case — a single ability lane ("when do I fire"), i.e. one nested editor.
  • Reuse the hierarchy machinery: rendering an ability lane is rendering a child track one level down, so this rides on the same nested-track work (§4 phase 5) rather than a separate multi-row widget.

4.5 — Declarative specs & "design in English"

This is the bet behind the "design instruments in English" goal. If an object instrument is a serializable spec the engine interprets, then "design an instrument in English" is "LLM emits a spec," which is safe (no arbitrary code execution), validatable (schema-checked), and uniform (every instrument plugs into params/ports/abilities the same way). If instruments are bespoke React components (Tyler), the LLM has to write and you have to run untrusted three.js — a much worse place to be.

// A RenderSpec: primitives + how props/ports map onto them.
// The old VisualObjectEngine's .withScale(ctx=>…) idea — but as DATA, not closures.
interface RenderSpec {
  primitive: 'circle' | 'box' | 'line' | 'points' | 'plane' | …;
  bindings: {                       // property ← expression over params + port values
    scale:    "param.size * (1 + port.scale)";
    color:    "hsl(param.hue + port.hue*60, 0.65, 0.6)";
    opacity:  "port.opacity";
    emissive: "0.2 + port.energy * 1.2";
  };
  onAbility?: { spawnBurst: "emit 12 points, radial, life 0.5s" };
}
  • The engine ships a small, safe expression vocabulary (arithmetic, hsl(), map(), easing, sin/noise) — the modern, serializable version of the old doc's MappingUtils.
  • Built-in / performance-critical instruments keep the code escape hatch ({ component }) — they're hand-tuned and don't need to be LLM-authored.
  • The grammar can grow: start with primitive + scalar/color bindings (covers most of your modulators), add generators (the old doc's forEachInstance → a repeat node) and ability effects over time.
The honest tradeoff: a declarative spec is less expressive than arbitrary code, and you'll periodically hit "the grammar can't say this." Mitigate with (a) the code escape hatch for built-ins, and (b) treating the grammar as a living vocabulary you extend as real instruments demand it. Don't try to design the whole grammar up front — grow it from concrete instruments.
Explicit goal — port over as many of Tyler's instruments as possible. This is a separate aim from building the engine, but the architecture must not preclude it (and the hybrid model is exactly what enables it). Each of Tyler's instruments is already a React/three.js VisualComponent, so it drops straight into the code escape hatch ({ component }) and — because we're borrowing his renderer + plugin chain wholesale (§4.6) — runs largely unmodified. They come in as code-hatch instruments first and can be re-expressed as declarative specs later where it's worth it. Concrete implication: keep the VisualComponent contract and the per-frame state it reads compatible with Tyler's as you build, so importing is a copy-in rather than a rewrite.

4.6 — Rendering layer

Borrow Tyler's renderer wholesale in shape:

  • R3F tree: VisualScene → ObjectRenderer(perObject); each object wrapped by its effects chain (plugins) — TransformWrapper (offset/rotate/scale) ▸ CloneWrapper (echo/tile/radial) ▸ ShaderChain (FBO post-fx) ▸ scene compositor + masks. Tyler's plugin contract (applyTransform/getClones/fragmentShader + settingsSchema) is good as-is — this is your effects/plugin layer, a separate concept from modulator/shaper instruments (see the note below).
  • Instancing pools: ManagedBlobPool/DotSet/LineSet for the "many objects / a tag of 100" case — one draw call, zero per-frame allocation.
  • Pull-based: renderers read resolved finalProps; no React re-render storm — imperative useFrame updates like today's Cube, but fed by the engine instead of scanning the project itself.
Terminology — three different words, three different things. A modulator (a.k.a. shaper) is the port-driving instrument kind — drives a value per frame through the matrix (§4.2). A modifier is a MIDI-transform track — mute / suppress / transpose / gate… that reshapes the note stream at resolve time (§4.7). An effect is a plugin — transform/clone/shader/mask applied to rendered output. So: a "rotate" modulator drives a rotation port (changes a value); a "suppress" modifier removes notes; a kaleidoscope effect post-processes the picture. Three layers; don't conflate them.

4.7 — Event modifiers (mute, suppress, transpose…)

A third way to "affect" something — distinct from modulators (which drive ports) and effects (which post-process output). An event modifier is a child track whose MIDI regions transform its parent's note stream. This is Tyler's 'modifier' track category, and it's worth importing wholesale — it's a deep, musical toolbox: suppress, transpose, shift, gate, rhythm, swing, scale (velocity), add, override, harmonyMap… plus mute.

mute ≠ suppress — the one subtlety. suppress is an event filter (resolve time): notes whose start falls in a marked region are removed, so they never fire. mute is a render-time blackout: it does not remove events — it flags a region where the instrument is fully hidden (a blackedOut flag in the per-track state), even though its notes "play." Filter vs visibility-gate — both drawn as MIDI regions in a child editor.
  • Where they sit in the pipeline: the event-transform modifiers (suppress / transpose / gate / rhythm / swing…) run in resolve (§4.3, "apply modifier children to event streams") — they reshape the note stream before anything is interpreted. mute's blackout is read in the compute / render pass. Neither touches the port matrix — they're an orthogonal layer.
  • They ride on the nested hierarchy: a modifier is a child of the track it modifies — another reason the hierarchy is functional, not decorative (the same machinery as inherited transforms and ability lanes).
  • Cleanest possible import from Tyler: these are pure event-stream functions (combine(parent, self) → events) with no rendering, so they lift over almost verbatim.

6 · Phased implementation plan

Each phase is shippable and leaves a working app. Borrow where marked.

  1. borrowEngine skeleton. Port Tyler's resolveProject + computeStatesAtBeat + a <VisualBeatSync/> driven by Cabin's currentBeat. Re-express the current Cube as an ObjectInstrumentDef reading engine state instead of scanning the project. No behavior change, new spine.
  2. buildObject defs + params + ports (static). Define ObjectInstrumentDef / ParamDef / PortDef. Give Cube its params (size, hue, …) — all automation-targetable, EDAW-style — and ports (scale, energy, hue) that map onto them. A registry + library panel (Tyler's pattern). Ports resting at defaults — nothing drives them yet.
  3. buildModulators + the matrix (direct routing). ModulatorInstrumentDef, a built-in Pulse (the Cube's inline math, generalized), and the per-frame matrix for single-track routings. First real "drag a Size-Pulse onto a shape" (it drives the scale port internally). This is the keystone phase.
  4. buildTags + group routing. Add track.tags, the tag → [object] index, and tag-scoped routings. Now one pulse drives a whole tagged group (your 04/10 "lead vocal").
  5. buildNested hierarchy + inheritance. Build nested tracks in the timeline (today it's top-level only — net-new), then wire inherited transforms/context down a subtree, plus subtree routing scope. The ability lanes (next) ride on this same nesting.
  6. buildAbility lanes. AbilityLaneDef[] → each ability rendered as its own nested sub-track with a full MIDI editor (reusing the nested-track work from phase 5). Triggers fire abilities.
  7. borrowEffects (plugins) + pools. Bring in Tyler's transform/clone/shader/mask plugins (the effects layer) and the managed instancing pools once you have multiple objects / tagged groups needing scale.
  8. buildDeclarative RenderSpec + interpreter. Replace one or two built-in objects with spec-driven ones; ship the safe expression vocabulary. Most new instruments become specs; built-ins keep the code hatch.
  9. laterEnglish → spec. Once RenderSpec is stable and a handful of instruments are specs, add LLM generation that emits + validates a spec. (Only viable because of the phase-8 substrate — don't attempt before it.)
Critical path: phases 1 → 3 prove the whole thesis (engine + ports + modulators + matrix). If those feel right, everything after is additive. Don't build tags/hierarchy/specs before the matrix earns its keep on one pulse driving one shape.

7 · Risks & open questions

  • riskSpec expressiveness vs code. The declarative grammar will hit walls. Mitigation: code escape hatch for built-ins; grow the grammar from real instruments, not speculation. Decide early which side new instruments default to.
  • riskMatrix cost at scale. objects × ports × routings per frame. Precompute tag/port indices at resolve-time; cap or batch; lean on instancing so 100 modulated objects is still few draw calls. Profile at phase 4.
  • riskResolve cost at scale. A full resolveProject() on every edit is O(whole project) and lags big projects (EDAW's exact failure mode). Fix: make resolve deferred (a debounced observer, off the edit's synchronous tick) and incremental (cache per-track outputs; dirty-diff by reference equality — tracks[id] !== prev.tracks[id] — and re-resolve only the changed track + its dependents). Turns per-edit cost from O(project) into O(change). See §4.3.
  • riskBuild-vs-borrow boundary. Tyler's renderer + pools + plugin chain + resolution are worth lifting; his conceptual model (no ports/modulators/tags) is not. Be deliberate: borrow mechanics, not metaphors — and budget for adapting his code to your data model.
  • openPort combine semantics. Is "energy" additive and "opacity" multiplicative by default? Who sets it — the port (recommended) or the routing? Get this right early; it shapes how stacking feels.
  • openAbilities = nested sub-tracks. Settled direction: each ability is its own editor/stream surfaced as a sub-track nested under the instrument (not labelled rows in one grid). Open: model them as real child tracks (reusing the hierarchy wholesale) vs a dedicated lanes[] on the parent; and how deep nesting + looping/reference blocks read in the tracks UI.
  • openModulator output typing & reaching effects. Scalars and colors are clear; vectors (position offsets) need a type. Decide whether effects (plugins) also expose ports — if a plugin's params are port-backed, a modulator can drive them too, and the one matrix covers both instruments and effects.
  • openCan effects (plugins) be tag-scoped? e.g. "apply this kaleidoscope effect to the lead-vocal tag." Likely yes — let effect application reuse the same tag/subtree scoping as modulator routing.

8 · Recommendation in one breath

Build Tyler's engine, think in your model.

  • Adopt the resolve → compute → pull pipeline, the R3F + imperative pools renderer, and the plugin chain (transform/clone/shader/mask — the effects layer, distinct from modulator/shaper instruments) — these are proven and you shouldn't reinvent them.
  • Make object vs modulator instruments first-class, connected by a port-based modulation matrix — this is the new core and the thing that makes the product feel like your vision instead of a generic visualizer.
  • Use hierarchy for organization and tags for cross-cutting modulation targeting — orthogonal, each with one clear job.
  • Author instruments as declarative specs with a code escape hatch, so "design in English" is generate-and-validate-a-spec — the only version of that goal that's safe and uniform.
  • Keep the code hatch compatible with Tyler's VisualComponent contract so his existing instruments can be imported wholesale — an explicit goal, and free once you've borrowed his renderer.
  • Prove it on the smallest loop first: engine skeleton → ports → one Pulse modulator → matrix (phases 1–3). Everything else is additive once that feels good.
