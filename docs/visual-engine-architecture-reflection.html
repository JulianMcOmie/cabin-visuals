This document is a reflection, not a plan. The two prior docs (visual-architecture.html, visual-architecture-implementation-plan.html) argued for an architecture and sketched how to build it. You then built it - fast, and with a handful of deliberate mid-course corrections. This doc walks the as-built engine phase by phase, reads the actual source, and draws it so the mental model is unavoidable. Where the code diverged from the plan, a as-built callout says exactly how.

The one thing to internalize: the engine is a pure observer of the project document. The editor writes a store; resolveProject() flattens that store into a resolved graph (derived, never saved); every frame computeAtBeat(beat) runs the matrix, samples automation, composes world transforms, and stashes an ObjectState per object; the R3F renderers pull that state imperatively in useFrame and mutate their meshes. Data flows one way: document → resolve → per-frame compute → render+effects. Nothing downstream ever writes back.

Six things to hold in your head

  1. Two type worlds. The document types (types.ts: Track, Block, Note, Routing) are what the user edits and what persists. The resolved-graph types (core/engine/types.ts: ResolvedObject, ModulatorInstance, ResolvedRouting, ObjectState) are derived and in-memory only. The dependency points one way: engine imports document types, never the reverse.
  2. Three input layers meet on an object. params ← automation (child tracks, pitch→value), ports ← modulators (the matrix), abilities ← their own child tracks (bespoke, consumed by the instrument's own render). All three converge in computeAtBeat.
  3. A modulator is a signal, routings are the wiring. The plan's monolithic "ModulatorInstance" was split: a ModulatorInstance is a pure signal (id + kind + triggers), evaluated once per frame; ResolvedRoutings fan that one output to many object ports. Scopes (track/tag/subtree) are pre-expanded to concrete object ids at resolve.
  4. Solo is real and object-scoped. If any object track is soloed, non-soloed objects go dark - but a modulator/modifier/automation child keeps its own mute, so soloing an object never silences its own plumbing. Abilities have their own per-object solo too.
  5. Hierarchy is functional. A parent's world matrix composes into its children's - graph.objects is in parent-before-child DFS order so a parent's world is always ready when a child reads it. Modifier / automation / ability children fold into their parent instead of resolving as their own object.
  6. Effects are a separate layer. Transform / clone / shader plugins wrap the rendered object (not the port matrix). Shaders are per-object FBO (Option A): the object renders offscreen and comes back as a full-frame post-processed overlay.

1 · Phase 1 - Resolve / engine skeleton

new core/engine/types.ts resolve.ts VisualEngine.ts VisualBeatSync.tsx components/visual/VisualScene.tsx ObjectRenderer.tsx   edit App.tsx instruments/Cube.tsx

The spine. The old Cube scanned the whole project itself every frame; phase 1 replaced that with the resolve → compute → pull pipeline, and rewired the Cube to read engine state instead. Zero visible behavior change - that was the point (a good canary: if the pulse changed, the wiring was wrong).

What resolveProject does (skeleton form)

resolve.ts is a pure function ProjectSnapshot → ResolvedGraph. The snapshot is a structural slice (tracks, rootTrackIds, beatsPerBar) - ProjectStore's state satisfies it structurally, so the engine never imports the store. flattenTree walks the forest depth-first (roots then descendants, with a seen guard against cyclic data), producing track ids in parent-before-child order - a property the world-matrix composition later depends on.

For each object track, resolve calls flattenNotes - which flattens track.blocks to absolute project beats, carrying each note's containing block bounds (blockStartBeat/blockEndBeat) so downstream code can tell which notes are "live" - and pushes a ResolvedObject. VisualEngine.ts holds the resulting graph, plus a states map (trackId → ObjectState) and a worldMatrices map, all reused across frames.

as-built A refinement the plan didn't foresee: syncParams(p). The subscription splits into two paths. A cheap synchronous pass rewrites just obj.params on the already-resolved objects (so dragging a size slider is reactive at 60fps), while the full structural setProject (which re-flattens the tree, rebuilds indices, re-splits objects/modulators) stays debounced. The scene keeps rendering the previous graph until the new one lands. The registry also landed as instruments/index.ts (not registry.ts).

2 · Phase 2 - Object defs, params, ports

edit instruments/types.ts Cube.tsx   new instruments/index.ts (registry)

Phase 2 gave objects a real schema: ObjectInstrumentDef with params (public numeric knobs) and ports (hidden modulation jacks). Everything is a number - even color is a hue param in 0–360 - which keeps automation and modulation uniform.

Crucially, phase 2 also introduced localTransform as data on the def. Rather than each component computing its own position/rotation/scale, an instrument declares a pure function (ctx) → LocalTransform. The Cube's is: position from baseXPosition, a steady spin from beat, and a breathing scale boosted by the energy port. The engine composes this into a world matrix (phase 5) and the component just draws at local origin - it only owns appearance (color/emissive) and its ability.

PortDef carries its own combine rule. { key, label, combine: 'add'|'multiply'|'max'|'replace', default, range? }. The combine rule lives on the port (the object author's choice) not the routing - so "energy adds", "a freeze replaces". The Cube's three ports all default to 0 and combine by add. Ports rest at their default until a modulator drives them.

2b · Automation - pitch→value parameter lanes

new core/engine/automation.ts   edit resolve.ts VisualEngine.ts (computeAtBeat) TrackContextMenu.tsx

Automation shipped as its own phase (4 commits). It's the params ← automation input layer. The trick borrowed from Tyler: an automation lane is a MIDI editor where a note's pitch encodes the value and its time is the keyframe time; the engine interpolates between keyframes.

At resolve, resolveAutomations gathers each object track's automation child tracks into ResolvedAutomation[] (each: {param, mode, keyframes}), skipping children with no/unknown target param, and honoring per-object automation solo. In computeAtBeat, if an object has any automations, its params are copied and overlaid: params[auto.param] = sampleLane(auto.keyframes, beat, auto.mode) - this happens before localTransform reads them, so an automated baseSize flows straight into the world matrix.

as-built An automation track is a real child track (type:'automation', no instrument, targetParam, interpolation), added opt-in via the track right-click menu and locked to its object (can't be re-parented). It revived a dormant stub (TrackType 'automation', generateAutomationRows) rather than inventing a new container - mirroring the modifier-track pattern ("a child automates its parent").

3 · Phase 3 - Modulators + the matrix ★ keystone

new core/engine/matrix.ts modulators.ts instruments/modulators/Pulse.ts modulators/index.ts   edit VisualEngine.ts ProjectStore.ts (setTrackTargets)

The keystone. A modulator is an instrument that renders nothing - its notes are triggers, and its per-frame output drives a port on whatever object(s) it's routed to. A track is a modulator iff its instrumentId is in the modulator registry.

as-built Two deltas matter here. (1) The signal/routing split. The plan's single "ModulatorInstance {triggers, shape, routings}" became a pure signal ModulatorInstance { id, kind, triggers } plus separate ResolvedRouting { modulatorId, targetObjectId, targetPort, amount } links, bucketed in routingsByPort. So one signal is evaluated once and fanned to every object its scopes hit. (2) The built-in self-pulse. resolve synthesizes an implicit modulator per object (${trackId}:pulse) from the object's own notes, routed to its own energy port with amount 1 - that's the Cube's original self-pulse preserved. Explicit modulator tracks stack on top. The store action is setTrackTargets (not setRoutings).

4 · Phase 4 - Tags + group routing

edit types.ts (Routing, Track.tags) resolve.ts (tagIndex, scope expansion) matrix.ts ProjectStore.ts (setTrackTags) TrackEditor.tsx

Phase 4 made routing cross-cutting. A Routing is { port, scope: track|tag|subtree, amount }. The scope is the only user-set part - one drag onto a tag and the pulse hits every object carrying it.

Solo is real, and object-scoped. anyObjectSolo checks whether any object track (has an instrument, not a modulator) is soloed; if so, objectOff(track) darkens non-soloed objects. But modulators, modifier children, and automation children keep their own mute - soloing an object never disables the modulator targeting it or its own automation. Ability-lane solo is separate again (per object, inside resolveAbilityEvents).
as-built subtree scope was a phase-4 stub (resolved to no objects) and became live in phase 5 once the hierarchy existed - objectsInSubtree(rootId) now walks the root and its descendants, keeping only ids that resolved to objects.

5 · Phase 5 - Nested hierarchy + world-matrix inheritance

edit resolve.ts (parentId, objectsInSubtree) VisualEngine.ts (composeLocal, world)   new core/engine/transforms.ts   edit timeline (nested rows)

Hierarchy became functional: a child object inherits its parent's transform. Because flattenTree emits parent-before-child, computeAtBeat can compose in one forward pass - a parent's world matrix is always ready when its child reads it.

The renderer side: ObjectRenderer wraps its content in a placement group whose matrix is set each frame from state.world.decompose(g.position, g.quaternion, g.scale). So the world transform lives on the group; the instrument component draws its mesh at local origin. The group also carries the mute blackout (g.visible = !state?.blackedOut).

6 · Phase 6 - Ability child tracks (reworked from lanes)

edit instruments/types.ts (AbilityLaneDef) types.ts (Track.abilityKey) resolve.ts (resolveAbilityEvents) Cube.tsx (Shatter) TrackContextMenu.tsx

An ability is a signature move an instrument declares (AbilityLaneDef { key, label, editor?, color? }). The Cube declares one: Shatter. Playing notes on its ability track fires it.

as-built · SUPERSEDED PLAN This is the biggest divergence from the architecture doc. The plan said abilities were a parallel lanes[] structure on the object track (indented sub-rows drawn by a dedicated AbilityLaneRow widget with bespoke useLaneGestures), explicitly not child tracks. As shipped, abilities are real ability child tracks - functionally identical to automation tracks (Track.type='ability' + abilityKey, added opt-in via the right-click menu). The win: every block (normal / automation / ability) is track.blocks flowing through one gesture system, so marquee / move / copy / paste / alt-duplicate are completely unified. AbilityLaneRow and useLaneGestures were deleted; the old lanes/laneKey plumbing on Track is dead-but-dormant.

7 · Phase 7 - Event modifiers (add / override / suppress / mute)

new core/engine/trackTypes.ts   edit resolve.ts (applyModifiers) VisualEngine.ts (blackout)

An event modifier is a no-instrument child track that reshapes its parent's note stream at resolve time. trackTypes.ts ports Tyler's combine(parent, self) family. Four types shipped: add, override, suppress, mute.

Region minimum. A modifier note's region is [beat, beat + (durationBeats || 0.25)) - a minimum quarter-beat so a zero-length "dot" still covers something. mute uses 0.25 in resolve for its blackout region; the region() helper in trackTypes.ts uses the same default for suppress hit-testing.

8 · Phase 8 - Effects (transform / clone / shader)

new plugins/types.ts plugins/index.ts plugins/{transforms,clones,shaders}/* components/visual/{Transform,Clone,Shader}Wrapper.tsx

Effects are a separate layer from modulators - plugins that post-process an object's rendered output. The contract (VisualPlugin) matches Tyler's: { category: 'transform'|'clone'|'shader', params, applyTransform?, getClones?, fragmentShader? }. A track's visualPlugins: PluginInstance[] attaches them; time is the beat (music-synced), not wall-clock.

as-built Deferred from Tyler (named, not built): the multi-scene SceneCompositor + masks (unneeded for per-object post-fx), the instancing pools (lib/three/* - an instrument concern), and automating effect params (automation targets instrument params only so far). Enum/boolean plugin params render as numeric sliders (e.g. includeCenter · 0/1).

9 · Phase 9 - RenderSpec interpreter

new core/engine/renderSpec.ts instruments/specInstrument.tsx instruments/shapes.tsx

A RenderSpec is a serializable object-instrument description: a geometry primitive plus per-property expression bindings over the object's params and ports. So a new shape is data, not a hand-written component (and later, something an LLM can emit). Circle and Triangle are specs.

10 · Phase 10 - English → spec (the frontier)

not started

The last phase - an LLM endpoint that emits and validates a RenderSpec - is deliberately unbuilt. The substrate is now in place: a serializable spec (renderSpec.ts), a safe expression vocabulary that throws on anything outside it, and a spec-to-def bridge (specInstrument.tsx). "Design an instrument in English" becomes "LLM writes a RenderSpec, engine validates it against the grammar, specInstrument mounts it" - no arbitrary code execution.

Why it waits. The grammar is intentionally minimal (no generators, no onAbility effects yet). Per the plan, the grammar should grow from real specs, not be designed up front. English authoring is only worth wiring once the vocabulary has proven it covers the shapes people actually want.

12 · Instrument porting (as-built)

new IcosahedronBurst.tsx Sun.tsx HexagonDots.tsx CylinderFlight.tsx ParticleRiser.tsx

The recent work: porting Tyler's instruments through the code escape hatch. Each is an ObjectInstrumentDef whose component is a hand-written R3F visual - exactly the "keep the code hatch compatible with Tyler's VisualComponent" goal. They read per-frame state via getObjectState(trackId) and mutate meshes imperatively, just like the Cube.

The ported instruments demonstrate the escape hatch cleanly: IcosahedronBurst keeps its own imperative useFrame (spawning/aging wireframe shells, disposing dead ones), shares its own params/ports, and gets note-onset behavior from the engine's activeNotes. It never scans the project - it reads one ObjectState. That's the compatibility promise made good: the code hatch runs Tyler-style components largely unmodified, fed by the engine instead of by a project scan.

The through-line. Every delta pulls in the same direction: unify machinery, don't multiply it. Abilities and automation became child tracks so one gesture system edits every block. A modulator became a signal + routings so one evaluation fans out. A spec became a def so ObjectRenderer never learned about specs. The engine you built is smaller and more uniform than the one the plan drew - and that's why it holds together in your head.
