Instrument Plugin UI - Architecture

The underlying problem, in Julia's words: UI friction when adjusting instrument params. The Logic instinct - every instrument gets its own editor - decomposed into three independently shippable layers, so the architecture never demands a big bang.

1 · Three layers, three ship dates

2 · Layer 2's contract - the part worth getting right

RuleWhy it's load-bearing
Editors write ONLY through setParam (→ setTrackParam / setTrackStringParam)Undo bursts, autosave, the params fast-path (syncParams at 60fps), and automation overlays all hang off the store path. An editor with private state forks the document - the one way this feature can rot the codebase.
Editors read params as props, render-pureThey're views of the document, not owners; an automation lane driving a param must visibly move the custom editor's controls (same live-value affordance as sliders).
Generic sliders remain the fallback and the escape hatchNo editor is required; a "show raw params" toggle on custom editors keeps every value reachable when a bespoke UI hides something.
Docked panel first, floating windows maybe neverThe TrackEditor panel is already the instrument's home; Logic's floating windows solve a multi-plugin-visibility problem this app doesn't have yet. Re-evaluate after three pilots.
A small kit, not 28 bespoke castlesShip an editor-widgets kit (XY pad · hue ring · curve/enum pickers · labeled dial) so custom editors are compositions, and visual language stays coherent across instruments.

3 · The pilot - Cube

One instrument proves the seam: Cube's editor = an XY position pad (x/y params; z as a side dial), a size dial, the hue ring, and spin. Success criteria: positioning the cube takes one drag instead of three slider hunts; automation of x still moves the pad's dot live; nothing about the generic path regressed for the other 27 instruments. Second pilot should be a 2D-canvas instrument (FractalTunnel - sliders for symmetry/branches beg for a radial widget) to prove the kit generalizes beyond transforms.

4 · Decisions & non-goals

DecisionWhy
Layer order is the roadmap1 (friction now) → 2 (seam + Cube pilot) → widgets kit + second pilot → 3 (gizmos). Each stop is a stable resting point.
The registry hook is one optional fieldeditorComponent?: ComponentType<InstrumentEditorProps> on the def - the same pattern as fullFrame and abilities: instruments declare, the shell honors.
Layer 3 rides the canaryIgnore overlayGizmos are UI furniture in the scene - the drag-highlights pair establishes the escape hatch; gizmos are its second tenant, not a new invariant exception.
Non-goalsFloating/multi-window management · third-party plugin API · editor-in-canvas overlays for ALL params (gizmos are for spatial params; dials stay dials) · touch support.
The one-sentence mental model: fix the sliders this week, add one optional def field that lets an instrument bring its own editor next, grow a small widget kit through two pilots - and only then put drag handles in the scene, through the same canary escape hatch the hover highlights use.
