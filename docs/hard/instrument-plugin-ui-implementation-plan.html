Instrument Plugin UI — Implementation Plan

Four phases matching the architecture's layers — slider upgrades, the registry seam with a Cube pilot, the widget kit with a second pilot, then scene gizmos. Every phase is a stable stopping point; commit to the next only after using the last.

C · Phase-by-phase

Phase 1 — Slider upgrades ParamSlider (independent quick win)

  • Double-click → type-in (number input, Enter/Esc); Shift-drag → 10× finer; Alt-click → reset to def default; wheel over slider → step nudge. All through the existing onChange.
  • Verify on three params with very different ranges (hue 0–360, size 0.2–4, position −10–10); undo granularity stays one-burst-per-gesture.

Phase 2 — Seam + Cube pilot types · TrackEditor CubeEditor

  • Land the seam exactly as §B; generic path untouched when the field is absent (zero-risk to 27 instruments).
  • new instruments/editors/CubeEditor.tsx: XY pad (baseX × baseY, z dial beside), size dial, hue ring, spin dial; automated params render the live-dot treatment.
  • Verify: one-drag positioning; automation on baseX moves the pad dot while the base stays editable; raw-params disclosure shows everything; the OTHER instruments' panels byte-identical.

Phase 3 — Widget kit + second pilot components/instrument-widgets/*

  • Extract from the Cube editor: XYPad · Dial · HueRing · WidgetRow with one shared look (size/spacing tokens, the live-dot affordance built in).
  • Second pilot: FractalTunnel editor (symmetry as a radial segment picker, branches/generations dials, spiral dial) — proves the kit against a non-spatial instrument.
  • Verify: both editors compose from the kit only; a third instrument could be built by pattern-matching (that's the review question).

Phase 4 — Scene gizmos GizmoLayer (the direct-manipulation layer)

  • new components/visual/GizmoLayer.tsx in the canaryIgnore overlay group: when the selected track's def declares spatial params (a small gizmos?: [{kind:'position', keys:{x,y,z}}] hint on the def), render a drag handle at the object's position; drag raycasts against a camera-facing plane → setParam per frame (undo burst-collapses, as slider drags do).
  • Selection-scoped: gizmos only for the selected track; Esc/deselect hides them.
  • Verify: dragging the cube in the viewport writes baseX/Y (and Z with a modifier); canary silent while parked with gizmos visible; export unaffected (overlay group never captured — it renders only when NOT pinned... confirm the export pin hides gizmos: pin() should also hide the overlay group — one line, listed here so it isn't forgotten).

E · Risks & open questions

  • risk Editor sprawl — 28 bespoke editors is a maintenance cliff. The kit + "compositions only" review rule is the fence; instruments without obvious spatial/visual metaphors simply keep sliders.
  • risk Gizmo drags fighting camera controls if/when orbit controls exist — gizmo pointer-down must capture before any camera handler; design the z-order now (overlay on top), revisit when a camera rig lands.
  • open Whether the editor panel needs MORE SPACE for custom editors (the TrackEditor column is ~narrow) — the Cube pilot answers it; a pop-out is the fallback, not the premise.
  • note Phases 1 and 2 are genuinely independent — phase 1 could ship in the next easy batch while this doc's remainder waits.
The one-sentence mental model: one optional def field and a strict "write only through setParam" contract buy instrument-native editors without forking the document's truth — pilot on Cube, extract the kit, and let gizmos be the overlay group's second tenant.
