Incremental Resolve — Implementation Plan

Phase 0 is a stopwatch and a gate. If the gate opens: split resolve at the compile/assembly joint, memoize compiles in a WeakMap keyed by track reference, and prove output-equality against a corpus before trusting a single cached entry.

B · Phase-by-phase

Phase 0 — Measure, then maybe stop the gate

  • Dev-only probe: wrap setProject's resolve in performance.mark/measure; log p50/p95 over an editing session on Julia's largest real project (plus a synthetic 100-track / 10k-note document as the stress ceiling).
  • Gate: p95 > ~8ms on real work → proceed. Under it → record the numbers in this doc, close the item as "premature", keep the probe behind a flag for the future.
  • Also attribute: how much of the cost is per-track compile vs assembly? If assembly dominates (unexpected), this plan's premise is wrong — stop and re-architect rather than memoizing the wrong half.

Phase 1 — The split (no caching yet) resolve.ts

  • Refactor resolve into compileTrack(track, beatsPerBar): CompiledTrack (notes flattened + loop-tiled, modifiers folded, ability events, lanes) and assemble(compiled[], projectState): ResolvedGraph (tag index, routings, solo, ordering, port buckets). Pure functions, no cache.
  • Acceptance: deep-equal output vs the pre-refactor resolve on a corpus: current real projects (export a few documents as fixtures) + fixtures with modifiers, abilities, tag routings, loops, tag automation. This phase carries ALL the refactor risk — land it alone, verified, before any memoization.

Phase 2 — The cache resolve.ts (~20 lines)

  • WeakMap<Track, CompiledTrack> + a module-level cachedBeatsPerBar (mismatch → new WeakMap). compileTrack consults it; assemble never caches.
  • WeakMap on the track object ⇒ no eviction code, no leak (dropped tracks GC), no invalidation API — the immutable store IS the invalidation.
  • Verify: the phase-1 corpus again (cache on) → still deep-equal; then mutation sequences (edit notes on A, check B/C hit cache via a dev counter; undo restores an OLD reference — cache HIT, by design, and correct because content ≡ reference).

Phase 3 — Numbers + guardrails

  • Re-run phase 0's probe: report before/after p50/p95 on the same projects in this doc. Expected shape: single-track edits drop from O(project) to O(1 track + assembly).
  • Dev-flag ?fullResolve=1 forces the uncached path — the escape hatch for "is the cache lying?" debugging forever after.
  • Add the corpus equality check as a repeatable script (run before any future resolve change — loops, tag automation, and mute-lane features all touch compile).

D · Risks & open questions

  • risk Hidden cross-track reads inside per-track compile (the refactor's landmine — e.g., a modifier consulting a sibling). Phase 1's corpus equality is the detector; any such read either moves to assembly or becomes an explicit compile input.
  • risk Future features breaking the joint — tag automation and nested mute/solo both belong in ASSEMBLY (cross-cutting); the doc comment on compileTrack must say "no reads outside this track" loudly.
  • open The gate number (~8ms) is a proposal — Julia may prefer "when I feel it", which is also a valid gate; write down whichever is chosen.
  • note If phase 0 says "not yet": that's success too. The probe and this plan wait, and the item stops haunting the backlog.
The one-sentence mental model: measure, split resolve at its compile/assembly joint with output-equality as the safety net, then let a WeakMap on immutable track references do all the invalidation — twenty lines of cache after a refactor whose only job is to make those twenty lines safe.
