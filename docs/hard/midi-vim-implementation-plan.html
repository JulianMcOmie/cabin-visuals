B · The state machine (the only new "engine")

// components/midi/vim/vimState.ts - pure reducer, exhaustively testable
interface VimState {
  mode: 'off' | 'normal' | 'visual'
  caret: { beat: number; rowIndex: number }     // rowIndex in RowModel space
  count: number | null                          // pending count prefix
  pendingOp: 'd' | 'y' | null
  yankBuffer: Note[]                            // block-relative, caret-relative on put
}
// step(state, key, ctx: { rowModel, quantize, notes, blockBeats }) → { state, effects[] }
// effects are DESCRIPTIONS ('insertNote', 'deleteNotes', 'setSelection', 'setFilter', …)
// executed by a thin adapter over the existing note actions - the reducer never touches stores.

Pure reducer + effect descriptions = unit tests without a DOM, and the adapter is the single file that knows how vim verbs map onto useMidiEditorState/useNoteGestures mutations.

C · Phase-by-phase

Phase 0 - Keymap review Julia + Tyler · dependency gate

  • Walk the architecture doc's §2 table in the promised "midi vim discussion"; freeze v1 keys. RowModel merged (row-hiding pair) before any code here.

Phase 1 - Mode + caret + motions vim/*

  • Toggle (toolbar + hotkey) → 'normal'; key capture on the piano-roll container (scoped: only when the editor has focus; inputs excluded; existing shortcuts pass through when OFF, vim swallows when ON).
  • Caret rendering (cell outline on the grid) + h/l/j/k/w/b/0/$ with counts; caret auto-scrolls into view. No mutations yet.
  • Verify: motions respect quantize and the ACTIVE row filter (j under scale filter = next scale degree - the substrate payoff, demonstrated day one).

Phase 2 - Operators adapter over existing actions

  • i/a insert (quantize-length note at caret, last-used velocity) · x/dw delete · v visual driving the EXISTING selection set · y/p through the EXISTING note clipboard · < > nudge · J/K transpose by visible row.
  • Undo: every effect batch = one gesture (the editor's commit-per-gesture pattern) so u behaves like vim users expect.
  • Verify: enter a 4-bar drum line entirely by keys in under a minute (the acceptance demo); mouse editing mid-vim-session stays coherent (shared selection/clipboard prove themselves).

Phase 3 - View commands + status chip

  • zf filter cycle · zs scale-from-caret · +/− row scale · octave jumps; a small status chip in the toolbar shows NORMAL · 4 (mode + pending count) - modes without indicators are user hostility.
  • Verify Julia's original scenario end-to-end: hide rows to two lanes, enter kicks a bar apart with a 16l a 16l…, zero scrolling.

E · Risks & open questions

  • risk Key-capture leaks - vim-ON must not eat the transport space bar unless focused, and must fully release on toggle-off/editor close; the scoping rules in phase 1 are the contract, tested explicitly.
  • risk Grammar creep - every vim user wants their next favorite binding; the constants file + "drives existing actions only" rule keeps additions cheap and bounded.
  • open Should i preview-play the note's pitch (audio)? There's no synth - visual flash at the object instead? Decide in the keymap review.
  • note Macros (q/@) are the highest-value follow-up for pattern entry - the reducer's effect stream is recordable by design; note it, don't build it.
The one-sentence mental model: a pure key→effects reducer whose effects are executed by one adapter over the editor's existing mutations - vim as vocabulary, the piano roll as the only engine, and the RowModel as the space every motion moves through.
